# 1. 总体拆分原则

主链路：
- 输入目标对象
- 解析与标准化
- 发现缺陷
- 结构化 finding
- 形成裁决
- 生成 owner review
- 导出 AuditPack

当前版本目标：
- 跑通一个窄内核闭环
- 支持一个 scenario class
- 支持一个 environment profile
- 支持一种连续审计类型
- 支持一种标准交付物

当前版本坚决不扩：
- 不做多场景矩阵
- 不做完整控制台平台
- 不做高拟真环境家族
- 不做自治飞轮
- 不做全自动归因

模块分层方法：
- `Core Chain Modules`：不做主链路就死
- `Support Modules`：不做主链路不死，但质量、稳定性或交付会明显受损
- `Governance Modules`：控制边界、升级、授权、预算、版本
- `Deferred Modules`：现在不做实现，只留接口
- `Forbidden-for-Current-Version Modules`：明确禁止进入当前版本

拆分口径：
- 先发现，再裁决，再交付
- 确定性执行进脚本
- 规则化判断和解释进 skill
- 当前主链路允许双态，但不能让双态反过来吞掉确定性骨架

# 2. 模块分层总览

## Core Chain Modules
纳入原则：
- 不做则当前最小闭环无法跑通
- 有明确输入输出
- 能给出可观察完成证据
- 不是为未来通用性提前加的

模块：
- Intent Intake Gateway
- Skill Parsing and Normalization
- Contract and Schema Validation
- Rule and Boundary Scanning
- Risk Pattern Detection
- Finding Construction
- Finding Adjudication
- Owner Review Builder
- Continuous Audit Session Runtime
- Evidence and AuditPack Builder

## Support Modules
纳入原则：
- 不是闭环生死线
- 但能显著降低误判、提升稳定性、提升交付质量

模块：
- Result Stability Checks
- Runtime Recovery and Telemetry
- Template and Report Composition

## Governance Modules
纳入原则：
- 用于控制升级、范围、预算、版本、权限、长期治理
- 不要求第一天做满，但必须有清晰边界

模块：
- Policy and Scope Control
- Escalation Gate
- Version and Residual Risk Registry

## Deferred Modules
纳入原则：
- 当前不做不会导致最小闭环死亡
- 但后续大概率需要
- 当前只留接口，不实现

模块：
- Knowledge Distillation
- Intelligence Store
- Multi-environment Adapter Layer
- Budget Automation

## Forbidden-for-Current-Version Modules
纳入原则：
- 明显属于膨胀项
- 当前引入会拖垮主链路

模块：
- Multi-scenario Matrix
- High-fidelity Environment Family
- Full Control Tower Dashboard
- Fully Automatic Root-cause Engine
- Autonomous Strategy Optimization
- Hypothetical Customer Prebuild

# 3. 模块逐个拆解

## 模块名称：Intent Intake Gateway
### 1) 模块目标
接收审计目标和运行上下文，把输入固定成可审计的起点对象。

### 2) 模块边界
负责什么：
- 接收 `intent_id / repo_url / commit_sha / at_time / optional manifest_path`
- 固化一次审计输入快照

不负责什么：
- 不负责发现缺陷
- 不负责裁决
- 不负责生成报告

### 3) 输入
- CLI 或 API 输入
- 必填：`intent_id`, `repo_url`, `commit_sha`
- 可选：`at_time`, `entry_path`, `manifest_path`, `environment_profile_id`

### 4) 输出
- `IntakeRequest`
- `AuditTargetRef`
- 固定化的输入快照记录

### 5) 前置依赖
- 无

### 6) 核心子能力
- 输入解析
- 参数合法性检查
- 缺失字段拒绝
- `repo_url + commit_sha` 强制化

### 7) MVP 范围
- 只支持 GitHub repo + commit_sha
- 只支持单 repo 审计输入
- 不支持 MCP、OpenClaw、插件市场、多源聚合

### 8) 完成证据
- 可运行命令：
  - `skillforge run <intent_id> --repo <url> --commit <sha>`
- 生成固定格式 `intake_request.json`
- 日志中可见固定化输入对象

### 9) 验收标准
- 缺少 `commit_sha` 时必须失败
- 非法 `intent_id` 必须失败
- 相同输入重复执行时输出完全一致的 `IntakeRequest`

### 10) 风险点
- 仍允许 branch 输入，会破坏可复现性
- 输入对象字段漂移，后续模块会全部跟着失控

### 11) 当前版本判定
- 当前必须实现

## 模块名称：Skill Parsing and Normalization
### 1) 模块目标
把 repo 中的 skill/manifest/依赖结构读懂并标准化，形成后续检查的统一输入。

### 2) 模块边界
负责什么：
- 解析 skill 文件
- 读取 manifest
- 提取依赖、输入输出合同、边界声明

不负责什么：
- 不负责风险裁决
- 不负责 owner 展示

### 3) 输入
- `AuditTargetRef`
- repo checkout 内容
- `manifest_path` 或自动发现入口

### 4) 输出
- `NormalizedSkillSpec`
- `DependencyGraph`
- `ManifestSnapshot`

### 5) 前置依赖
- Intent Intake Gateway

### 6) 核心子能力
- `SkillParser`
- `ManifestReader`
- `DependencyExtractor`
- `InputNormalizer`

### 7) MVP 范围
- 只支持当前仓库约定的 skill 目录结构
- 只处理 Python skill 和 manifest
- 不做通用多语言解析框架

### 8) 完成证据
- 固定格式 `normalized_skill_spec.json`
- 单元测试覆盖：
  - 正常 skill
  - 缺失 manifest
  - 依赖异常

### 9) 验收标准
- 能稳定解析现有样例 skill
- 解析失败时输出明确错误码和失败原因
- `NormalizedSkillSpec` 字段完整率达到 100%

### 10) 风险点
- 解析规则写得太泛会导致误判
- 过早做多语言支持会使结构膨胀

### 11) 当前版本判定
- 当前必须实现

## 模块名称：Contract and Schema Validation
### 1) 模块目标
对结构化 skill 对象做确定性合同校验，先拦掉一批明确错误。

### 2) 模块边界
负责什么：
- schema 校验
- contract 校验
- 输入输出字段一致性检查

不负责什么：
- 不负责模式风险识别
- 不负责业务裁决

### 3) 输入
- `NormalizedSkillSpec`
- `OutputContract`
- policy / error code registry

### 4) 输出
- `ValidationReport`
- `validation_failures[]`

### 5) 前置依赖
- Skill Parsing and Normalization

### 6) 核心子能力
- `SchemaValidator`
- `ContractChecker`
- `SkillContractValidator`
- `OutputSchemaValidator`

### 7) MVP 范围
- 只做结构和合同层检查
- 不做语义理解层判断

### 8) 完成证据
- 可运行校验命令
- 输出 `validation_report.json`
- 失败样例固定报错

### 9) 验收标准
- 结构缺陷必须被稳定拦截
- 合法样例不能误报失败
- 每个失败项必须带错误码、字段路径、修复建议

### 10) 风险点
- 规则过宽导致误报
- 规则过松导致后续模块背锅

### 11) 当前版本判定
- 当前必须实现

## 模块名称：Rule and Boundary Scanning
### 1) 模块目标
用确定性规则扫描危险模式、边界缺口和高风险动作暴露。

### 2) 模块边界
负责什么：
- 静态规则扫描
- 边界检查
- 危险动作/权限扫描

不负责什么：
- 不负责最终风险等级裁决
- 不负责 owner 卡片生成

### 3) 输入
- `NormalizedSkillSpec`
- 代码/配置文件
- 规则库

### 4) 输出
- `RuleScanReport`
- 初始 `raw_findings[]`

### 5) 前置依赖
- Skill Parsing and Normalization
- Contract and Schema Validation

### 6) 核心子能力
- `RuleScanner`
- `BoundaryChecker`
- 权限/危险动作规则集
- 依赖异常规则

### 7) MVP 范围
- 只扫描已定义的规则集
- 不做动态执行

### 8) 完成证据
- 固定格式 `rule_scan_report.json`
- 能稳定命中已知敏感模式样例

### 9) 验收标准
- 对当前定义的规则样例命中率达到预期
- 每个命中项都能定位到来源位置
- 每个命中项都能追溯到规则版本

### 10) 风险点
- 规则集未版本化会造成结果不可复核
- 扫描范围过大导致噪音过多

### 11) 当前版本判定
- 当前必须实现

## 模块名称：Risk Pattern Detection
### 1) 模块目标
在确定性扫描之上识别常见风险模式和治理缺口，形成更高价值的候选 finding。

### 2) 模块边界
负责什么：
- 识别已知 anti-pattern
- 识别治理缺口组合

不负责什么：
- 不负责凭空发现新型问题
- 不负责最终结论

### 3) 输入
- `RuleScanReport`
- `ValidationReport`
- `NormalizedSkillSpec`
- pattern library

### 4) 输出
- `PatternDetectionReport`
- `pattern_findings[]`

### 5) 前置依赖
- Rule and Boundary Scanning

### 6) 核心子能力
- `RiskPatternMatcher`
- `AntiPatternLibrary`
- `GovernanceGapDetector`

### 7) MVP 范围
- 只支持少量高价值模式
- 不追求“覆盖所有模式”

### 8) 完成证据
- 固定样例能命中预期模式
- 输出 `pattern_detection_report.json`

### 9) 验收标准
- 至少覆盖当前锁定的 4 类缺陷中的高价值子集
- 每个模式命中必须能说明来源依据

### 10) 风险点
- 过早扩模式库会导致假通用性
- 语义层过强会掩盖确定性证据不足

### 11) 当前版本判定
- 当前必须实现

## 模块名称：Finding Construction
### 1) 模块目标
把结构缺陷、规则缺陷、模式缺陷统一收束成可裁决的 finding 对象。

### 2) 模块边界
负责什么：
- finding 结构化
- 严重度初赋值
- 证据绑定
- 初始置信度

不负责什么：
- 不负责最终裁决
- 不负责 owner 翻译层

### 3) 输入
- `ValidationReport`
- `RuleScanReport`
- `PatternDetectionReport`
- 证据引用

### 4) 输出
- `Finding[]`

### 5) 前置依赖
- Contract and Schema Validation
- Rule and Boundary Scanning
- Risk Pattern Detection

### 6) 核心子能力
- `FindingBuilder`
- `EvidenceBinder`
- `SeverityAssigner`
- `ConfidenceInitializer`

### 7) MVP 范围
- finding 统一字段先固定
- 不做复杂 finding 图谱

### 8) 完成证据
- `findings.json`
- 每条 finding 均带：
  - id
  - title
  - category
  - severity
  - confidence
  - evidence refs

### 9) 验收标准
- 所有 finding 都能追溯到输入来源和证据
- finding 字段完整率 100%
- 没有无法定位依据的“裸 finding”

### 10) 风险点
- 证据绑定不严格，后续裁决会虚
- finding 结构经常改字段，会拖垮报告和前端

### 11) 当前版本判定
- 当前必须实现

## 模块名称：Finding Adjudication
### 1) 模块目标
把技术 finding 转成可判定的裁决对象，回答“这像不像真问题”。

### 2) 模块边界
负责什么：
- 真实度判断
- 证据硬度判断
- 影响等级判断
- 建议动作判断

不负责什么：
- 不负责重新取证
- 不负责渲染 UI

### 3) 输入
- `Finding[]`
- 证据等级
- 替代解释记录
- 规则化裁决模板

### 4) 输出
- `AdjudicatedFinding[]`

### 5) 前置依赖
- Finding Construction

### 6) 核心子能力
- `FindingAdjudicator`
- 替代解释压缩
- 证据充分性判断
- 动作建议器

### 7) MVP 范围
- 先支持固定 4 档：
  - 已确认
  - 高概率真实
  - 可疑未证实
  - 大概率误报
- 先支持固定 4 档影响级别

### 8) 完成证据
- `adjudicated_findings.json`
- 每条 finding 带 7 字段裁决卡数据

### 9) 验收标准
- 每条 finding 必须有明确裁决状态
- 每条 finding 必须有“最大疑点”
- 每条 finding 必须有“建议动作”

### 10) 风险点
- 把“异常现象”直接升级成“真实缺陷”
- 没有替代解释字段会让 owner 被误导

### 11) 当前版本判定
- 当前必须实现

## 模块名称：Result Stability Checks

# 5. 第一阶段必须落地的最小闭环

当前最小闭环必须由以下模块组成：
- Intent Intake Gateway
- Skill Parsing and Normalization
- Contract and Schema Validation
- Rule and Boundary Scanning
- Risk Pattern Detection
- Finding Construction
- Finding Adjudication
- Owner Review Builder
- Policy and Scope Control
- Escalation Gate
- Continuous Audit Session Runtime
- Evidence and AuditPack Builder

少了哪个模块就无法形成真正闭环：
- 少 `Intent Intake Gateway`：没有确定输入，无法复现
- 少 `Skill Parsing and Normalization`：后续全部无统一对象
- 少 `Contract and Schema Validation`：结构缺陷无法确定性拦截
- 少 `Rule and Boundary Scanning`：没有核心发现能力
- 少 `Risk Pattern Detection`：高价值治理缺口难以形成 finding
- 少 `Finding Construction`：发现信号无法变成对象
- 少 `Finding Adjudication`：结果无法裁决
- 少 `Owner Review Builder`：owner 无法验收
- 少 `Policy and Scope Control`：无边界，系统不可信
- 少 `Escalation Gate`：L3 到 L4 会失控
- 少 `Continuous Audit Session Runtime`：连续审计无法成立
- 少 `Evidence and AuditPack Builder`：无法交付

只是锦上添花的模块：
- Result Stability Checks
- Runtime Recovery and Telemetry
- Template and Report Composition
- Version and Residual Risk Registry

# 7. 建议的开发顺序

## 第 1 批
- Intent Intake Gateway
- Skill Parsing and Normalization
- Contract and Schema Validation
- Rule and Boundary Scanning
- Risk Pattern Detection
- Finding Construction

为什么这样排：
- 先把“发现”做实
- 先用确定性能力把最基本的缺陷抓出来
- 没有这一批，后面裁决和交付都只是空转

## 第 2 批
- Finding Adjudication
- Owner Review Builder
- Policy and Scope Control
- Escalation Gate

为什么这样排：
- 这一批把“发现结果”变成“可判结果”
- 同时防止系统无边界扩张
- 是从技术发现走向业务验收的关键层

## 第 3 批
- Continuous Audit Session Runtime
- Evidence and AuditPack Builder
- Runtime Recovery and Telemetry
- Template and Report Composition
- Version and Residual Risk Registry

为什么这样排：
- 这一批把系统从“能发现和判”推进到“能连续审计、能交付、能结案”
- 是 L4 最小闭环的实现层
