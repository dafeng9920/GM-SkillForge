交付给 Codex 的实施文档**

主题：长时间审计 / Continuous Audit / Long-Run Audit  
用途：作为 SkillForge 从 L3 收尾过渡到 L4
的骨架设计、对象模型、流程规范与实施顺序说明  
受众：Codex / 后续新账号 / 未来的实现者与复核者

| **项目阶段** | L3 主链路收尾 -\> L4 起步前的架构固化点                                                                |
|--------------|--------------------------------------------------------------------------------------------------------|
| **定位**     | 长时间连续审计是 L4 的标志性产物之一，但应以最小可运行骨架切入，而不是一次性做成全能系统。             |
| **核心判断** | 执行层必须每案隔离与重置，知识层必须沉淀进飞轮；L3 负责发现与定界，L4 负责连续验证、放大、固化与结案。 |
| **文档目标** | 让 Codex 能直接据此落地一版可运行、可交付、可复核、可扩展的 Continuous Audit Runtime。                 |

**1. 文档定位与总判断**

本文档服务于 SkillForge 在 L3 收尾与 L4
起步之间的关键过渡：把此前围绕“长时间审计 / 连续审计 /
长窗口专项审计”的讨论，整理成可直接指导 Codex
落地的骨架文档。其目标不是给出全部代码，而是把系统边界、对象模型、流程规则、结案标准、环境策略、交付逻辑与实施顺序先固定下来。

| **核心结论** L4 的标志性产物不是单一测试脚本，而是 Continuous Audit Runtime：一套支持长时间运行、分阶段观察、异常固化、结案裁决与知识蒸馏的连续审计运行框架。 |
|---------------------------------------------------------------------------------------------------------------------------------------------------------------|

> • L3 负责发现问题、形成结构化 finding、输出基础证据与深测建议。
>
> • L4
> 负责把被挑选出来的高价值问题推进到连续验证、条件放大、长期观察与可交付证据闭环。
>
> • 执行层必须隔离与重置，知识层必须继承与蒸馏；不能二者混淆。
>
> • 连续审计不应默认追求 100%
> 拟真环境，而应围绕风险路径做高拟真、可控、可复现的环境声明。

**2. 为什么要在 L3 收尾时植入这套框架**

如果等到 L4
做了一半才补连续审计骨架，后面极有可能出现对象模型重写、主流程断裂、报告结构不兼容、交付边界失控等返工问题。因此，正确的时机是在
L3 后端主链路已经收敛时，把 L4
将来必然需要的会话、证据、事件、结案、环境与专项立案接口先埋进去。

建议在当前阶段就固定的核心对象

| **对象**           | **用途**                                                            | **是否应立即埋入** |
|--------------------|---------------------------------------------------------------------|--------------------|
| AuditSession       | 连续审计会话主对象，承载时间窗口、状态、预算与环境引用。            | 是                 |
| EvidenceItem       | 统一表示截图、录像、trace、日志片段、DOM 快照等证据。               | 是                 |
| Incident           | 统一表示告警、异常、事件升级与人工处理记录。                        | 是                 |
| EnvironmentProfile | 声明本案应用环境、代理策略、账号权限、故障注入与重置规则。          | 是                 |
| EscalationCaseCard | 从 L3 finding 升级到 L4 专项长时间审计的立案卡。                    | 是                 |
| CaseVerdict        | 最终裁决对象，决定本案属于信号、finding、已复现问题还是高可信缺陷。 | 是                 |

**3. 长时间审计的定义收敛**

“长时间审计”不能被定义成一个无边界的大筐。为了后续编码和交付清晰，建议在系统中明确拆分为三类。

| **类型**         | **主要目标**                                   | **典型触发**                                       | **验收重点**                                 |
|------------------|------------------------------------------------|----------------------------------------------------|----------------------------------------------|
| Continuity Audit | 验证连续运行下的稳定性、存活性与结果一致性。   | 任务长跑、定时触发、日内连续执行                   | 是否持续可运行、异常率、恢复行为、结果一致性 |
| Stress Audit     | 在更高压条件下暴露失败路径与边界。             | 高频调用、代理切换、并发、资源受限                 | 问题暴露频率、阈值、保护动作、证据硬度       |
| Drift Audit      | 观察跨时间态的状态漂移、累积偏差与长期副作用。 | token 老化、session 漂移、长链路重试、外部回调迟滞 | 偏差轨迹、触发窗口、影响边界、替代解释排除   |

| **实践口径** 长时间审计的核心不是“时间越长越好”，而是“是否跨越了关键时间态并捕获了时间相关风险”。8 小时的高质量专项，可能比 72 小时的空转更有价值。 |
|-----------------------------------------------------------------------------------------------------------------------------------------------------|

**4. 总体架构：Continuous Audit Runtime**

为了避免后期把录像、代理、监控、日志、告警与结案逻辑拼成一堆工具碎片，建议从一开始就采用三平面结构。

| **平面**        | **组件**                                                                                                    | **职责**                                                          |
|-----------------|-------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------|
| Control Plane   | Audit Session Orchestrator / Policy Engine / Alerting & Escalation / Human Oversight Console                | 负责会话生命周期、策略约束、告警升级、人工接管与 owner 决策钩子。 |
| Execution Plane | Environment Provisioning / Traffic & Proxy Control / Scenario Executor / Evidence Capture / Fault Injection | 负责真正的运行、场景执行、网络行为、故障注入与证据采集。          |
| Data Plane      | Log Store / Artifact Store / Telemetry Stream / AuditPack Builder / Intelligence Store                      | 负责实时数据、证据归档、报告构建、知识蒸馏与后续数据飞轮。        |

**4.1 架构原则**

> • 运行基础设施先系统化，审计逻辑再 skill
> 化；不要一开始把整个框架硬塞成 skill。
>
> • 状态运行态与知识经验态必须分仓；前者需要隔离，后者需要蒸馏。
>
> •
> 历史知识可以影响优先级、证据策略与误报排查顺序，但不能直接决定本案结论。
>
> • 系统必须能停、能收、能恢复、能结案，而不只是能一直跑。

**5. 对象模型**

以下对象是建议优先固化的最小骨架。字段不是穷尽版本，但建议 Codex
先围绕这些对象组织代码与存储。

| **对象**           | **关键字段建议**                                                                                                                             | **说明**                                             |
|--------------------|----------------------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------------|
| AuditSession       | session_id, target_id, protocol_version, audit_type, risk_level, start_time, planned_end_time, status, environment_ref, budget_ref           | 本案主对象。所有时间、环境、成本、结案都挂在它上面。 |
| ScenarioRun        | scenario_id, session_id, step_graph, current_step, retry_count, result_state, start_at, end_at                                               | 表示一次具体场景运行。                               |
| EvidenceItem       | evidence_id, session_id, scenario_id, step_id, timestamp, evidence_type, file_ref, hash, environment_fingerprint, annotation                 | 统一证据抽象。必须绑定时间、会话与环境。             |
| Incident           | incident_id, session_id, severity, category, trigger_time, signal_source, linked_evidence_refs, resolution_state                             | 异常、告警、升级与人工处理的统一对象。               |
| EnvironmentProfile | profile_id, scenario_class, auth_mode, api_dependencies, network_policy, proxy_region, data_seed, state_reset_policy, fault_injection_policy | 定义应用环境，不等于每案都从零搭一套。               |
| EscalationCaseCard | case_id, source_finding_id, hypothesis, trigger_condition, evidence_goal, stop_rule, out_of_scope                                            | L3 升级到 L4 专项前必须完成的立案卡。                |
| Checkpoint         | checkpoint_id, session_id, phase, timestamp, resumable_state_ref, validity_tag                                                               | 用于长案中断恢复与有效性声明。                       |
| CaseVerdict        | case_id, status_class, confidence, evidence_level, residual_risk, closure_reason, retest_recommendation                                      | 最终裁决对象。                                       |
| DistilledPattern   | pattern_id, pattern_type, source_cases, confidence, rule_payload, last_validated_at                                                          | 知识蒸馏结果，用于飞轮，而非参与本案执行态。         |

**6. 场景与环境：不要一 skill 一环境现搭**

应用环境需要，但不应为每个 skill
临时手工搭一套“完整真实世界”。建议采用“三层环境模型”。

| **层级**                       | **内容**                                                                      | **设计原则**                         |
|--------------------------------|-------------------------------------------------------------------------------|--------------------------------------|
| Canonical Baseline Environment | 统一的操作系统、浏览器、依赖、日志、录像、网络控制、时间同步与 agent 底座。   | 尽量固定，保持可复现与低漂移。       |
| Scenario Class Template        | 按浏览器自动化、API 工作流、数据处理、外部动作、多步 agentic 流程等做模板。   | 维护少数几类模板，而不是无限多环境。 |
| Skill-Specific Overlay         | 针对账号权限、目标站点、API key、地区代理、测试数据、故障注入规则做参数覆盖。 | 覆盖差异，不重建底座。               |

**6.1 拟真度声明**

面对“不是 100% 拟真环境”的质疑，系统不应争论，而应输出 Fidelity
Declaration。即：清楚声明本案属于 Simulated / Staging-like /
Production-like Controlled
的哪一档，覆盖了哪些关键路径，未覆盖哪些真实变量，结果在哪些边界内有效。

**7. 从 L3 到 L4：升级机制**

并非所有 L3 finding
都值得进入长时间审计。需要在两段之间加一个“收敛层”，避免前浅后散。

**7.1 L3 Finding 最低合格要求**

> • 必须有可定位的问题路径，不能只有一句模糊描述。
>
> • 必须有初始证据：至少能够指出异常出现的步骤、输入与现场片段。
>
> • 必须有问题假设 hypothesis。
>
> • 必须写明 evidence gap：当前还缺什么证据。
>
> • 必须给出 deep-test target：如果升级，长时间审计要证明什么。

**7.2 Escalation Gate**

建议只让满足以下至少两条的 finding
升级：风险等级达到阈值；涉及关键动作链；出现时间累积型特征；存在状态漂移/异步/重试/限流类特征；潜在损失高；客户或
owner 明确要求验证。

**7.3 Deep Audit Case Card**

| **字段**          | **要求**                                                       |
|-------------------|----------------------------------------------------------------|
| Finding           | 本次要验证哪一个问题，不允许模糊化。                           |
| Hypothesis        | 为什么认为该问题会发生。                                       |
| Trigger Condition | 准备在何种条件下诱发或放大它。                                 |
| Evidence Goal     | 要拿到什么证据才算完成。                                       |
| Stop Rule         | 满足哪些条件就停；包含证据达标、反证成立、范围漂移等停止条件。 |
| Out of Scope      | 本次明确不测什么，防止专项退化成全量重测。                     |

**8. Time Governance / Reset / Recovery / Cost / Scope**

这五层是此前容易被低估、但决定后续是否可经营、可结案、可持续的重要控制层。

| **控制层**                    | **为什么必须存在**                                            | **最小实现建议**                                                                                   |
|-------------------------------|---------------------------------------------------------------|----------------------------------------------------------------------------------------------------|
| Time Governance               | 不同问题在不同时间态上出现；单纯“跑 72h”没有观察结构。        | 把会话拆成预热期、稳定期、重点观察期、异常放大期、收口期，并为每阶段设采集密度与阈值。             |
| State Reset & Recovery        | 长案中断并不等于被测 skill 失败；需要区分环境失败与目标失败。 | 引入 checkpoint、恢复策略、恢复后有效性标记、是否允许跨恢复段合并证据。                            |
| Cost Gating                   | 连续审计会持续烧云机、代理、存储与人工响应。                  | 实时感知预算占用，触发降采样、追加预算确认或低价值案自动终止建议。                                 |
| Authorization & Scope Control | 长时间审计是持续行为，不明确授权很容易越界。                  | 固化 ScopePolicy、ForbiddenActionSet、AllowedStressProfile、EvidenceRetentionPolicy。              |
| Adjudication & Closure        | 没有结案层，就只是在堆证据，而不是结案。                      | 使用 CaseVerdict 明确 status、confidence、residual risk、closure reason 与 retest recommendation。 |

**9. 证据体系与问题确认分级**

对 SkillForge
来说，最危险的并不是漏报，而是把“现象”误写成“已确认缺陷”。因此系统必须内建分级。

| **级别**           | **定义**                                         | **允许怎样表述**                 |
|--------------------|--------------------------------------------------|----------------------------------|
| Signal             | 观察到异常现象，但还不能证明真实问题成立。       | 观察到异常信号。                 |
| Structured Finding | 异常已结构化记录，具备初始证据与问题假设。       | 发现了需要进一步验证的问题候选。 |
| Reproduced Issue   | 在可控条件下重复出现，触发条件更明确。           | 该问题在声明条件下可重复复现。   |
| Confirmed Defect   | 已排除主要替代解释，证据密度高，可稳定支撑结论。 | 高可信确认缺陷。                 |

**9.1 证据充分性规则**

> • E1：单点现象；通常不应直接进入最终缺陷结论。
>
> • E2：多模态证据支持（如截图 + 日志 + 时间线）。
>
> • E3：重复复现；可控条件下多次出现。
>
> • E4：替代解释已大幅压缩；可以进入高可信结论。

**9.2 Alternative Explanations**

每个高价值 finding
都应记录已考虑、已排除、尚未排除的替代解释，例如代理问题、第三方波动、账号脏数据、环境配置错误、故障注入副作用等。只有这样，owner
才不会被“像真的”误导。

**10. 单人运营视角：Owner Review Layer**

由于该系统默认服务于“没有技术团队的
owner”，因此系统不能只输出技术报告，还必须输出给 owner
看得懂的裁决卡与总裁决页。

| **字段**   | **说明**                                        |
|------------|-------------------------------------------------|
| 问题名称   | 一句人话概括，不堆术语。                        |
| 真实度判断 | 已确认 / 高概率真实 / 可疑未证实 / 大概率误报。 |
| 影响等级   | 致命 / 高 / 中 / 低。                           |
| 证据硬度   | 硬 / 中 / 弱 / 很弱。                           |
| 主要依据   | 1-3 句说明为何更像真问题。                      |
| 最大疑点   | 为什么现在还不能 100% 下结论。                  |
| 建议动作   | 关闭 / 观察 / 让对方修 / 进入专项长时间审计。   |

| **单人模式下的分工** 系统负责取证与初判；AI 负责复核与解释；owner 负责商业裁决与是否继续投入资源。没有这层翻译，连续审计即使能跑，也无法被一个人稳定运营。 |
|------------------------------------------------------------------------------------------------------------------------------------------------------------|

**11. 数据飞轮：执行隔离，认知继承**

每次审计都应该有崭新的执行通道，但不应让知识层失忆。建议采用三仓分离。

| **仓**                   | **存什么**                                                            | **原则**                               |
|--------------------------|-----------------------------------------------------------------------|----------------------------------------|
| Session Runtime Store    | 当前会话状态、日志流、截图、录像、代理状态、告警事件。                | 强隔离；不直接复用到下一案执行初始化。 |
| Evidence Archive         | AuditPack、时间线、环境快照、结案结论、原始证据。                     | 用于追溯、争议、复核；不污染新案执行。 |
| Audit Intelligence Store | 误报模式、风险画像、证据充分性模板、升级结果、最有效 case card 模式。 | 只存经过蒸馏的经验，不存脏运行态。     |

必须有一条硬规则：历史知识只能影响验证优先级、证据采集策略与误报排查顺序，不得单独作为当前案件成立的依据。

**12. 交付、收费与经营结构**

连续审计不是“长时间等待后出一份报告”，而是“在固定时间窗口内持续产出阶段性交付物的专项审计”。

| **层**      | **建议内容**                                                             |
|-------------|--------------------------------------------------------------------------|
| 预约/立案层 | 测试授权书、环境冻结单、时间窗口、证据标准、提前终止规则、预算边界。     |
| 运行层      | 启动确认、每日或阶段性快照、异常即时上报、预算消耗提醒、owner 决策钩子。 |
| 结案层      | AuditPack、CaseVerdict、残余风险说明、是否建议复测与整改。               |

**12.1 收费结构建议**

> • 包月负责轻量持续治理，不应默认包含 72h 长时间审计。
>
> • 长时间审计应作为 add-on 或专项项目收费。
>
> • 收费宜采用：启动费 + 时间窗口费 + 资源费 + 风险附加费 + 复测费。
>
> • 对欧美市场，72h 审计更适合以专项窗口售卖，而不是塞进低价月费。

**13. 最小可落地版本（MVP）**

Codex
首轮实现不应试图把所有理想能力一次性做满。更现实的路线是先跑通一条最小闭环。

| **阶段**           | **必须具备**                                                                   | **暂时不要做满**                                   |
|--------------------|--------------------------------------------------------------------------------|----------------------------------------------------|
| Phase 1 - 最小闭环 | 会话创建、单场景执行、基础日志、定时截图、异常记录、证据包导出、owner 裁决卡。 | 多云调度、复杂代理编排、自动归因、完整人工值守台。 |
| Phase 2 - 运行质量 | 外部监控、告警分级、checkpoint、恢复策略、EscalationCaseCard、预算感知。       | 高拟真环境矩阵、并行大规模多案。                   |
| Phase 3 - 智能治理 | 知识蒸馏、误报模式库、自动化策略建议、复测推荐。                               | 全自动闭环自治。                                   |

**15. Codex 实施时必须遵守的硬边界**

> • 不要把整个 Continuous Audit Runtime 一开始就做成 skill；基础设施与
> runtime 应先做成系统组件。
>
> • 不要把 100% 拟真环境作为目标陈述；必须使用 Fidelity Declaration。
