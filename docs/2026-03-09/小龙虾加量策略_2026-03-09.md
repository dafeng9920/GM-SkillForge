# 小龙虾加量策略

- 日期：`2026-03-09`
- 目的：在不破坏治理硬度的前提下，逐步提高小龙虾的有效产能

## 一、当前判断

小龙虾当前定位：

- `可用，但仅限强约束执行层`
- `适合低风险、单模块、文档/报告/规范化类任务`
- `不适合首单验流程、架构设计、高风险代码修改`

当前基础证据：

- `REAL-TASK-002` 已冻结为首个小龙虾正式低风险样本
- 已知问题：
  - 上下文窗口易爆
  - 路径/环境口径易漂
  - 需要强合同和强状态词约束

## 二、加量原则

不是“多喂”，而是“按条件升级”。

### 允许加量的前提

1. 连续 `2` 单低风险任务通过：
   - `Review: ALLOW TO NEXT GATE`
   - `Compliance: PASS`
2. 不出现：
   - 假完成
   - 假闭环
   - 路径漂移
   - task_id 混用
3. 运行时统计三件套持续产出
4. 问题统计进入 `governance-telemetry` 可量化范围

## 三、加量分级

### Level A：当前级

- 单模块
- 低风险
- 文档/报告/规范化类
- 每次仅 `1-2` 个文件

### Level B：小幅加量

触发条件：

- `REAL-TASK-003-R1` 或后续等价 remediation 样本成功
- 且最近一单独立 `Compliance = PASS`

允许：

- 单模块
- 低风险
- 文档/报告/规范化类
- 每次 `2-4` 个文件
- 可同时产出更多 evidence / telemetry

### Level C：有限扩域

触发条件：

- 连续多单稳定
- `false_completion_rate` 下降
- `manual_intervention_per_task` 下降

允许：

- 文档/报告类之外
- 进入低风险测试补齐、非关键配置统一、patch proposal 类任务

### 禁止直接跳级

禁止从当前直接跳到：

- 复杂代码任务
- 跨模块任务
- 首单流程验证任务
- 真实 absorb / final accept 相关裁决任务

## 四、推荐喂任务类型

优先：

- Header / status / wording 一致性修正
- completion_record / review / compliance 文档规范化
- telemetry 报告整理
- patch proposal only 任务

慎用：

- shell/python 脚本规范化补丁提案
- 低风险测试补齐

禁止：

- 架构草案
- 合同设计
- Permit / final gate 裁决
- 真实业务代码复杂修改

## 五、失败回退规则

若任一单出现以下情况，立即回退到 `Level A`：

- 假完成
- 假闭环
- 任务号混用
- 未按 `PROCEED` 规则执行
- 未产出 runtime telemetry

若连续两单失败：

- 暂停小龙虾扩量
- 改由云端 Codex 回到“样板修正/首单验证”模式

## 六、当前建议

当前建议执行顺序：

1. `REAL-TASK-003` 已确认：
   - `Review: ALLOW TO NEXT GATE`
   - `Compliance: FAIL`
2. 先执行 `REAL-TASK-003-R1` remediation
3. remediation 通过前，保守留在 `Level A`
4. 用治理遥测记录本次失败原因：
   - `path_drift / boundary_breach`
   - `completion_record state pollution`

## 七、一句话原则

小龙虾的加量，不以“它看起来很快”为准，  
而以“它能否连续在强约束下稳定交付”为准。
