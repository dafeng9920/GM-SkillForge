## 2) 原有 6 大缺口（保持主线不变）

> 当前状态：以下 6 项均未整体达成，暂按 `待实现` 管理。

1. 原生命令化闭环（待实现）  
`skillforge` 一条命令跑完生成/审计/回归/发布候选，而非分散脚本。
2. 策略版本化（待实现）  
L1-L5/L6 规则、阈值、权重配置化，报告可追溯 `policy_version`。
3. 证据链硬约束（待实现）  
每次运行强制产出 `run_id/evidence_ref/input_hash/result_hash`，支持回放。
4. CI/CD 门禁自动化（待实现）  
PR/nightly 自动审计与回归，失败阻断，不允许人工兜底放行。
5. n8n 编排规模化（待实现）  
多任务并发、失败重试、降级策略、队列隔离稳定可观测。
6. 前端运营化看板（待实现）  
能直接支持放行决策：趋势、失败分层、证据钻取、复盘入口。

## 3) 新增可吸收能力（来自“网页端 GPT 对话内容”）

### A. 立即纳入主线（本阶段）

> 当前状态：以下能力方向已定，但未完成“可验收实现”。

1. 文档到 Skill 管线（Doc-to-Skill）（待实现）
- 流程统一为：`ingest -> parse -> contract -> gate -> render`。
- 目标：减少手工转译误差，让需求输入可标准化进入执行层。

2. 一致性契约（Consistency Contract）（待实现）
- 每次变更都必须声明：不变量、允许变化、重大变更触发条件。
- 目标：把“感觉一致”变成可机审规则。

3. 版本重置语义统一（待实现）
- 统一并落地：`iterate / rollback_reset / rebase_reset`。
- 目标：避免多人协作时“重跑/回滚”口径不一致。

4. 双通道交互模型（发散/执行）（待实现）
- 发散窗口只做建议与翻译；执行窗口才触发系统动作。
- 目标：降噪、保主线、保可审计。

5. 执行窗口拒绝权（Intent Not Buildable）（待实现）
- 对不构成 skill 的请求直接 `DENY`，并返回 `required_changes`。
- 目标：节省 token，减少无效执行。

## 4) 阶段门槛（L4.5 判定）

达到以下条件才可标记 “L4.5”：

1. 命令入口：`skillforge` 系统级命令可用，且支持一键全流程。
2. 强制证据：所有 run 默认带证据哈希链，可回放可追责。
3. 门禁自动化：CI/nightly 失败自动阻断发布。
4. 编排可靠性：并发、重试、降级、队列隔离均有压测证据。
5. 交互闭环：双窗口可用，执行窗口具备可解释拒绝。

## 5) 执行顺序（建议）

1. 先做基础硬化：系统级 `skillforge` + policy_version + 证据链强制化。
2. 再做治理闭环：CI 门禁 + 一致性契约 + 重置语义统一。
3. 再做交互闭环：双窗口 + 执行拒绝权 + 触发展示式前端。
4. 最后做规模验证：并发压测、重试/降级、长期稳定性观察。

## 6) 产出定义（每阶段都要有）

- 代码：功能实现与默认配置。
- 文档：规则口径、失败示例、运维手册。
- 证据：报告（JSON/MD/UI）+ gate_decision + 可回放索引。
- 验收：`ALLOW / REQUIRES_CHANGES / DENY` 明确结论。
