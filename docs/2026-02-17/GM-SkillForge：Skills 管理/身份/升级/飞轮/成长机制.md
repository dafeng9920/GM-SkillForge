GM-SkillForge：Skills 管理/身份/升级/飞轮/成长机制（精华对齐稿）
0) 一句话总纲

把 Skill 当作可审计的供应链资产来管理：
身份证（repo@commit → revision）+ 门禁（8 Gate）+ 证据（EvidenceRef）+ 交付（L3 AuditPack）+ 复现（at-time/tombstone）+ 成长（Experience Capture）。

1) 5 张卡片映射到 SkillForge 体系
01 Skills 管理的痛点

痛点：技能来源杂、版本漂移、不可复核、不可追责、升级易失控

对策：一切用 GateDecision + EvidenceRef + AuditPack 代替口头判断

02 给 Skills 一张身份证

身份=位置+版本指纹

repo_url + commit_sha (+ path)：外部不可变指纹（GitHub hash）

revision：系统侧版本指纹（绑定规则/口径/工具链）

目标：同一 repo@commit (+ at_time) 必须可复现同一 revision 与证据引用

03 你的专属技能管家（Skill Manager/Registry）

作用：把所有技能的状态台账集中管理：在哪、什么版本、是否可信、证据在哪

最小字段：skill_id, repo_url, commit_sha, revision, level, status, audit_pack_ref/hash, last_audited_at, tombstone_state, permit

边界：管家不做业务编排；只做治理入口与资产台账

04 打造自进化的大脑（3D RAG）

3D RAG 的正确拆分：

内核化（不可 Skill 取代）：at-time 复现口径 + tombstone 语义 + system-of-record/index schema

Skill 化（可替换）：抽取/切分/索引构建/检索策略/重排/评测

目标：历史可复现检索，且 tombstone 不可绕过

05 转动进化的飞轮（供应链闭环）

闭环：审计 → 修复（PR/required_changes）→ 复审 → 发布 → 升级重审 → tombstone

n8n 负责抓取/入队/发布/通知；SkillForge 负责裁决/证据/permit

核心硬规则：无 permit 不得发布/激活

2) 8 Gate = 8 棒位（同一骨架、现在工程化）

固定 8 Gate：

intake_repo（确定性快照、provenance）

license_gate

repo_scan_fit_score（识别/适配度）

draft_skill_spec（合同草案）

constitution_risk_gate（风险裁决）

scaffold_skill_impl（封装/骨架）

sandbox_test_and_trace（v0 最小化：契约/测试/trace）

pack_audit_and_publish（L3 AuditPack 固化 + 索引写入）

3) “升级命令”必须有：升级=候选 → 审计 → 放行

正确口径不是“先升级再审计”，而是：

先生成升级候选：指向新 commit_sha

再跑 8 Gate：产新 revision + L3 AuditPack

PASS 才签发 permit：n8n 才能发布/激活

建议落为 intent：upgrade_skill_revision（输入强制 target_commit_sha）。

4) evolution.json：Skill 的“进化存档”（不覆盖）

作用：记录每次审计/修复/升级的演进轨迹，形成可追溯“经验账本”

原则：append-only，绑定 revision，写入 AuditPack（system-of-record）

每次记录应能回指 IssueKey + EvidenceRef（避免“只有结论没有证据”）

5) Experience Capture（成长机制）必须有（已冻结）

目标：把每次审计/修复/测试产出的“经验点”自动沉淀，避免重复踩坑。
冻结口径：

从 GateDecision / IssueKey / diffs / logs / scan_report 自动提取经验

追加写入 evolution.json 与 SKILL.md

每条经验必须带 IssueKey + EvidenceRef

在 pack_audit_and_publish 时固化进 L3 AuditPack

append-only + 内容哈希去重

6) 角色分工（避免回到 old new-gm）

SkillForge（Governor/内核）：裁决、证据闭环、L3 pack、at-time/tombstone 口径、permit

n8n（Orchestrator）：抓取 GitHub、入队、开 PR、发布/同步、通知与重试

永远不变的硬约束：n8n 不裁决；无 permit 不发布

如果 Codex 只记住三句就够开工：

身份证：repo@commit(+path) → revision（系统口径指纹）

升级：候选 commit → 8 Gate → L3 pack → permit（PASS 才能发布）

成长：Experience Capture → evolution.json + SKILL.md → 固化进 L3 AuditPack（EvidenceRef 闭环，去重）
7) 执行补全（把蓝图变成可执行手册）

7.1 Skill 生命周期状态机（强制统一）

状态集合：
- draft
- candidate
- audited_passed
- permitted
- active
- deprecated
- tombstoned

状态迁移规则：
- draft -> candidate：提交候选（绑定 repo_url + commit_sha + revision）
- candidate -> audited_passed：8 Gate 全通过
- audited_passed -> permitted：签发 permit
- permitted -> active：由编排层执行激活（n8n 可触发）
- active -> deprecated：新 revision 激活后旧版退役
- deprecated -> tombstoned：满足下架条件并完成 tombstone 记录

禁止迁移：
- candidate 不能直接到 active
- audited_passed 不能绕过 permit 到 active
- tombstoned 不可回到 active

7.2 Registry 唯一键与冲突规则

最小唯一键：
- skill_id
- repo_url
- commit_sha
- revision

冲突处理：
- 同唯一键重复写入：拒绝覆盖（append-only），返回 CONFLICT
- skill_id 相同但 commit_sha 不同：视为新候选 revision
- revision 已存在但摘要哈希不同：判定为违规，REJECTED

必备索引字段：
- status
- permit
- audit_pack_ref
- audit_pack_hash
- tombstone_state
- last_audited_at

7.3 Gate 契约清单（每 Gate 必有 request/response）

每个 Gate 必须定义：
- request schema（输入字段、必填项）
- response schema（PASS/REJECTED、error_code、evidence_ref）
- fail_closed 条件
- evidence checkpoint（写入路径）

固定 8 Gate 的最低输出要求：
- intake_repo：provenance_ref
- license_gate：license_verdict + evidence_ref
- repo_scan_fit_score：fit_score + evidence_ref
- draft_skill_spec：spec_ref + contract_hash
- constitution_risk_gate：risk_verdict + constitution_ref/hash
- scaffold_skill_impl：artifact_ref + trace_ref
- sandbox_test_and_trace：test_verdict + trace_ref
- pack_audit_and_publish：audit_pack_ref/hash + permit_decision

7.4 发布闸门细则（permit 强制）

以下动作全部视为“发布动作”：
- git tag
- release publish
- runtime activate
- registry promote

统一规则：
- 无 permit：动作必须阻断（BLOCKED）
- permit 过期/不匹配 revision：动作必须阻断（REJECTED）
- permit 校验必须在 SkillForge 侧完成，n8n 不得自判

7.5 升级触发与最小回归要求

触发 upgrade_skill_revision 的条件：
- target_commit_sha 变化
- constitution/rubric 版本变化
- critical gate 规则变化
- 真实故障案例触发 required_changes

最小回归要求：
- 至少 1 个 PASSED 样例
- 至少 1 个 REJECTED 样例
- critical path 用例全通过
- evidence_ref 完整性 100%

升级放行条件：
- 8 Gate PASS
- L3 AuditPack 完整
- permit 已签发

否则：
- 保持旧 revision active
- 新候选标记 REJECTED，不得激活
8) Experience Capture（成长机制）执行条款（强制）

8.1 目标
- 自动提取每次审计/修复/测试产生的经验点，形成可追溯、可复用、可回放的经验资产。

8.2 经验点最小结构（必填）
```yaml
experience_item:
  issue_key: string            # 必填，问题键（如 GATE.FC-2 / RISK.L5）
  evidence_ref: string         # 必填，证据引用路径或 URI
  source_stage: string         # 必填，来源阶段（audit/fix/test）
  summary: string              # 必填，经验摘要
  action: string               # 必填，修复/防御动作
  revision: string             # 必填，绑定 skill revision
  created_at: string           # 必填，ISO-8601
  content_hash: string         # 必填，经验内容哈希
```

8.3 写入规则
- 追加写入 `evolution.json`（append-only，不得覆盖历史项）。
- 同步摘要写入对应 `SKILL.md` 的“经验增量”段。
- 任一经验点缺 `IssueKey` 或 `EvidenceRef`：拒绝写入（Fail-Closed）。

8.4 去重规则（内容哈希）
- 以 `content_hash` 作为去重主键。
- 命中同哈希：仅增加引用计数或关联 revision，不重复插入正文。
- 不同哈希：追加新条目。

8.5 固化时机（pack_audit_and_publish）
- 在 `pack_audit_and_publish` Gate 执行时，必须将以下文件固化进 L3 AuditPack：
  - `evolution.json`
  - `SKILL.md`（含经验增量）
  - `experience_index.json`（可选，用于检索）
- 若固化失败或字段缺失：Gate 结果为 `REJECTED`。

8.6 审计判定
- 通过条件：
  - 每条经验均具备 `IssueKey + EvidenceRef`
  - append-only 规则未被破坏
  - 内容哈希去重生效
  - 已固化进本次 L3 AuditPack
- 否则：`REJECTED`，不得签发 permit。
