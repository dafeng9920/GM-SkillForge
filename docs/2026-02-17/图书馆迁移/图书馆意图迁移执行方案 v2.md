# 图书馆意图迁移执行方案 v2

## 0. 目标与边界
目标：不拷旧代码，只迁移语义（intent semantics），把 old GM OS 的意图集合迁移为 SkillForge 的  
`Intent Catalog (contracts-first) + Gate Pipeline + L3 AuditPack`。

边界：
1. 迁移对象是“意图目录、意图合同、Intent->Gate 映射”。
2. 不迁移旧实现代码，不迁移旧 UI/外螺旋编排细节。
3. v0 北极星：`intent + repo@commit + at_time -> EvidenceRef -> L3 AuditPack`。

## 1. 迁移成功判定
同一个 `intent + repo_url + commit_sha + at_time`，稳定产出同一份 L3 AuditPack（关键字段与 manifest hash 可复现）。

## 2. 三件核心迁移资产
1. Intent 目录：GM OS 里有哪些意图（生成/发布/升级/审计/适配/回滚...）。
2. 每个 Intent 的合同：输入/输出/约束/必须证据/门禁等级。
3. Intent -> Gate Pipeline 映射：走哪些 Gate，失败 error_code，产什么 EvidenceRef。

## 2.1 组件级迁移对象（明确范围）
本轮“图书馆全量扫描”至少覆盖以下四类资产：
1. 三大哲学思想（原则层资产）
2. 53 个轴度（能力维度资产）
3. 10 大神器（方法/工具策略资产）
4. 四大基石（架构支撑资产）

## 2.2 四类资产的迁移策略（不是整包搬运）
1. 三大哲学思想：先迁为宪法附录与判定原则，不直接变执行代码。
2. 53 个轴度：先做轴簇化与参数化，首批仅迁 Top 10 高价值轴。
3. 10 大神器：先做“不可替代性评估”，仅迁通过准入闸门者。
4. 四大基石：必须先映射到 `Contract/Gate/Evidence/AuditPack/permit` 才可进入实现。

## 3. 最小迁移方案（v0）
### 3.1 建 Intent Catalog（contracts-first）
路径建议：`contracts/intents/<intent_id>.yml`

每个 intent 最小字段：
1. `intent_id`
2. `summary`
3. `contract_version`
4. `schema_hash`
5. `inputs_schema`（最小：`repo_url`, `commit_sha`，可选：`entry_path`, `manifest_path`, `at_time`）
6. `required_gates`（8 Gate 子集）
7. `minimum_level`（v0 统一 L3）
8. `required_evidence`（如 provenance、scan_report、diff、pack_manifest）
9. `outputs`（AuditPack、revision、tombstone_state）

### 3.2 固定“生成 skill”流水线（复用 8 Gate）
1. `intake_repo`（强制 `repo_url + commit_sha + at_time`）
2. `license_gate`
3. `repo_scan_fit_score`
4. `draft_skill_spec`
5. `constitution_risk_gate`
6. `scaffold_skill_impl`
7. `sandbox_test_and_trace`
8. `pack_audit_and_publish`

关键：不同 intent 只在策略与规则上变化，不重造流程骨架。

### 3.3 迁移映射表
文件建议：`docs/INTENT_MIGRATION_MAP.md` 或 `orchestration/intent_map.yml`

每条映射应包含：
1. 旧 GM OS Intent 名称/入口
2. 新 `intent_id`
3. Gate 路径
4. 失败 `error_code`
5. 必须产出的 `EvidenceRef` 列表
6. `migration_status`（`draft/mapped/validated/frozen`）

## 4. 今晚可执行最小清单（优先级）
1. 先落 3-5 个核心意图（不要全搬）：
- `audit_repo_skill`
- `generate_skill_from_repo`
- `upgrade_skill_revision`
- `tombstone_skill`

2. 把 `draft_skill_spec` 做成策略插件：
- `draft_skill_spec/strategies/<intent_id>.py`
- 输入：`scan_report + intent_contract`
- 输出：`skill_spec + required_changes`

3. Evidence 容器真实化：
- 落地 `evidence.jsonl` 或 `evidence/`
- 每条记录必须含：`IssueKey + EvidenceRef + hash + source_locator + tool_revision`

4. 回归黄金样例：
- 固定 1-2 个 repo fixture
- 锁定 `commit_sha`
- 断言 L3 pack manifest 关键字段稳定

## 5. 四个收口点（强制）
1. 每个 intent contract 必须包含 `contract_version` 与 `schema_hash`。
2. `INTENT_MIGRATION_MAP` 必须包含 `migration_status`（`draft/mapped/validated/frozen`）。
3. `pack_audit_and_publish` 必须校验 `required_evidence` 全量存在；缺失即 `REJECTED`。
4. 回归中必须验证：同输入二次运行 `AuditPack manifest hash` 一致（可复现硬验收）。

## 6. 防止旧复杂度回流
1. 只迁“意图语义 + 合同 + 映射”，不迁“旧实现细节”。
2. old GM OS 的 UI/外螺旋/五段式编排作为未来上层 skill，不进入 v0 审计内核。
3. n8n 仅编排，SkillForge 才能裁决；无 permit 不发布。

## 7. 验收出口
满足以下即视为 v0 迁移闭环成立：
1. 首批 intent（3-5）均可跑通 Gate。
2. 每个 intent 都能产完整 EvidenceRef 与 L3 AuditPack。
3. 同输入复跑 manifest hash 稳定。
4. 失败路径能稳定 `REJECTED`，且可追溯到 error_code + evidence。
