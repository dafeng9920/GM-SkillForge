# TASK-MAIN-04 前端承接完整链路任务定义

- 日期：2026-03-11
- 定位：下一波主线任务
- 目标：先用前端承接、展示、实测完整主链，再挑真实任务跑通成功样本

## 一、为什么前端要前置

当前主链已经具备：

- `Permit`
- `pre_absorb_check`
- `absorb`
- `local_accept`
- `final_accept`
- `RunResult / ArtifactManifest`
- `unified_validation_gate`

但还缺一个“人能看见、能点、能验证”的统一承接面。

所以 `TASK-MAIN-04` 不是再造治理件，而是把现有链路前置到一个可展示、可测试、可观察的前端壳里。

## 二、前端承接面

优先复用现有前端，不新开第二套：

- `ui/app/src/pages/execute/RunIntentPage.tsx`
- `ui/app/src/components/governance/`
- `ui/app/src/views/L4Workbench.tsx`

本轮前端承接只需要做到三件事：

1. 展示主链步骤
   - `Permit`
   - `pre_absorb_check`
   - `absorb`
   - `local_accept`
   - `final_accept`

2. 展示主链状态
   - `ALLOW / REQUIRES_CHANGES / DENY`
   - 当前步骤
   - 关键证据入口

3. 承接一次完整链路测试
   - 从前端发起一个低风险测试任务
   - 接到主链输出
   - 在前端显示完整结果

## 三、任务边界

本轮做的是：

- 前端链路壳
- 主链步骤展示
- 一次完整链路测试承接

本轮不做的是：

- 全量产品化前端重构
- 新设计系统
- 多页面扩展
- 云端链路
- SEO/量化支线接入

## 四、成功标准

满足以下条件即算 `TASK-MAIN-04` 成功：

1. 前端可视化展示完整主链
2. 至少一条测试链路可从前端触发或承接结果
3. 能看到 `RunResult / ArtifactManifest / Gate decision`
4. 不再只靠文档判断链路是否通

## 五、推荐拆分

### TASK-MAIN-04A

- 前端主链步骤展示

### TASK-MAIN-04B

- 前端接入主链状态与证据

### TASK-MAIN-04C

- 前端承接一次低风险完整链路测试

## 六、一句话

- 下一波主线先做“前端承接完整链路”，再用它去跑真正的完整成功样本。
