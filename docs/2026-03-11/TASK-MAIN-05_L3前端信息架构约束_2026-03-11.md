# TASK-MAIN-05 L3前端信息架构约束

- 日期：2026-03-11
- 适用任务：`TASK-MAIN-05`
- 定位：前端产品化重构前的硬约束文档

## 一、结论先行

系统真正需要的前端，不是单纯的：

- 展示页
- 工作台
- 审计面板

而是一个同时承接以下三套口径的统一前端体系：

1. **执行主链**
2. **五层审计维度**
3. **L3 封板摘要**

只有把这三层融合起来，前端才符合当前系统真实需要。

同时必须承认：

- 这是一个结构性矛盾

也就是：

- 前端必须建立裁决权威与可信感
- 前端又不能通过组件内容泄露系统核心能力、治理细节和绕过路径

这不是普通 SaaS 的信息多寡问题，而是裁决系统天然存在的“可信展示 vs 机制保护”矛盾。

## 二、三套口径分别是什么

### 1. 执行主链

来自当前主线装配结果，关注“任务如何被执行”：

- `Permit`
- `pre_absorb_check`
- `absorb`
- `local_accept`
- `final_accept`

这是 **运行链**。

### 2. 五层审计维度

来自统一审计口径，关注“一个执行结果是否可信”：

- `L1 合同审计`
- `L2 控制审计`
- `L3 安全姿态`
- `L4 证据审计`
- `L5 复现审计`

这是 **审计链**。

### 3. L3 封板摘要

来自历史 L3 总控材料，关注“系统是否达到可封板状态”：

- A 治理门禁
- B 意图合同
- C 证据审计
- D 复现性
- E Skill + CI
- F 指标
- G 演练
- H 文档签核

这是 **封板链**。

## 三、必须明确的原则

### 原则 1：三层不能混成一层

前端必须明确区分：

- 执行链
- 审计链
- 封板链

不能把：

- “某一步执行成功”

误写成：

- “五层审计全部通过”

更不能误写成：

- “L3 已封板”

### 原则 2：外部展示页与内部工作台必须拆开

#### 外部展示 / 落地页

只负责：

- 展示系统价值
- 展示可信执行能力
- 展示可审计、可追溯、可复核

不负责：

- 展示全量 Gate 细节
- 展示 EvidenceRef 列表
- 展示内部签核和 drill 细节

#### 内部工作台

负责：

- 展示完整执行主链
- 展示五层审计维度
- 展示 L3 封板摘要
- 展示证据入口、Gate 状态、运行结果

### 原则 3：前端必须支持“部分成立”状态

系统经常会出现：

- 执行链部分完成
- 五层审计部分完成
- 封板链未完成

所以前端必须天然支持：

- `pending`
- `running`
- `pass`
- `fail`
- `deny`
- `partial`
- `not_applicable`

不能只用“成功 / 失败”两态。

### 原则 4：展示可信性，不展示核心机制

前端应该优先展示：

- 裁决结果
- 当前状态
- 可复核承诺
- 证据存在性

前端不应默认展示：

- 完整规则树
- 判定阈值
- 模块调用链
- 阻断触发条件
- 可被逆向的内部机制

前端展示的是：

- **可信裁决**

不是：

- **防线源码**

## 四、系统需要的前端结构

## 4.1 对外展示页

目标：

- 可上线
- 可介绍 SkillForge
- 可建立信任

必须包含的模块：

1. **价值主张 Hero**
   - SkillForge 是什么
   - 为什么可信
   - 为什么不是普通 AI workflow

2. **能力摘要**
   - 可执行
   - 可审计
   - 可复现
   - 可签核
   - 可裁决

3. **主链能力示意**
   - 用摘要方式展示执行主链
   - 但不展开内部治理细节

4. **L3 能力摘要**
   - 用摘要方式展示：
     - Gate
     - Evidence
     - Reproducibility
     - Signoff

5. **真实结果样本入口**
   - 链接到内部或受控 demo
   - 展示真实已有链路结果

## 4.2 内部工作台

目标：

- 真正给主控和 AI 军团使用
- 承接真实结果
- 做日常验证和封板判断

必须包含的模块：

1. **执行主链面板**
   - `Permit -> pre_absorb_check -> absorb -> local_accept -> final_accept`
   - 每一步显示：
     - 状态
     - decision
     - evidence_refs
     - timestamp

2. **五层审计面板**
   - `L1-L5`
   - 每层显示：
     - 当前状态
     - 关键检查项
     - 主要阻断点

3. **L3 封板摘要面板**
   - A-H 八类摘要
   - 每类显示：
     - PASS / FAIL / PARTIAL
     - 关键 EvidenceRef
     - 是否阻断封板

4. **运行结果面板**
   - `RunRequest`
   - `RunResult`
   - `ArtifactManifest`

5. **证据抽屉 / 证据详情**
   - 可点击展开 EvidenceRef
   - 支持 source locator / hash / artifact path

6. **Final Gate 裁决位**
   - 显示：
     - `ALLOW`
     - `REQUIRES_CHANGES`
     - `DENY`
   - 并明确裁决范围

## 五、必须具备的能力组件

前端产品化重构时，以下组件属于必备，不是可选增强：

### A. Main Chain Step Panel

作用：

- 展示执行主链五步

### B. Main Chain Status Panel

作用：

- 承接真实 `RunResult / ArtifactManifest / Gate` 状态

### C. Audit Dimension Panel

作用：

- 展示 `L1-L5` 五层审计维度

### D. Seal Readiness Panel

作用：

- 展示 L3 封板摘要
- 汇总 A-H 维度

### E. Evidence Drawer

作用：

- 打开证据详情
- 提供 hash / locator / path / decision 上下文

### F. Final Gate Decision Card

作用：

- 在 UI 中明确显示当前最终裁决
- 不让用户自己猜

### G. Scope Statement Card

作用：

- 明确本次结果到底证明了什么
- 没证明什么

这个组件非常重要，因为它防止前端把“部分成功”误展示成“全部完成”。

## 六、页面层级建议

### Page A: Landing

对外展示页

### Page B: Workbench

内部工作台首页

### Page C: Audit Detail

审计维度详情页

### Page D: Evidence Detail

证据详情页

### Page E: Seal Readiness

L3 封板准备度页

### Page F: Controlled Detail

受控详情层

说明：

- 只向内部高权限角色开放
- 允许看审计摘要与证据摘要
- 不允许看完整规则树、完整阈值、完整绕过路径

## 七、哪些内容必须受控展示

以下内容不能默认暴露在对外页：

- 内部文件路径
- 全量 EvidenceRef
- permit 原文
- 环境变量
- 内部任务元数据
- 细粒度 Gate 日志
- 规则树
- 判定阈值
- 内部脚本名
- 内部服务标识符
- 模块调用顺序
- 可被模仿的阻断方式

这些内容只能出现在：

- 内部工作台
- 审计详情
- 受控 demo

以下内容即使在内部工作台也不能默认直接展开：

- 完整规则树
- 完整评分权重
- 完整调用拓扑
- 可被直接逆向的安全策略细节

这些内容若必须显示，只能出现在：

- 受控详情层
- 且按角色裁剪返回

## 八、前端暴露最小化原则

### 1. 结论先行，机制沉底

前端先展示：

- 结果
- 状态
- 裁决
- 证据存在性

后展示：

- 审计摘要
- 证据摘要

不直接展示：

- 机制细节

### 2. 名称可见，逻辑不可见

可以展示：

- `L3 安全姿态`
- `L4 证据审计`
- `Final Gate`

不应直接展示：

- 它们内部到底检查了哪些规则
- 规则如何组合
- 规则如何命中

### 3. 证据可复核，链路不可逆向

前端可以让用户知道：

- 有证据
- 证据存在
- 证据可复核

但不能让用户通过前端直接反推出：

- 证据是由哪个内部节点、哪个规则组合、哪个敏感调用产生的

### 4. 权限控制应在数据层完成

同一页面不同角色看到的信息密度可以不同。

但必须通过：

- API / 数据层裁剪

而不是仅靠：

- 前端隐藏 DOM

### 5. 权威感来自形式，不来自泄密

权威感应来自：

- 清晰裁决
- 稳定结构
- 明确状态
- 时间戳
- EvidenceRef 存在性
- 封板感与归档感

而不来自：

- 塞更多内部字段
- 展示更深的治理细节

## 九、对 TASK-MAIN-05 的直接约束

### 对 05A 的约束

- 不能把工作台直接改个皮就当落地页
- 必须做独立的信息结构
- 必须强调“可信裁决”和“可复核承诺”
- 不得展开完整治理细节

### 对 05B 的约束

- 必须承接：
  - 执行主链
  - 五层审计维度
  - L3 封板摘要
- 必须支持摘要视图与受控详情视图分层
- 不能把执行成功误写成封板成功

### 对 05C 的约束

- 必须接真实已有结果
- 不得再走 mock
- 不得只接执行主链而忽略审计/封板摘要
- 必须明确“承接真实已有结果”与“实时触发新执行”的边界

## 十、最终总原则

- 前端负责出示 **可信裁决** 与 **可复核证据**，不负责暴露内部机制与防线细节。

## 十一、最终一句话

- 系统需要的前端，不是一张页面，而是一个同时承接 **执行链、审计链、封板链** 的双面前端体系：
  - 对外可上线
  - 对内可裁决
