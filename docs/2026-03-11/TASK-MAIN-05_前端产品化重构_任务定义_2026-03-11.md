# TASK-MAIN-05 前端产品化重构任务定义

- 日期：2026-03-11
- 定位：下一波主线任务
- 前提：
  - `TASK-MAIN-04 = ALLOW`
  - 当前前端已能承接真实已有链路结果
  - 当前前端仍不具备上线展示质量

## 一、任务结论先行

当前前端只能算内部原型，不适合作为：

- 对外展示页
- 产品落地页
- 可上线页面

因此下一波主线不是继续补主链文档，而是做：

- **前端产品化重构**

## 二、任务目标

把当前前端拆成两条明确产品线：

### 1. 外部展示 / 落地页

用途：

- 对外展示 SkillForge
- 讲清产品价值
- 可上线、可演示、可作为入口页

### 2. 内部工作台

用途：

- 承接真实主链结果
- 展示 `Permit -> pre_absorb_check -> absorb -> local_accept -> final_accept`
- 供主控和 AI 军团日常使用

## 三、严格边界

本轮必须避免的错误：

- 不再把“内部工作台”做成“对外落地页”
- 不再把“对外展示页”塞满内部治理信息
- 不再把当前 UI 原型直接当成可上线页面

## 四、拆分任务

### TASK-MAIN-05A

名称：

- `外部展示/落地页重构`

目标：

- 做一个真正可上线的 SkillForge 展示页
- 讲清产品定位、能力、主链价值、使用入口

最低要求：

- 视觉达到上线标准
- 信息结构清晰
- 移动端可用
- 不是工作台翻版

### TASK-MAIN-05B

名称：

- `内部工作台 UI 重构`

目标：

- 重构当前工作台视觉与布局
- 保留主链承接能力
- 让状态、证据、执行结果可读可用

最低要求：

- 完整主链状态可视化
- 真实结果承接不丢
- 不再“丑得不能展示”

### TASK-MAIN-05C

名称：

- `重构后工作台接入真实已有结果`

目标：

- 把 `TASK-MAIN-04` 的真实结果承接能力接到新工作台 UI
- 不再依赖旧页面壳

最低要求：

- 可加载真实 `RunResult`
- 可加载真实 `ArtifactManifest`
- 可展示 Gate 状态

## 五、设计原则

### 对外页

- 强调价值与可信度
- 少术语，多结果
- 强视觉，不做表格堆砌

### 内部台

- 强结构
- 强状态
- 强证据入口
- 强可测性

## 六、成功标准

满足以下条件即算 `TASK-MAIN-05` 成功：

1. 外部展示页达到可上线标准
2. 内部工作台达到可日常使用标准
3. 重构后工作台仍能承接真实已有链路结果
4. 对外页和内部台不再混用

## 七、一句话

- 下一波主线不是继续补“链路能不能跑”，而是把前端重构成“能上线展示”和“能内部使用”的两张产品脸。
