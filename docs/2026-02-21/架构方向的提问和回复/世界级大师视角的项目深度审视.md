## 待审视项目

### 项目名称：SkillForge (GM-SkillForge)

### 项目定位

SkillForge 是一个 **AI Skill 生产线平台**，核心理念是将"原始代码"转化为"可审计、可治理、可复现的生产级 Skill 工件"。

### 核心能力

1. **8-Gate Pipeline（七门流水线）**
   - G1: intake_repo（仓库摄入）
   - G2: license_gate（许可证门禁）
   - G3: repo_scan_fit_score（适配度评分）
   - G4: draft_skill_spec（技能规范草稿）
   - G5: constitution_risk_gate（风险门禁）
   - G6: scaffold_skill_impl（脚手架实现）
   - G7: sandbox_test_and_trace（沙箱测试追踪）
   - G8: pack_audit_and_publish（打包审计发布）

2. **Permit 治理系统**
   - Permit 签发（Issuer）
   - Permit 校验（Gate）
   - TTL 管理、签名生成
   - Fail-Closed 机制：无 Permit 不发布

3. **Skill 封装体系**（已实现 11 个 Skill）
   - permit-governance-skill
   - release-gate-skill
   - rollback-tombstone-skill
   - ci-skill-validation-skill
   - master-signoff-skill
   - drill-evidence-collector-skill
   - experience-template-retriever-skill
   - task-pack-writer-skill
   - ai-response-improvement-skill
   - seed-gate-validator-skill
   - example-data-processing-skill

4. **Tombstone 持久化**
   - Append-Only Log
   - 7 年保留期
   - at-time 回放能力

5. **n8n 编排集成**
   - 3 个核心能力：run_intent / fetch_pack / query_rag
   - 边界契约：n8n 仅触发/路由，SkillForge 负责最终裁决

6. **Experience Capture**
   - 自动捕获执行经验
   - evolution.json 持久化
   - 66 条经验已固化

### 技术栈

- Python 为主
- JSON Schema (Draft 2020-12) 契约
- Pytest 测试框架
- n8n 工作流引擎
- HS256 JWT 签名
- OPA/IAM 集成

### 当前状态（2026-02-20）

- ✅ L4 已收口（前后端三链路联调完成）
- ✅ L4.5 已开工（n8n 接入、Fail-Closed 验证通过）
- ✅ 76/76 测试通过

### 核心设计原则

1. **Fail-Closed**：无 permit 不发布，tombstone 默认不可见
2. **合同优先**：所有接口必须符合 JSON Schema
3. **单向依赖**：SkillForge → gm-os-core，禁止反向
4. **可审计性**：run_id / evidence_ref / permit_id 全链追溯

### 已完成验证

- Batch Release 多目标 all-or-nothing
- at-time 回放一致性
- E001/E003 阻断语义验证

## 审视任务

### 一、产品维度 - "这是人们真正需要的东西吗？"

1. **核心价值主张**：用一句话说清楚 SkillForge 解决了什么问题？如果不存在的世界里，开发者会如何痛苦？
2. **目标用户画像**：谁会为此付费？谁会用后爱上它？谁会因为复杂而放弃？
3. **竞争优势**：与 LangChain、AutoGPT、Dify、Flowise 等现有解决方案相比，不可替代的价值是什么？
4. **用户体验摩擦点**：新用户从"不知道 SkillForge"到"能发布第一个 Skill"，需要多少步骤？哪里会流失？

### 二、架构维度 - "这能撑得起 10 年吗？"

1. **架构清晰度**：8-Gate Pipeline + 11 个 Skill + Permit 系统 + Tombstone，这个复杂度是必要的吗？新人能在 30 分钟内理解系统全貌吗？
2. **依赖健康度**：n8n / OPA / IAM / gm-os-core 这些依赖是否可控？有没有"单点故障"？
3. **扩展瓶颈**：如果 Skill 数量增长到 1000 个，如果并发请求量增长 100 倍，哪里会先崩溃？
4. **技术债务清单**：当前设计中"临时方案"有哪些？什么时候会反噬？

### 三、代码与工程维度 - "代码是资产还是负债？"

1. **认知负载**：理解核心流程需要读多少代码？"Fail-Closed"、"at-time 回放"、"Permit 签发链"这些概念之间的关系，新人能理解吗？
2. **测试信心**：76/76 测试通过，但这些测试能捕获多少真实 bug？重构时敢动吗？
3. **可观测性**：出问题时能快速定位吗？还是有太多"黑盒"？
4. **文档与现实**：40+ 个文档说的是真相吗？还是已经和代码脱节？

### 四、亮点挖掘 - "这个项目做对了什么？"

1. **独特创新点**：Permit 治理、Fail-Closed、at-time 回放，这些设计是业界首创还是常见模式？有什么独特价值？
2. **防御性优势**：哪些决策让竞争对手（如 Dify、Flowise）难以复制？
3. **可复用资产**：哪个 Skill 可以独立开源或商业化？

### 五、关键问题与洞察 - "如果你是 CEO/CTO，你会优先改什么？"

1. **Top 3 紧急问题**：不解决会在未来 3 个月内造成实际损害的问题
2. **Top 3 长期风险**：可能在 1 年后爆发的隐患
3. **一个"杀死"建议**：如果有一个功能/模块应该被删除或大幅简化，是什么？
4. **一个"押注"建议**：如果只能投入资源做一个改进，做什么收益最大？

## 审视原则

- **残酷诚实**：不要客气，不要"三明治法"，问题越尖锐越好
- **穿透表象**：不要只看"有没有"，要看"好不好用"
- **用户视角**：始终从使用者角度思考，而非开发者自我感动
- **长期主义**：短期可妥协，长期不可逆的决策要特别警惕
- **对标业界**：与 LangChain、Dify、Flowise、AutoGen 等竞品对比思考
