# SkillForge 第 2 批施工单

适用范围：
- 对应第 1 批已完成后的第 2 批主链
- 目标是打通“裁决链”，不进入第 3 批连续审计 runtime

本批目标：
- 让系统从“会发现问题”升级到“能说明为什么放 / 为什么不放”
- 建立规则主裁决、判断覆盖例外、例外结构化留痕的最小闭环
- 让问题产出进入可交付、可整改、可治理的前半段资产链

本批完成后必须得到的结果：
- 可以把 `findings.json` 转成标准化裁决对象
- 可以输出：
  - `adjudication_report.json`
  - `coverage_statement.json`
  - `release_decision.json`
  - `owner_review.json`
  - `issue_records.json`
  - `fix_recommendations.json`
  - `audit_pack.json`

本批明确不做：
- 不做连续审计 runtime
- 不做 revision invalidation 全链
- 不做 residual risk writeback 全链
- 不做 Permit / Release 完整运行时闭环
- 不做 DistilledPattern 全知识飞轮

---

## 施工单 1：Finding Adjudication 与 RuleDecision

### 目标
把 finding 从“发现对象”推进成“可裁决对象”。

### 负责模块
- Finding Adjudicator
- RuleDecision

### 交付物
- `adjudication_report.json`
- `rule_decisions[]`

### 开发内容
- 定义 adjudication schema
- 为每条 finding 生成：
  - truth_assessment
  - impact_level
  - evidence_strength
  - primary_basis
  - largest_uncertainty
  - recommended_action
- 定义规则直出结论

### 完成证据
- 固定 findings 样例可生成稳定 adjudication 输出

### 验收标准
- 相同输入多次运行输出一致
- 无 evidence 的 finding 不得进入 adjudication 成功态

### 当前判定
- 当前必须实现

---

## 施工单 2：CoverageStatement 与 EvidenceLevel

### 目标
把“测了什么、没测什么、证据有多强”正式对象化。

### 负责模块
- Coverage Statement Builder
- Evidence Level Builder

### 交付物
- `coverage_statement.json`
- `evidence_level.json`

### 开发内容
- 声明已覆盖的风险类别
- 声明未覆盖的上下文/组合态
- 定义证据等级 E1-E5 最小口径

### 完成证据
- 给固定审计 session 生成 coverage / evidence level 两份产物

### 验收标准
- 未覆盖项必须显式列出
- 不允许把未覆盖默认写成已完成

### 当前判定
- 当前必须实现

---

## 施工单 3：JudgmentOverride / ResidualRisk / ReleaseDecision

### 目标
建立“规则主裁决、判断覆盖例外”的最终放行对象。

### 负责模块
- Judgment Override
- Residual Risk
- Release Decision
- Escalation Gate

### 交付物
- `judgment_overrides.json`
- `residual_risks.json`
- `release_decision.json`

### 开发内容
- 定义哪些场景允许进入 judgment override
- 定义 override 留痕结构
- 输出最终裁决：
  - Reject
  - Escalate
  - Conditional Release
  - Limited Release
  - Release

### 完成证据
- 至少 3 组样例：
  - 规则直出拒绝
  - 条件放行
  - 升级专项验证

### 验收标准
- contract 失败 / 无证据 finding / 明确高危命中，不允许被判断洗白
- judgment override 必须有结构化理由和证据

### 当前判定
- 当前必须实现

## 施工单 4：Owner Review Layer

### 目标
让 owner 能基于裁决结果做业务判断，而不是读技术原始报告。

### 负责模块
- Owner Review Builder

### 交付物
- `owner_review.json`
- `owner_review.md`

### 开发内容
- 二层报告制
- 单 finding 裁决卡固定 7 字段
- owner summary

### 完成证据
- 对固定 adjudication 输入生成 owner review 结果

### 验收标准
- owner review 不得丢失 blocking findings
- 不得把技术 finding 原样堆给 owner

### 当前判定
- 当前必须实现

### 验收标准

- 当前必须实现

## 施工单 7：第 2 批总装与最小 AuditPack

### 目标
把第 2 批所有对象串成一条“发现 -> 裁决 -> 交付”的最小闭环。

### 负责模块
- AuditPack Builder
- Batch 2 integration

### 交付物
- `audit_pack.json`
- 回归样例
- 统一命令

### 开发内容
- 串起 findings -> adjudication -> release decision -> owner review -> issue/fix -> audit_pack
- 固定输出目录
- 至少 3 组回归样例

### 完成证据
- 一条统一命令可稳定输出最小 AuditPack

### 验收标准
- 不需要人工拼接
- 运行日志与 EvidenceRef 一致
- 不引入第 3 批能力

### 当前判定
- 当前必须实现
