# 云端治理资产纯载荷包模板

- 日期：`2026-03-08`
- 用途：本地 Codex 向 Gemini 交付可上云的治理资产纯载荷
- 适用模式：`PAYLOAD-FIRST`

## 1. 使用说明

当本地无法直连云端，或决定不依赖直连时，使用本模板组织一份**可审计、可桥接、可验证**的治理载荷包。

本模板不要求云端能访问本地主仓。  
它要求的是：

- 本地给出完整载荷
- Gemini 负责桥接上云
- 云端按真实落点消费

## 2. 建议目录结构

```text
payloads/<payload_id>/
├── payload_manifest.json
├── INSTALL_ORDER.md
├── VERIFY_STEPS.md
├── skills/
│   ├── gm-multi-agent-orchestrator-skill/
│   ├── lobster-task-package-skill/
│   ├── lobster-absorb-gate-skill/
│   └── lobster-cloud-execution-governor-skill/
├── scripts/
│   ├── verify_governance_env.sh
│   ├── pre_absorb_check.sh
│   └── absorb.sh
└── docs/
    ├── CODEX_SYNC_GOVERNANCE_v1.md
    ├── CLOUD_SKILL_SYNC_INSTALL_PROTOCOL_v1.md
    └── LOBSTER_TASK_PACKAGE_SPEC.md
```

## 3. payload_manifest.json 模板

```json
{
  "payload_id": "governance-payload-2026-03-08-v1",
  "generated_at": "2026-03-08T20:30:00Z",
  "authoritative_source": "SSOT_ROOT",
  "delivery_mode": "PAYLOAD_FIRST",
  "bridge_operator": "Gemini",
  "execution_consumer": "Cloud Codex",
  "targets": {
    "skills_root": "APP_ROOT/skills or DROPZONE_ROOT/skills_sync",
    "scripts_root": "CLOUD_BOX/scripts",
    "docs_root": "DOCS_ROOT/2026-03-08"
  },
  "skills": [
    "gm-multi-agent-orchestrator-skill",
    "lobster-task-package-skill",
    "lobster-absorb-gate-skill",
    "lobster-cloud-execution-governor-skill"
  ],
  "scripts": [
    "verify_governance_env.sh",
    "pre_absorb_check.sh",
    "absorb.sh"
  ],
  "docs": [
    "CODEX_SYNC_GOVERNANCE_v1.md",
    "CLOUD_SKILL_SYNC_INSTALL_PROTOCOL_v1.md",
    "LOBSTER_TASK_PACKAGE_SPEC.md"
  ],
  "expected_outputs": [
    "cloud_skill_sync_report.md",
    "installed_paths.txt"
  ],
  "status": "READY_FOR_BRIDGE"
}
```

## 4. INSTALL_ORDER.md 模板

```md
# INSTALL ORDER

1. 创建技能目标目录
2. 同步 4 个 skills
3. 同步 3 个 scripts
4. 同步 3 个 docs
5. 验证每个 SKILL.md 存在
6. 验证每个 shell 脚本存在
7. 运行 verify_governance_env.sh
8. 生成 cloud_skill_sync_report.md
```

## 5. VERIFY_STEPS.md 模板

```md
# VERIFY STEPS

## Skill Checks
- ls -la <skills_root>/gm-multi-agent-orchestrator-skill/SKILL.md
- ls -la <skills_root>/lobster-task-package-skill/SKILL.md
- ls -la <skills_root>/lobster-absorb-gate-skill/SKILL.md
- ls -la <skills_root>/lobster-cloud-execution-governor-skill/SKILL.md

## Script Checks
- ls -la <scripts_root>/verify_governance_env.sh
- ls -la <scripts_root>/pre_absorb_check.sh
- ls -la <scripts_root>/absorb.sh

## Runtime Check
- bash <scripts_root>/verify_governance_env.sh
```

## 6. Gemini 回传要求

Gemini 回传时必须至少给出：

1. `cloud_skill_sync_report.md`
2. 每个 skill 的实际落点
3. 每个 script 的实际落点
4. 至少一次真实存在性验证
5. `INSTALLED / PARTIAL / BLOCKED` 总结

## 7. 云端 Codex 使用前提

云端 Codex 只有在以下条件全部满足时才可开始执行：

1. 所需 skill 已真实落地
2. `SKILL.md` 可读取
3. 所需脚本已真实存在
4. `verify_governance_env.sh` 至少跑过一次

## 8. 失败处理模板

若桥接失败，Gemini 必须按以下格式回传：

```text
payload_id: governance-payload-2026-03-08-v1
status: BLOCKED
blocked_step: scripts_sync
reason: target path read-only
partial_success:
- gm-multi-agent-orchestrator-skill installed
- lobster-task-package-skill installed
missing:
- absorb.sh
- pre_absorb_check.sh
```

## 9. 最小完成定义

以下 4 条同时满足，才算本载荷包完成：

1. 至少 3 个新 skill 已在云端有真实副本
2. 3 个桥接脚本已在云端有真实副本
3. 有最小存在性验证输出
4. `cloud_skill_sync_report.md` 已生成

## 10. 备注

本模板服务于当前制度：

- 主仓 authoritative
- Gemini 桥接
- 云端 Codex 消费
- 本地 absorb + final accept

它不是临时补丁，而是当前网络不稳定条件下的正式治理载荷模板。
