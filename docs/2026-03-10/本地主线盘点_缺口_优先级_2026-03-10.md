## 三、主要缺口

### P0 缺口：主链后半段未统一代码化

当前缺口：

- `local accept` 没有统一脚本或统一接口入口
- `final accept` 没有统一脚本或统一接口入口

影响：

- 现在能跑到 `absorb`
- 但 `accept` 仍主要依赖文档口径和主控记录

### P0 缺口：最小运行接口未统一

当前缺口：

- 未看到统一、稳定、明确落地的：
  - `RunRequest`
  - `RunResult`
  - `ArtifactManifest`

影响：

- 各条执行链仍偏“任务定制化”
- 不利于本地多Agent并行统一接入

### P1 缺口：主链分散

当前状态：

- `Permit`、三哈希、delivery、absorb 分别散落在多个脚本和文档中

影响：

- 单点都存在
- 但作为“本地主线统一入口”还不够集中

### P1 缺口：验证入口分散

当前状态：

- `validate_permit_binding.py`
- `validate_three_hashes.py`
- `pre_absorb_check.sh`
- `antigravity_2_guard.py`
- 其他若干验证脚本并存

影响：

- 验证能力不少
- 但对当前主线来说，缺一个“最小统一验证入口”

### P2 缺口：环境口径仍有历史分叉

当前状态：

- `/root/openclaw-box/...`
- `/home/node/.openclaw/...`

影响：

- 已不再阻断单次执行
- 但仍会污染理解和后续接入成本

## 四、优先级排序

### Priority 0

1. 补齐 `local accept / final accept` 的统一入口  
2. 固化最小运行接口：`RunRequest / RunResult / ArtifactManifest`

### Priority 1

3. 形成“本地主链统一入口”  
4. 形成“本地统一验证入口”

### Priority 2

5. 收口环境历史分叉口径  
6. 继续清理旧文档与旧口径残留

## 五、最现实的推进顺序

先做：

1. 把主链后半段代码化
2. 把运行接口统一

再做：

3. 把验证入口收成一个主入口
4. 再收环境口径和旧文档

原因很直接：

- 现在最缺的不是“更多脚本”
- 而是“把已有能力收成统一主线”

## 六、结论

当前本地主线不是从零开始，  
而是已经有一半以上真实落点：

- `Permit` 有

- 最小运行接口还没统一

## 七、一句话

本地主线当前最该优先补的，不是再造新制度，  
而是把已经存在的 `Permit -> pre_absorb_check -> absorb` 前半段，
