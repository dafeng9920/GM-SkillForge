我会把这份方案压成一套“治理与放行中枢”的前端设计，不会滑向普通 builder。先锁定产品气质，再展开 IA、关键页面、用户路径、状态与权限边界。

Thought for 39s

结论：
GM-SkillForge 的前端不该被设计成“AI builder”或“skill marketplace”，而应被设计成一个 治理中枢 + 审计工作台 + 放行控制层。
对外，它卖的是“可审、可判、可放、可复现”；对内，它展示的是“证据链、8 Gate、Permit、三权分立、版本追踪”。
最优前端策略是：首页讲价值与边界，应用内讲状态与裁决，核心界面围绕 Audit → Gap → Permit 展开，Forge 只是中间站，不是舞台中心。

置信度：0.91

关键风险点：

最容易做歪成“普通 builder”：把画布编排、节点拖拽、生成按钮放到视觉中心。

最容易失去可信度：权限边界不清，Execution/Audit/Compliance 混在一个页面里操作。

最容易过度设计：太早做企业门户、社区、模板市场、多角色协同大系统，反而削弱 v0/v1 的锋利定位。

A. 整体前端定位

推荐定位：审计中枢（Audit Control Center） + Release Gate System
不是 skill factory 作为主叙事，也不是 builder console 作为主叙事。

为什么这样定

你的系统真正差异化的价值，不在“做出 skill”，而在：

skill 做出来之后，能不能被验证

能不能被裁决

能不能被放行

出问题时能不能追溯

未来能不能按同一口径复现

所以前端气质应更接近：

控制台：因为它有状态、门禁、裁决、证据和历史

审计中枢：因为核心是 evidence、gap、decision、permit

release gate system：因为最终用户要的是“能不能放”

不建议主气质做成：

纯 skill factory：会让人以为你是生成器

纯 workflow builder：会把价值重心错放到搭积木

纯 developer dashboard：会太偏工具，不够传达治理与商用价值

一句话气质建议：
“像一个工业级发布控制层，而不是一个 AI 小工具站。”

B. 整体信息架构（IA）
一级导航建议
1. Overview

存在必要性：
作为总览中枢，集中展示当前资产池、待审项目、风险分布、Permit 状态、Gate 健康度。
这是用户进系统后第一眼理解“这不是 builder，而是治理控制层”的关键页面。

2. Intake / Prompt Pack

存在必要性：
承接输入态和孵化态。这里不是让用户随便 brainstorm，而是把模糊需求整理成可进入 Forge 的结构化输入。
它的任务是把“想法”推到“可治理输入”。

3. Forge

存在必要性：
这是从需求到正式结构的炼钢层。
页面职责不是炫技生成，而是展示：DemandSpec、TestSpec、ControlSpec、Manifest、Contract、Hashes 等如何被压缩成可审结构。

4. Audit

存在必要性：
这是核心模块。用户真正愿意为之付费的是这里。
包括 8 Gate 裁决、EvidenceRef、红线、可修复项、失败原因、证据链闭环。

5. Release / Permit

存在必要性：
这是系统的商业闭环页面。
它把“通过”变成“可放行”，把“审计完成”变成“被批准上线/交付”。
Permit 是最强价值锚点之一。

6. Registry

存在必要性：
存放已认证、待修复、已下架、已废弃、历史版本等资产。
让用户看到系统不是一次性审计，而是持续治理。

7. History / Cases

存在必要性：
记录 revision、decision、hash、gap、permit、tombstone。
这是复现、追责、handoff、客户交付的底座。

8. Policies

存在必要性：
展示 Constitution / Contract / Rules / Gate policy / Error policy。
这里决定系统“可信且可解释”。

9. Team / Approval

存在必要性：
即使 v1 不完整，也必须预留。
因为团队交付、企业审批、合规签发最终都会走这里。
没有这个入口，B2B 感会弱很多。

10. Settings

存在必要性：
项目级配置、组织级配置、集成、通知、默认规则包、环境设定。

E. 完整用户路径
1) 个人用户路径

用户通常从“我有个 skill/workflow，想知道能不能上”开始。

路径：
Landing Page → Create Audit / Prompt Pack → Forge → Audit Detail → Gap Analysis → Re-submit → Permit / Export Package

关键体验：

不需要理解全系统，也能完成一次审计

系统明确告诉他哪里危险、能不能上线

最终拿到可交付包或 Permit

最终价值感：
“我不只是得到建议，我得到一个能交付的结果。”

2) 团队交付路径

团队通常从“我们要把这个交付给客户/内部接手团队”开始。

路径：
Dashboard → Registry → 某资产 Revision → Audit Detail → Gap Analysis → Team Approval → Permit → Handoff Package

关键体验：

审计留痕完整

谁修、谁批、谁放一目了然

客户接手时可看证据链和 Permit

最终价值感：
“这套交付是可被他人接手和复核的。”

3) 企业放行路径

企业用户通常从“这玩意能不能进生产/进入平台目录”开始。

路径：
Dashboard → Audit Queue → Audit Detail → Policy / Rule → Compliance Approval → Permit Issuance → Registry / History

关键体验：

看得见门禁

看得见谁有权签发

看得见版本绑定与撤销机制

最终价值感：
“我们不是在凭感觉上线，而是在制度下放行。”

F. 视觉风格建议
总体风格

工业控制台风格为主，现代 SaaS 风格为辅。

不要做成黑客炫技风，也不要做成轻飘飘的 AI 工具站。
最合适的是：

框架冷静

信息密

层级硬

状态明确

局部用高级感强调价值

颜色基调

建议主色调：

深石墨 / 深蓝灰 作为主背景或主框架

冷白 / 中灰 作为内容层

蓝色 表示系统状态与信息可信

橙色 表示警示与待修复

红色 表示 blocked / critical

绿色 只用于 Permit / Passed / Released，且慎用

核心原则：
绿色不能泛滥。
因为一旦满屏都绿，Permit 的价值就失去稀缺感。

信息密度

偏高，但必须克制。
建议：

总览页中密度高

首页密度中等

审计详情页高密度

Forge 页结构化强于视觉装饰

置信度：0.9

关键风险点：

Dashboard 如果做成普通数据看板，会丢掉“治理控制层”的气质。

Audit Detail 如果解释力不够，会失去“可信裁决”的核心价值。

Permit 页面如果只是一个通过结果页，而不是正式凭证页，商业锚点会变弱。

一、先定 3 个页面的角色分工
1. Dashboard

一句话职责：
让用户一眼看到“哪些资产可放、哪些被拦、为什么卡住”。

它卖的是：

全局掌控感

风险透明度

放行优先级

2. Audit Detail

一句话职责：
让每一个通过/不通过都能被解释、被复核、被修复。

它卖的是：

可解释裁决

非黑盒信任

修复闭环

3. Permit

一句话职责：
把“审计完成”变成“正式可放行的生产级凭证”。

它卖的是：

发布门禁

可交付性

商业凭证感

二、Dashboard 线框方案
页面目标

用户打开系统后 10 秒内知道：

当前有多少资产正在审

哪些资产已经可放行

哪些资产被阻断

最主要的阻断原因是什么

下一步应该处理什么

推荐布局
顶部区：Global Command Bar

放在页面最上方，横向一整条。

包含：

Workspace / Project 切换

全局搜索

新建入口：Run Audit

筛选器：Status / Risk / Gate / Owner

时间范围

右上角通知与审批提醒

作用：
一上来就是控制感，而不是内容流。

第一屏：核心状态带

建议 5 张高权重卡片横排。

卡片 1：In Review

当前待审与审计中数量

卡片 2：Blocked

被红线拦截数量

卡片 3：Fix Required

有缺口但可修复数量

卡片 4：Ready for Permit

已通过审计、待签发数量

卡片 5：Permitted

已正式放行数量

这 5 张卡片是整个 Dashboard 的灵魂。
它们比“总资产数”更重要，因为它们直接体现治理状态。

第二屏：中枢图区

左右双栏。

左栏：8 Gate Funnel / Gate Health

内容：

8 Gate 流程漏斗

每个 Gate 的通过率

各 Gate 当前阻塞数

每个 Gate 的 top failure reason

交互：

点击某个 Gate，右侧列表联动筛出对应资产

hover 显示该 Gate 的说明和红线示例

右栏：Priority Queue

内容：

最需要处理的 5–10 个资产

每条显示：Name / Risk / Current Gate / Last Decision / Next Action

排序建议：

Ready for Permit

Critical blocked

Large client delivery waiting

Recently changed revision

这样就不是普通“最近项目”，而是“治理优先队列”。

第三屏：证据与修复区

左右双栏。

左栏：Evidence Coverage

内容：

Evidence completeness 分布

哪些资产证据不足

哪些审计结论存在 evidence weakness

右栏：Gap Hotspots

内容：

最常见缺口类型

对应 Rule / Gate

受影响资产数量

平均修复耗时

这块很重要，因为它强化“你不是在生成，而是在治理”。

第四屏：Permit Activity + Revision Watch

左右双栏。

左栏：Recent Permit Events

Newly issued

Revoked

Expired

Replaced

右栏：Revision Watch

最近 revision 变更

哪些变更触发了重审

哪些 Permit 因 revision 失效

这两块把系统从“审计工具”拉升到“持续治理系统”。

Dashboard 信息优先级

状态

阻断原因

放行准备度

证据完整性

最近变化

Dashboard 核心按钮

只保留高价值动作：

Run Audit

Review Gaps

Issue Permit

View Audit Trail

不要放：

Generate Skill

Start Building

Open Canvas

Dashboard 关键状态设计
空状态

适用于新用户。
内容不要写成“还没有数据”，而要写成：

No governed assets yet

Start by auditing your first skill or workflow

系统流程示意：Audit → Gap → Permit

高风险状态

当 Blocked 数量高时，页面顶部应出现红色提醒条：
“Critical assets are blocked by Gate 3 / Gate 6 rules.”

待签发状态

当 Ready for Permit > 0 时，页面顶部可出现蓝色工作提示：
“3 assets passed audit and are waiting for permit issuance.”

三、Audit Detail 线框方案

这是最核心页面，要做成“判决书 + 审计工作台”。

页面目标

回答 5 个问题：

它现在是什么状态

为什么通过/不通过

证据是什么

缺口是什么

下一步怎么处理

推荐布局
顶部：Decision Header

高度要足，不能小气。

包含：

Asset name

Revision ID

当前状态徽章：Passed / Failed / Blocked / Fix Required / Ready for Permit

审计时间

审计版本

decision_hash

主操作按钮

右侧按钮建议：

Review Gaps

Export AuditPack

Re-run Audit

Submit for Permit

这一条头部要像正式裁决页，不像普通详情页。

第一模块：Decision Summary

一整块摘要卡。

内容：

Final decision

Confidence summary

Why it passed / failed

Highest impact issue

Permit readiness

Evidence sufficiency level

建议用一句主结论 + 三条支撑点。

例如：

Failed due to unresolved execution boundary leakage

Evidence is sufficient for rejection

2 critical gaps must be fixed before re-review

第二模块：8 Gate Timeline

横向或纵向都可以，但建议纵向更稳。

每个 Gate 卡包含：

Gate 名称

状态：Pass / Warning / Fail / Block

简短说明

Triggered rules count

Evidence count

点开后展开：

Rule triggered

Explanation

EvidenceRef

Suggested fix

这块是整个页面的“可解释性发动机”。

第三模块：EvidenceRef Panel

放在页面右侧固定侧栏，或者中下部独立模块。

内容：

EvidenceRef 列表

来源类型

证据摘要

证据强度等级

是否用户可见

是否仅显示摘要

你前面强调“哪些内容用户可见、哪些不可见但有摘要说明”，这里必须显式体现。

建议每条 evidence 有 3 层标记：

Visible

Summary only

Restricted internals

这样前端就把“反黑盒但不全透底”表达清楚了。

第四模块：Red Lines vs Fixable Gaps

必须分成两栏，不要混在一起。

左栏：Red Lines

特点：

不能忽视

不能带病放行

通常导致 Blocked

右栏：Fixable Gaps

特点：

可修复

可重新提交

有预期修复方式

这是前端解释力的关键。如果混在一起，用户会看不懂。

第五模块：Control & Contract Snapshot

内容：

Contract 摘要

Constitution / Rule Pack 版本

ControlSpec 摘要

生效策略

审计口径

这块不是给所有用户天天看，但必须存在。
它会让企业用户立刻意识到：这不是“模型说了算”，是“规则约束下裁决”。

第六模块：Hash & Reproducibility

专门一个模块，不要塞进角落。

内容：

demand_hash

contract_hash

decision_hash

linked Manifest

revision lineage

作用：
把“可复现”变成产品可见价值，而不是一句口号。

第七模块：Next Actions Footer

底部固定动作栏。

按状态给不同 CTA：

如果 Blocked

View critical fixes

Export fix brief

Re-submit after changes

如果 Fix Required

Go to Gap Analysis

Assign owner

Re-run audit

如果 Passed

Submit for Permit

Export AuditPack

如果 Ready for Permit

Issue Permit

Review release conditions

Audit Detail 信息优先级

结论

原因

证据

缺口

规则与复现口径

Audit Detail 用户感受

看完必须产生这个感觉：

“系统拦我，不是因为它神秘；系统放我，也不是因为它随意。”

四、Permit 页面线框方案

Permit 页不能做成普通 success page。
它要像“数字许可凭证 + 放行控制单”。

页面目标

明确告诉用户：

这个资产是否被允许放行

在什么边界下被允许

基于什么证据和哪个裁决

谁签的

何时失效

何时需要重审

推荐布局
顶部：Permit Certificate Header

视觉上要比普通页面更正式。

内容：

Permit ID

Asset name

Revision

Permit status：Active / Pending / Revoked / Expired / Superseded

Signed by

Effective date

Scope

右上角按钮：

Export Permit

View linked Audit

Revoke Permit

View history

第一模块：Permit Decision Block

大卡片展示：

Permit granted / not granted

基于哪个 audit decision

当前是否可 release

是否有附加条件

建议这里加入一句非常强的文案：

Audit pass is not release approval. Permit is.

这句话能直接把产品差异打进用户脑子里。

第二模块：Scope & Conditions

左右分栏。

左栏：Release Scope

适用环境

适用版本

适用资产边界

适用调用边界

右栏：Conditions

Must remain unchanged from certified revision

Re-audit required on contract change

Permit invalidates on policy mismatch

这一块非常商业化，因为它把“通过”提升成“受控允许”。

第三模块：Linked Basis

内容：

Linked AuditPack

decision_hash

contract_hash

evidence summary

applicable ruleset version

作用：
说明 Permit 不是拍脑袋签的，而是“基于裁决依据签发”。

第四模块：Lifecycle Status

用时间线显示：

Audit completed

Ready for permit

Permit issued

Active

Replaced / Revoked / Expired

点击任一节点可看详情。

第五模块：Compliance Signature Area

即使 v0 只是单签，也要设计成可扩展。

内容：

signed by

signed at

approval note

release comments

企业用户很在意这块，因为它决定制度感。

第六模块：Release Risk Reminder

即使 Permit 已签发，也要显示：

residual risks

operational warnings

monitoring expectations

这会增强可信度。
因为工业级产品不会假装“签了 Permit 就完全没风险”。

Permit 页面信息优先级

是否有效

生效范围

附加条件

依据

生命周期

Permit 页面用户感受

“这是一份正式放行凭证，不是一个绿勾截图。”

五、三权分立如何在前端可视化

这是你的重要差异，必须强表现。

推荐做法：固定权力边界条

在 Audit Detail、Permit、History 页面顶部或右侧，放一个很明确的边界组件。

显示三列：

Execution

Can run

Can produce outputs

Cannot approve

Audit

Can inspect

Can evaluate evidence

Cannot release

Compliance

Can issue permit

Can revoke

Cannot modify execution outputs

这个组件的价值极高。
因为它让用户不是“听你说有边界”，而是“看见边界”。

推荐做法：动作按钮按权力域分组

例如在 Audit Detail 页：

Execution 区动作：

Run

Re-run

Generate fix candidate

Audit 区动作：

Review evidence

Mark fail

Send to gap review

Compliance 区动作：

Submit for permit

Issue permit

Revoke permit

这样视觉上天然把权力隔开。

六、最关键的交互细节
1. 所有“通过”都不能直接显示大绿勾结束

必须显示：

Passed audit

Pending permit

这样才能建立“审计通过 ≠ 放行通过”的认知。

2. 所有失败都必须带“为什么”

至少包括：

触发了哪条规则

哪个 Gate 阻断

有哪些证据

哪些属于红线

哪些属于可修复项

3. 所有 Permit 都必须绑定 revision

不能出现“某 skill 已放行”这种模糊表达。
必须是：

asset x

revision y

hashes z

permit active

4. 所有历史都必须可回看

History 页哪怕 v0 简版，也必须支持：

时间线

决策变化

Permit 生命周期

九、你现在最该先画的线框顺序

建议顺序：

第一步：Audit Detail
因为这是产品价值最集中的页面。

第二步：Permit
因为这是商业闭环页。

第三步：Dashboard
因为它负责把前两者组织成管理体验。

这个顺序比先画 Dashboard 更稳。
因为一旦 Audit Detail 没钉住，整个产品语言都会发虚。
