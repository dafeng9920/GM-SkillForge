# 交互画布状态机与路由契约

## 1. 目的

这份文档用于固定 GM-SkillForge 下一阶段的主交互框架，避免继续在“首页 -> Dashboard -> 目标页”的传统多页跳转模型中反复返工。

目标不是把系统做成聊天产品，也不是做成传统后台，而是建立：

- 固定输入框：用户始终拥有控制权
- 可切换画布：系统随着进程展示不同阶段的结果
- 受控分流：系统只在高置信情况下进入明确路径
- 可审计承接：每一步输入、判断、下一动作都有记录

一句话：

**用户持续在输入框内工作，右侧画布承接并展示当前治理阶段。**

---

## 2. 总体原则

### 2.1 固定不变

以下元素在主框架中保持常驻：

- `Govern Composer`
- 当前输入态 / 已确认输入
- 建议动作入口
- 当前治理阶段标签
- 语言切换与全局状态

### 2.2 可切换

以下内容由右侧画布承接，并随状态变化：

- Intent / Clarify
- Vetting
- Audit
- Permit
- Overview

### 2.3 不再采用

以下策略不再作为主路径：

- 所有输入默认先跳到 `Dashboard`
- 每个阶段都强迫用户重新理解一整页后台
- 首页能对话，内页变成单向输出

---

## 3. 画布顺序

## 3.1 标准顺序

主链顺序固定为：

1. `Intent Canvas`
2. `Clarify Canvas`（仅低置信时出现）
3. `Vetting Canvas`
4. `Audit Canvas`
5. `Permit Canvas`

`Overview Canvas` 不在主链中间强制出现，只在以下情况下进入：

- 用户主动点击“概览 / Overview”
- 用户输入明确是“查看工作台 / 查看待办 / 看整体状态”
- 当前输入无法高置信分流，且用户选择进入工作台辅助判断

## 3.2 顺序含义

- `Intent`：系统复述输入，给出理解与建议方向
- `Clarify`：系统承认当前无法安全判断，要求补充上下文
- `Vetting`：处理外部 Skill / 外部资产引入前审查
- `Audit`：处理 revision / evidence / gap / adjudication
- `Permit`：处理正式放行与许可凭证
- `Overview`：工作台、待办、运营观察，而不是默认承接页

---

## 4. 每个画布的职责

## 4.1 Intent Canvas

页面职责：

- 复述用户当前输入
- 给出系统理解结果
- 给出主推荐动作
- 提供 1~2 个替代路径

必须回答的问题：

- 你刚才说了什么
- 系统理解成了什么
- 为什么建议你走这条路
- 你现在该点哪里

禁止承载：

- Gate 矩阵
- 证据覆盖总表
- Permit 事件流
- Revision watch

## 4.2 Clarify Canvas

页面职责：

- 承认当前输入不足以安全分流
- 明确说明缺少哪类上下文
- 引导用户补充

必须回答的问题：

- 为什么不能直接进入下一步
- 你需要补充的是“外部 Skill / Revision / Permit”中的哪一类

禁止承载：

- 强行推荐具体目标页
- 在低置信输入下直接跳路由

## 4.3 Vetting Canvas

页面职责：

- 展示外部 Skill / 外部资产的入口审查
- 展示来源、风险、红线、权限匹配、安装门
- 承接进入报告与 Permit 决策

核心问题：

- 这个外部资产是否可以进入系统
- 是阻止、修复，还是进入许可流程

## 4.4 Audit Canvas

页面职责：

- 展示裁决解释
- 展示 Evidence / Red Lines / Fixable Gaps / Contract-Control Snapshot
- 承接进入 Permit

核心问题：

- 为什么通过 / 为什么失败
- 当前 revision 还缺什么
- 哪些是阻断项，哪些是可修复项

## 4.5 Permit Canvas

页面职责：

- 展示正式 Permit 凭证
- 绑定 revision / hash / linked adjudication
- 呈现 scope / conditions / lifecycle / issuer

核心问题：

- 现在能不能正式放行
- Permit 绑定了什么
- 什么条件会导致失效

## 4.6 Overview Canvas

页面职责：

- 提供“今天要做什么”的工作台
- 提供优先队列、工作项、系统级观察
- 不作为所有输入的默认中转页

核心问题：

- 我今天最该处理什么
- 哪些条目最紧急
- 系统哪里压力最高

## 5. 输入到画布的分流规则

## 5.1 高置信输入

若输入包含明显意图信号，则直接进入目标画布：

- 外部引入类：
  - 关键词示例：`外部` `导入` `repo` `package` `skill`
  - 目标：`Vetting Canvas`

- 审计 / 修订类：
  - 关键词示例：`revision` `审计` `证据` `gap` `裁决`
  - 目标：`Audit Canvas`

- 放行 / Permit 类：
  - 关键词示例：`permit` `release` `放行` `签发` `许可`
  - 目标：`Permit Canvas`

## 5.2 低置信输入

若输入缺少足够语义，例如：

- 纯数字
- 指代模糊
- 只有“这个能不能过”
- 上下文缺失

则进入：

- `Clarify Canvas`

而不是：

- 直接进 `Dashboard`
- 直接猜成某条路径

## 5.3 显式总览意图

只有用户明确表达：

- 打开工作台
- 查看概览
- 看今天待处理事项
- 看整体状态

才进入：

- `Overview Canvas`

---

## 6. 路由策略

## 6.1 路由保留

当前已有页面路由继续保留：

- `/home`
- `/dashboard`
- `/intake/vetting`
- `/intake/vetting/report`
- `/audit/detail`
- `/permit`
- `/release/vetting-permit`

## 6.2 路由语义

这些路由不再理解为“用户必须跳进去才能继续”，而理解为：

- 右侧画布的独立可访问形态
- 深链接入口
- 可分享、可审计、可恢复的状态地址

## 6.3 主交互行为

主交互优先级应改为：

1. 用户留在固定输入框
2. 系统切换右侧画布
3. 只有当用户明确进入独立页面时，才发生整页路由切换

也就是说：

**画布切换优先，整页跳转次之。**

## 7. API 契约草案

## 7.1 `intent_resolution`

用途：

- 首页或任意全局输入后，用于判定当前意图

建议响应结构：

```json
{
  "input_text": "审查这个外部 skill 是否可以安装",
  "confidence": 0.92,
  "intent": "vetting",
  "reason_summary": "检测到外部资产与安装前判断信号",
  "alternative_intents": ["audit", "permit"],
  "requires_clarification": false,
  "next_canvas": "vetting"
}
```

## 7.2 `clarify_resolution`

用途：

- 低置信输入时，返回需要补充的上下文

建议响应结构：

```json
{
  "input_text": "14145454548484",
  "confidence": 0.18,
  "intent": "unknown",
  "requires_clarification": true,
  "clarify_prompt": "请说明这是外部 Skill、当前 Revision，还是 Permit 放行需求。",
  "missing_dimensions": ["asset_kind", "goal_type"]
}
```

## 7.3 `vetting_summary`

用途：

- Vetting 画布概览信息

建议响应结构：

```json
{
  "skill_name": "Partner Intake Skill",
  "source_type": "repo",
  "source_locator": "github.com/org/repo",
  "fingerprint": "pkg-2026-0313-a1",
  "source_trust": "unknown",
  "red_lines": 2,
  "fixable_gaps": 3,
  "install_gate": "conditional",
  "recommendation": "manual_approval_required"
}
```

## 7.4 `audit_summary`

用途：

- Audit 画布概览信息

建议响应结构：

```json
{
  "asset_name": "Payment Agent Skill",
  "revision": "R-014",
  "decision_status": "fix_required",
  "primary_reason": "evidence gap in Gate 4",
  "red_lines": 1,
  "fixable_gaps": 2,
  "evidence_sufficiency": "summary_only",
  "next_action": "review_gaps"
}
```

## 7.5 `permit_summary`

用途：

- Permit 画布概览信息

建议响应结构：

```json
{
  "permit_id": "PMT-2026-0313-014",
  "asset_name": "Payment Agent Skill",
  "bound_revision": "R-014",
  "status": "conditionally_allowed",
  "linked_audit_id": "AUD-2026-0313-022",
  "decision_hash": "dec-xxx",
  "contract_hash": "ctr-xxx",
  "scope_summary": "prod-read-only",
  "residual_risk": "manual approval still required for high-risk actions"
}
```

## 7.6 `overview_summary`

用途：

- Overview 画布或概览页

建议响应结构：

```json
{
  "today_queue_count": 3,
  "blocked_count": 4,
  "ready_for_permit_count": 6,
  "priority_items": [
    {
      "asset": "Payment Agent Skill",
      "revision": "R-014",
      "status": "ready_for_permit",
      "next_action": "issue_permit"
    }
  ]
}
```

## 8. 前端组件职责

## 8.1 `GovernComposer`

职责：

- 常驻输入
- 记录当前输入与最近确认输入
- 提交意图解析
- 发起 Clarify / Vetting / Audit / Permit 画布切换

不负责：

- 展示全部后台诊断模块
- 决定深层业务逻辑

## 8.2 `ContextCanvas`

职责：

- 承接当前治理阶段的展示
- 根据 `canvas_state` 切换不同视图

状态枚举：

- `intent`
- `clarify`
- `vetting`
- `audit`
- `permit`
- `overview`

## 8.3 `BridgeSummary`

职责：

- 复述输入
- 说明系统理解
- 提供主推荐动作

## 8.4 `DiagnosticsDrawer`

职责：

- 后置展示系统压力、证据覆盖、Permit 事件等
- 不占据首次认知主舞台

---

## 9. 实施顺序

## Phase 1

- 固定 `GovernComposer`
- 固定 `ContextCanvas`
- 完成 `Intent / Clarify / Vetting / Audit / Permit` 五态切换

## Phase 2

- 把首页输入器升级为全局 composer
- 把首页从“独立输入页”变成主壳层中的初始画布

## Phase 3

- 让 `Dashboard` 退化成 `Overview Canvas`
- 不再作为默认中转页

## Phase 4

- 后端按 `intent_resolution / vetting_summary / audit_summary / permit_summary / overview_summary` 收敛接口

---

## 10. 明确禁止

- 禁止所有输入默认跳到 `Dashboard`
- 禁止低置信输入被硬猜成具体治理路径
- 禁止把 `Overview` 当成所有页面的默认第二站
- 禁止把右侧画布做成固定不变的大总览
- 禁止让用户每一步都重新理解一个完整后台页面

---

## 11. 最终结论

GM-SkillForge 的正确交互框架不是：

- 传统多页跳转
- 首页输入，内页只读
- 所有请求都先中转到 Dashboard

而是：

**固定输入框 + 可切换治理画布**

这里的关键不在“像不像聊天”，而在：

- 用户控制权始终在输入框
- 系统上下文始终连续
- 右侧结果面始终承接当前治理阶段
- 页面跳转只在必要时发生，而不是作为默认交互路径
