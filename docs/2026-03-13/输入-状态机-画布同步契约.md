# 输入-状态机-画布同步契约

## 1. 目的

这份文档用于把 GM-SkillForge 的前端主框架彻底收口成三层：

1. 输入层
2. 状态机中间层
3. 画布承接层

并明确它与后端能力路由的边界关系。

这份契约解决的问题不是“页面长什么样”，而是：

- 输入如何被准确接住
- 状态如何被确定性裁决
- 画布为什么切到这个阶段
- 后端能力由谁决定调用
- 为什么不能让页面自己猜

一句话：

**不是输入框驱动画布，而是输入框驱动状态机，状态机驱动画布与后端能力调用。**

---

## 2. 核心原则

## 2.1 输入统一入口

所有以下动作统一从固定输入框进入：

- 用户自然语言输入
- 生成请求
- 外部来源植入
- 附件上传
- 对上一轮结果的补充说明
- 对当前阶段的追问
- 对下一步动作的请求

禁止：

- 每个页面各自维护一套独立主输入入口
- 页面级表单绕开全局输入层直接决定治理路径

## 2.2 状态机统一裁决

状态机中间层负责：

- 意图裁决
- 置信判断
- 澄清判断
- 路径确认
- 后端能力分派
- 下一动作生成
- 画布状态切换

禁止：

- 由页面组件自行解析输入并决定流程
- 由页面直接根据自然语言跳路由

## 2.3 画布只负责承接

画布层只负责：

- 展示当前状态
- 展示结构化解释
- 展示后端结果
- 展示当前可执行动作

画布不负责：

- 直接裁决输入意图
- 直接编排后端能力调用
- 直接决定下一条治理路径

---

## 3. 三层模型

## 3.1 输入层

输入层负责采集原始输入事件，不做治理判断。

### 输入层数据结构

```json
{
  "session_id": "sess-2026-03-13-001",
  "turn_id": "turn-0007",
  "raw_input": "审查这个外部 skill 是否可以安装",
  "attachments": [],
  "locale": "zh-CN",
  "current_canvas": "home",
  "current_route": "/home",
  "selected_asset": null,
  "selected_revision": null,
  "selected_permit": null,
  "input_mode": "text"
}
```

### 输入层职责

- 保存用户原始输入
- 保存当前上下文
- 发出 `INPUT_SUBMITTED` 事件
- 接收 `INPUT_CONFIRMED / INPUT_NEEDS_CLARIFICATION / INPUT_ROUTED` 反馈

### 输入层禁止事项

- 通过关键词直接切页面
- 直接决定调用 Vetting / Audit / Permit API
- 跳过状态机写入本地 UI 状态

---

## 3.2 状态机中间层

状态机是整个系统的交互编排核心。

它不是普通前端 reducer，也不是页面内逻辑，而是：

**Constitution-Aware Interaction Orchestrator**

### 主要职责

1. 读取输入事件
2. 根据宪法、规则包与上下文判定当前意图
3. 判断置信度是否足够
4. 决定需要澄清还是可以进入下阶段
5. 决定调用哪个后端能力
6. 决定切换哪个画布
7. 输出结构化状态供画布消费

### 输入

```json
{
  "event": "INPUT_SUBMITTED",
  "payload": {
    "raw_input": "审查这个外部 skill 是否可以安装",
    "attachments": [],
    "current_canvas": "home",
    "current_route": "/home"
  }
}
```

### 输出

```json
{
  "state_id": "st-2026-03-13-021",
  "resolved_intent": "vetting",
  "confidence": 0.93,
  "requires_clarification": false,
  "clarify_prompt": null,
  "next_canvas": "vetting",
  "route_decision": "/intake/vetting",
  "backend_action": "intent_resolution -> vetting_summary",
  "next_actions": [
    "open_vetting_home",
    "view_sample_report"
  ],
  "trace_id": "trace-0313-aud-001"
}
```

---

## 3.3 画布层

画布是状态机结果的承接面。

它的职责是：

- 解释系统当前理解
- 展示结果
- 暴露允许的下一步动作
- 提示当前状态

### 画布消费的数据

```json
{
  "canvas_state": "vetting",
  "header": {
    "title": "外部 Skill 审查",
    "status": "需要入口裁决"
  },
  "summary": {
    "reason": "检测到外部来源与安装前判断信号",
    "next_step": "开始入口审查"
  },
  "actions": [
    {
      "id": "open_vetting_home",
      "label": "进入审查"
    }
  ]
}
```

### 画布禁止事项

- 直接读取原始自然语言后重新推断意图
- 自己补出另一套“下一步动作”
- 自己绕过状态机再调后端能力

## 5. 状态转移规则

## 5.1 标准输入流

```text
IDLE
  -> INPUT_DIRTY
  -> INPUT_SUBMITTED
  -> INTENT_CONFIRMED
  -> ROUTING_READY
  -> CANVAS_ACTIVE
```

适用条件：

- 输入具备明确上下文
- 置信度达到阈值
- 不需要额外澄清

## 5.2 低置信澄清流

```text
IDLE
  -> INPUT_DIRTY
  -> INPUT_SUBMITTED
  -> CLARIFICATION_REQUIRED
  -> INPUT_DIRTY
  -> INPUT_SUBMITTED
  -> INTENT_CONFIRMED
  -> ROUTING_READY
  -> CANVAS_ACTIVE
```

适用条件：

- 纯数字
- 指代模糊
- 只有对象没有动作
- 只有动作没有对象

## 5.3 动作执行流

```text
CANVAS_ACTIVE
  -> ACTION_EXECUTING
  -> ACTION_COMPLETED
  -> CANVAS_ACTIVE
```

或

```text
CANVAS_ACTIVE
  -> ACTION_EXECUTING
  -> ACTION_BLOCKED
  -> CLARIFICATION_REQUIRED
```

## 7. 意图裁决规则

## 7.1 高置信 Vetting

触发条件示例：

- `外部`
- `导入`
- `repo`
- `package`
- `安装前`
- `external skill`

输出：

- `resolved_intent = vetting`
- `next_canvas = vetting`
- `backend_action = vetting_summary`

## 7.2 高置信 Audit

触发条件示例：

- `revision`
- `审计`
- `证据`
- `gap`
- `为什么失败`
- `blocked`

输出：

- `resolved_intent = audit`
- `next_canvas = audit`
- `backend_action = audit_summary`

## 7.3 高置信 Permit

触发条件示例：

- `permit`
- `release`
- `签发`
- `放行`
- `invalidate`

输出：

- `resolved_intent = permit`
- `next_canvas = permit`
- `backend_action = permit_summary`

## 7.4 低置信 Clarify

触发条件示例：

- 纯数字
- “这个能不能过”
- “继续上次那个”
- 没有对象
- 没有动作

输出：

- `resolved_intent = unknown`
- `requires_clarification = true`
- `next_canvas = clarify`
- `backend_action = clarify_resolution`

## 8. 后端能力路由契约

## 8.1 角色定位

前端状态机：

- 决定“当前该请求哪类能力”
- 决定“结果应该展示在哪个画布”

后端能力路由：

- 决定“实际调用哪个服务/组件/模块”
- 返回结构化结果

### 关键原则

前端状态机**不**直接绑定某个页面组件。  
它绑定的是：

- 意图类型
- 能力类型
- 状态转移

后端能力层**不**直接绑定某个 UI。  
它返回的是结构化结果，不返回“页面决策”。

## 8.2 能力路由表

| intent | canvas | backend action | output payload |
|---|---|---|---|
| vetting | vetting | `vetting_summary` | vetting summary / report preview |
| audit | audit | `audit_summary` | decision / evidence / gaps |
| permit | permit | `permit_summary` | permit summary / status / scope |
| unknown | clarify | `clarify_resolution` | clarify prompt / missing dimensions |
| overview | overview | `overview_summary` | queue / system pressure / events |

## 8.3 严禁

- 页面组件直接决定调用哪个后端接口
- 根据 UI 路由名硬编码后端逻辑
- 直接把自然语言输入透传给多个页面各自解析

---

## 9. 本地状态与持久化

## 9.1 必须持久化

- 最近确认输入
- 最近状态机输出
- 当前 canvas_state
- 最近 trace_id
- 最近建议动作

## 9.2 可以清理

- 当前未提交输入
- 临时 hover / focus / ui animation 状态

## 9.3 本地记录用途

- 从首页进入内页时保持连续性
- 允许从首页确认输入后在其他画布继续承接
- 避免页面切换后用户上下文丢失

---

## 10. 路由与画布关系

## 10.1 原则

路由不是第一驱动，状态机才是第一驱动。

即：

- 先有状态
- 再决定是画布内切换还是整页跳转

## 10.2 路由保留用途

路由仍然保留，用于：

- 深链接
- 可分享页面
- 可恢复会话
- 浏览器历史导航
- 审计定位

## 10.3 路由不是默认流程

例如：

- 高置信输入可以先切到 `VETTING` 画布
- 只有用户明确点击“进入完整审查页”时才跳 `/intake/vetting`

这意味着：

**画布切换优先于整页跳转。**

---

## 11. 组件职责边界

## 11.1 `GovernComposer`

负责：

- 输入
- 附件
- 提交
- 显示当前输入态

不负责：

- 决定治理路径
- 直接跳路由

## 11.2 `InteractionStateMachine`

负责：

- 状态转移
- 意图解析结果消费
- 后端能力路由决策
- 画布状态输出

不负责：

- 具体视觉渲染

## 11.3 `ContextCanvas`

负责：

- 渲染当前 canvas 对应的 UI
- 显示状态机输出结果

不负责：

- 再次解析用户输入

## 11.4 `CapabilityBridge`

负责：

- 把状态机输出转成具体 API 调用
- 收到结果后回填状态机

不负责：

- 自己定义交互顺序

---

## 12. 失败与阻断规则

## 12.1 低置信不跳页

若 `confidence < threshold`：

- 不允许直接进入 `vetting / audit / permit`
- 必须先进入 `clarify`

## 12.2 未确认不执行

若输入仅停留在 `INPUT_DIRTY`：

- 不允许发起后端能力执行
- 不允许切换到最终工作画布

## 12.3 规则冲突时，以宪法为准

若路由建议、页面按钮建议、临时 UI 状态与宪法规则冲突：

- 一律以状态机输出为准
- 状态机以 Constitution / policy / contract 为准

---

## 13. 最终结论

这套系统真正的打通方式不是：

- 首页一个输入框
- 后面一堆页面各自猜
- 前端直接猜意图后跳页
- 后端被动提供页面数据

而是：

**输入层统一接住**

**状态机中间层统一裁决**

**后端能力路由统一执行**

**画布层统一承接结果**

其中最关键的一句是：

**画布必须读取状态机结果，而不是读取原始输入后自己再猜一遍。**
