# 《治理冻结清单 + 生产主线待建清单》v1

## 0. 文档目的

本文件用于明确当前阶段的执行策略：

* **治理线先冻结，不继续深挖**
* **生产线转为主攻方向**
* **两条线未来按既定接口重新合拢**

这里的“冻结”不是废弃，也不是否定，而是：

> **把治理内核暂时视为已成立基线，只做必要修补，不继续扩张。**

这里的“待建”不是泛泛而谈，而是：

> **围绕 Skill 工厂主线，列出必须补齐的生产能力。**

---

## 1. 当前阶段总判断

当前系统应正式拆成两条线：

### 线 A：治理线

定位：
对 Skill / Workflow / 执行结果进行发现、裁决、Review、审计留证、问题资产化。

当前判断：
**L4 治理内核已形成基线。**

### 线 B：生产线

定位：
把需求、想法、文档、流程，转成可验证、可运行、可治理的 Skill 候选产物。

当前判断：
**生产线尚未完成系统化定义，是下一阶段主线。**

---

# 第一部分：治理冻结清单

## 2. 治理冻结原则

### 2.1 冻结目的

防止系统继续在治理方向过度深挖，导致：

* 上游生产标准长期空缺
* 治理内核反复返工
* 项目节奏被“下游越做越深”拖住

### 2.2 冻结口径

冻结后的治理线：

* 作为当前**验收基座**
* 作为未来生产线的**下游接收器**
* 作为系统可信度的**最低保障层**

### 2.3 冻结期间允许做的事

仅允许：

* 修 bug
* 修接口定义
* 修明显逻辑断点
* 补最小必要文档
* 为未来生产线预留接口

### 2.4 冻结期间不做的事

暂不继续：

* 扩更多治理分支能力
* 增加更细颗粒度裁决规则
* 深挖更多问题资产化后半段能力
* 做复杂治理 UI
* 做更复杂的审计可视化

---

## 3. 已冻结的治理基线能力

以下内容暂时视为**已成立基线**：

### 3.1 发现能力

系统具备对问题、异常、差异、候选对象进行发现与识别的基础能力。

### 3.2 裁决能力

系统具备基础性的通过 / 不通过 / 待复核裁决逻辑。

### 3.3 Owner Review

系统具备把关键结果交给 Owner / 人工进行复核的能力。

### 3.4 问题资产化前半段

系统已具备把问题沉淀为可追踪对象的前半段能力。

### 3.5 最小 AuditPack

系统已具备输出最小 AuditPack 交付物的能力。

### 3.6 治理内核角色定位

系统已明确：

* 治理负责验收与裁决
* 不再承担上游生产主责

---

## 4. 治理冻结对象清单

以下对象暂时冻结为 v1 基线，不做结构性扩张：

### 4.1 《系统宪法》v1

冻结为最高约束文本。
当前不继续扩写，只在必要时做术语修正。

### 4.2 《模块权限清单》v1

冻结为权限边界基线。
当前不扩更多模块，只在代码映射时补注释。

### 4.3 当前治理状态机

保持如下主状态口径不变：

```text
draft → candidate → validated → released → deprecated → tombstoned
```

失败分支口径也先不改。

### 4.4 AuditPack 最小交付口径

冻结最小交付格式，先不要追求“大而全”。

### 4.5 Owner Review 基本流程

保留当前最小可用闭环，不扩复杂审批流。

---

## 5. 治理冻结期间的接口保留项

虽然冻结，但以下接口必须保留，供生产线未来接入：

### 5.1 Candidate 输入接口

治理线必须能接收上游生产线产出的 candidate skill / candidate workflow。

### 5.2 Validation 接口

治理线必须能接收：

* contract
* schema
* artifact
* state model
* evidence draft

### 5.3 Review 接口

治理线必须能让人工对 candidate / validated 对象进行 review。

### 5.4 Release Decision 接口

治理线必须保留正式裁决写口。

### 5.5 AuditPack 输出接口

治理线必须能对最终候选物生成最小审计交付。

---

## 6. 治理冻结期间的“只修不扩”事项

冻结期只允许修这几类：

### 6.1 明显越权漏洞

例如：

* 非授权模块还能写 released
* 编排器还能绕过 gate
* 审计对象可被静默覆盖

### 6.2 接口口径不一致

例如：

* candidate 输入结构不统一
* evidence 字段命名不稳定
* review 记录格式不统一

### 6.3 审计记录断裂

例如：

* run 没有对应 evidence
* review 没有留痕
* release 没有关联链路

# 第二部分：生产主线待建清单

## 7. 生产主线总目标

生产线的目标不是“做一个 Skill”，而是：

> **建立一条标准化、可复用的 Skill 生产流水线。**

它要回答的不是：

* 某个 Skill 怎么写

而是：

* 任意合理需求如何被稳定地转成 Skill 候选产物
* 候选产物如何进入治理线
* 整个生产过程如何可追踪、可复用、可标准化

---

## 8. 生产主线的核心产物定义

生产线至少要产出以下对象：

### 8.1 Intent Draft

把自然语言需求转成结构化意图草案。

### 8.2 Contract Draft

把意图草案转成 contract 初稿。

### 8.3 Candidate Skill

把 contract 转成可验证的候选 Skill。

### 8.4 Candidate Artifact

包括：

* 目录结构
* 配置
* schema
* 脚本
* 说明文件
* 打包草案

### 8.5 Production Record

记录生产过程发生了什么。

---

## 9. 生产主线待建模块清单

## 9.1 意图编译层

必须建设：

* `nl--skill` 作为 Intent Compiler 的稳定定义
* 意图提取
* 缺失参数补全
* 模板匹配
* 风险初判
* 草案生成

当前目标不是让它“直接出成品”，而是让它：

> **稳定地产出 Draft。**

---

## 9.2 Contract Builder

必须建设：

* `input_schema`
* `output_schema`
* `state_schema`
* `error_schema`
* `trigger_spec`
* `evidence_spec`

这是生产线从“聊天生成”走向“工程对象”的关键层。

---

## 9.3 Skill Compiler

必须建设：

* 根据 contract 生成 Skill scaffold
* 生成目录与文件骨架
* 生成 candidate skill
* 生成 packaging 草案
* 生成运行与配置说明

这一层是“把协议转成产物”的核心。

---

## 9.4 Artifact Builder / Registry 接入

必须建设：

* candidate artifact 存储
* package metadata
* version 草案
* build trace

生产线不能只有生成动作，还要有产物落点。

---

## 9.5 生产态状态模型

必须定义并实现：

* draft
* compiling
* candidate
* build_failed
* handed_to_governance
* rejected
* revised

注意：

> 生产态状态机和治理态状态机必须区分，但可以映射。

---

## 9.6 生产记录对象

至少要落地：

### a. ProductionRecord

记录一次生成过程：

* 输入来源
* 使用模板
* 生成时间
* 关键中间结果
* 失败原因
* 输出 candidate

### b. BuildRecord

记录一次候选产物构建：

* 构建步骤
* 文件变更
* 打包情况
* 依赖情况

这一步非常关键，否则生产线会变成“生成了但不可追”。

---

## 9.7 最小生产验证

虽然治理线负责最终验收，但生产线内部也要先做最小自检：

* schema 是否齐
* scaffold 是否完整
* 目录结构是否合法
* 必要文件是否存在
* candidate 是否可交给治理线

这属于“生产前自检”，不是最终发布裁决。

---

# 第三部分：生产线建设顺序

# 第四部分：治理与生产的合拢点

## 13. 合拢原则

治理和生产不是并行独立系统，未来必须在以下位置合拢：

### 13.1 Candidate 交接点

生产线把 candidate 交给治理线。

### 13.2 Validation 交接点

治理线对 candidate 做 validate / gate。

### 13.3 Review 交接点

治理线对关键对象触发 Owner Review。

### 13.4 Release 交接点

治理线决定是否 released。

### 13.5 AuditPack 交接点

治理线对正式结果产出审计交付。

---

## 14. 合拢前的约束

在正式合拢之前，生产线必须遵守：

* 不直接写 released
* 不直接替代 validation
* 不直接替代 review
* 不直接替代 audit pack

生产线的终点是：

> **把 candidate 安全交给治理线。**

# 第五部分：本阶段执行口径

## 15. 今日后的工作口径

从现在开始，团队统一口径如下：

### 15.1 治理线

* 视为已形成基线
* 只修不扩
* 作为未来验收基座

### 15.2 生产线

* 升级为当前主线
* 围绕 Skill 工厂建设
* 优先补齐 contract / candidate / artifact / state

### 15.3 两线关系

* 不是二选一
* 不是先后互斥
* 是“先冻结下游，再建设上游，最后重新合拢”

---

## 16. 本阶段完成标志

当前阶段算完成，至少要满足：

### 16.1 治理冻结成立

* 治理基线明确
* 治理不再无边扩张
* 治理接口保留清晰

### 16.2 生产主链路成立

* 能从需求产出 intent draft
* 能从 intent 产出 contract
* 能从 contract 产出 candidate
* 能把 candidate 交给治理线

### 16.3 角色边界成立

* 生产负责生成
* 治理负责验收
* 两者接口清晰

---

## 17. 当前阶段一句话定性

> **治理线先封板，生产线转主攻，未来以 candidate 为桥梁与治理内核重新合拢。**

---

结论：

**现在最合理的策略不是继续双线同时深挖，而是先冻结治理基线，把主力转到生产线建设；等生产主链路成立后，再与治理线在 candidate / validate / release / audit 四个接口处合拢。**

置信度：0.91

关键风险点：

1. **治理虽被冻结，但接口没有先钉死**
   后续生产线做出来后，仍可能接不上治理内核。

2. **生产线一上来就追求全自动出成品**
   会重新滑回“自然语言直接生成”的旧路径。

3. **生产记录与候选状态机不先落地**
   后面会再次出现“生成过，但不可追踪、不可复盘、不可验收”的问题。
