结论：

**对，最该立刻建立的不是更多功能，而是“日终收口制度”。**
它的作用不是记录进展，而是**防扩散、防越权、防假进展、防重新滑回 old path**。

你现在需要的不是泛泛的 todo list，而是一份每天晚上都能拿来直接打勾的：

# 《GM-SkillForge 日终收口检查模板 v1》

## A. 今日目标是否收口

1. 今天的唯一主目标是否只有 **1 个**？
2. 今天新增的代码/文档/配置，是否都直接服务于这个主目标？
3. 是否出现了“顺手做一点”的范围扩散？
4. 若有扩散，是否已经明确标记为：

   * 不做
   * 延后
   * 另开条目
5. 今天是否形成了一个明确可验证的产出物？

判定：

* 通过：目标单一，产出明确
* 不通过：目标漂移、顺手扩展、产出模糊

---

## B. 宪法层是否被触碰

1. 今天是否改动了：

   * 宪法规则
   * 内核边界
   * Gate 顺序
   * 裁决口径
   * Evidence 口径
   * AuditPack 最小清单
2. 若改动，是否有明确理由？
3. 若改动，是否同步更新了对应文档与约束？
4. 若没有必要，是否坚持了：

   * **宪法不动**
   * **内核不膨胀**
   * **Skill 不越权**

判定：

* 通过：宪法层稳定，或改动有明确依据
* 不通过：隐性改口径、暗改边界、Skill 事实性越权

---

## C. 内核与 Skill 的边界是否还清晰

今天新增内容属于哪一类？

### 允许进内核的

* validate
* 8 Gate 执行
* Evidence 写入/哈希
* L3 AuditPack 组装与硬检查
* 确定性 intake
* at-time / tombstone / revision 口径

### 应该 Skill 化的

* repo scan
* fit score
* static scan
* spec drafter
* scaffold
* 各类意图实现
* 外螺旋压缩后的实现逻辑

检查：

1. 今天新增内容是否本该 Skill 化，却塞进了内核？
2. 今天是否让 Skill 直接做了裁决？
3. 今天是否让 n8n 或外部流程拥有了事实性通过权？

判定：

* 通过：边界稳定
* 不通过：内核膨胀或 Skill 越权

---

## D. 是否仍然遵守“生产线优先，治理只修不扩”

1. 今天做的是生产主线，还是治理增强？
2. 如果做了治理增强，它是否属于：

   * 修 bug
   * 修接口
   * 修越权
   * 修证据断裂
3. 今天是否又开始给治理线加新分支、新规则、新展示层？
4. 生产主链路是否比昨天更完整了一步？

判定：

* 通过：治理冻结，生产推进
* 不通过：又回到治理扩建

---

## E. 主链路是否更接近成立

当前主链路固定为：

`需求输入 → Intent Draft → Contract Draft → Candidate Skill → 最小验证 → 交给治理线`

检查：

1. 今天推进的是哪一段？
2. 这段是否真正形成输入/输出？
3. 是否能被下一段消费？
4. 是否只是“看起来有东西”，但还没接到主链？
5. 今天新增的模块，是否已经接入这条主链？

判定：

* 通过：主链前进
* 不通过：组件增加，但主链没变强

---

## F. 契约与状态是否真实存在

1. 今天新增内容是否有明确输入 schema？
2. 是否有明确输出 schema？
3. 是否定义了状态变化？
4. 是否定义了失败输出？
5. 是否能追踪 run / build / release / review 中的至少一种记录？

判定：

* 通过：有 contract / state
* 不通过：仍靠自然语言和隐式逻辑撑着

---

## G. 证据链是否闭环

1. 今天新增能力是否会产出可引用证据？
2. 是否能定位到：

   * EvidenceRef
   * hash
   * source locator
   * tool revision
3. 是否出现“有结论、无证据”的新增逻辑？
4. 是否新增了 mock 结果但未标注为 mock？

判定：

* 通过：结论可追证
* 不通过：只有叙述，没有证据

---

## H. n8n 分层是否被守住

固定原则：

* **n8n = 编排**
* **SkillForge = 审核/裁决**
* **Skill = 实现**
* **宪法 = 最高约束**

检查：

1. 今天是否把本应在 SkillForge 内核中的能力交给了 n8n？
2. 今天是否让 n8n 拥有了：

   * 最终放行权
   * 最终发布权
   * 修改 Evidence 的权力
3. 今天是否让 n8n 只做了：

   * 抓取
   * 触发
   * 路由
   * 重试
   * 通知
   * 人工审批编排

判定：

* 通过：n8n 不越权
* 不通过：n8n 开始吞治理

## I. 是否出现“假进展”

今天最容易骗自己的 5 类假进展：

1. 写了很多文档，但接口没定
2. 写了很多代码，但没接主链
3. 增加了模块名，但没落输入输出
4. 增加了 Skill，但没有进入 Gate
5. 跑通了一次，但不可复现

检查：

1. 今天的成果是否至少有一个能被明天直接复用？
2. 是否有至少一个对象完成了“定义 → 接入 → 可验证”？
3. 是否只是讨论推进，而不是系统推进？

判定：

* 通过：真实进展
* 不通过：假进展

# 最重要的执行原则

这套模板不是为了“写日报”，而是为了每天做 4 个收口：

* **范围收口**
* **权限收口**
* **主链收口**
* **明日目标收口**

只要这 4 个收口每天都做，复杂度就会上升，但**不会失控**。

置信度：0.92

关键风险点：

1. **模板写出来，但不做硬执行**
   会变成心理安慰，而不是约束系统的制度。

2. **每天检查太泛，不围绕“唯一目标”**
   会继续出现看似忙碌、实则扩散。

3. **Codex 只检查代码，不检查越权与主链接入**
   会把很多假进展误判为有效推进。

现有错误码与审计口径也要继续保持一致，避免这份日终模板与既有 error policy / error codes 体系漂移，可继续对照你已上传的 GM-SkillForge Error Codes 文件 。
