# 《模块权限清单 v1》

## 0. 文档目的

本清单用于把《系统宪法》落到**可执行的模块边界**上。
目标不是描述“模块能做什么”，而是明确：

* **谁可以做**
* **谁不能做**
* **谁拥有最终写权**
* **谁只能建议，不能裁决**
* **哪些动作必须经过上游/下游授权**

本清单一旦确定，后续实现、接口、状态机、数据库写入、工作流编排，都必须以此为准。

---

## 1. 总原则

### 1.1 唯一裁决权

只有 **Deterministic Kernel** 及其授权子模块，拥有正式状态写权与发布裁决权。

### 1.2 建议权不等于写权

凡是 AI、编排器、调度器、草案生成器，默认只有：

* 建议权
* 草案生成权
* 候选生成权

没有：

* 发布权
* 正式状态写权
* 审计通过写权

### 1.3 读写分离

所有模块按四类权限划分：

* **R（Read）**：读取
* **W（Write）**：写入
* **P（Propose）**：提出候选/建议
* **A（Approve）**：批准/裁决

其中：

* `P ≠ W`
* `W ≠ A`
* `A` 是最高权限，默认只在内核

### 1.4 正式对象优先级

对象统一分级：

* Draft
* Candidate
* Validated
* Released
* Deprecated
* Tombstoned

越高等级对象，写权限越收敛。

## 3. 模块权限定义

---

## 3.1 `nl--skill`

**定位：** 自然语言意图编译入口

### 允许

* 读取用户输入
* 读取模板、规范、历史上下文
* 提取意图
* 推断缺失参数
* 生成 draft contract
* 生成 draft scaffold
* 生成 candidate 建议
* 生成风险提示
* 发起“请求验证”

### 禁止

* 直接写入 `released`
* 直接写入 `validated`
* 跳过 schema 校验
* 跳过 gate
* 覆盖历史 release record
* 删除正式 evidence
* 用自然语言结果替代结构化 contract

### 权限级别

* Input：R
* Draft Contract：P
* Candidate Skill：P
* Release State：无
* Evidence：只读
* Audit Result：只读

### 备注

它是**入口编译器**，不是发布器。

---

## 3.2 `contract-builder`

**定位：** 合同/协议生成模块

### 允许

* 生成 `input_schema`
* 生成 `output_schema`
* 生成 `state_schema`
* 生成 `error_schema`
* 生成 `trigger_spec`
* 生成 `evidence_spec`
* 写入 draft/candidate 级协议对象

### 禁止

* 直接判定协议有效
* 直接发布协议为正式版本
* 绕过 validator 写入 validated 状态
* 擅自修改 released contract

### 权限级别

* Schema Draft：W
* Schema Candidate：W
* Schema Validated：无
* Released Contract：无

### 备注

它有**构造权**，没有**认证权**。

---

## 3.3 `skill-compiler`

**定位：** 将合同转成可执行骨架/候选产物

### 允许

* 依据 contract 生成 scaffold
* 生成 candidate skill / candidate workflow
* 补齐默认目录与文件结构
* 注入模板化脚本/配置
* 写入 candidate package 草案

### 禁止

* 直接宣告“可上线”
* 直接写入 release registry
* 私自跳过 packaging validate
* 修改历史 release 包内容

### 权限级别

* Draft Artifact：W
* Candidate Artifact：W
* Validated Artifact：无
* Released Artifact：无

---

## 3.4 `validator`

**定位：** 结构与契约验证器

### 允许

* 读取 candidate 对象
* 校验 schema 完整性
* 校验字段类型与必填项
* 校验依赖、目录结构、打包合法性
* 写入 validation report
* 将对象标记为 `validated` 或 `validation_failed`

### 禁止

* 生成业务草案
* 修改用户意图
* 直接发布 release
* 删除 candidate 历史记录

### 权限级别

* Candidate：R
* Validation Report：W
* Validated State：W
* Released State：无

### 备注

`validated` 是它的上限，不是它的终点。

---

## 3.5 `gate-engine`

**定位：** 规则门禁与准入裁决前检查

### 允许

* 读取 validated candidate
* 执行门禁规则
* 评估风险、完整性、韧性、证据要求
* 写入 gate result
* 输出 pass / fail / conditional pass

### 禁止

* 直接写 `released`
* 跳过 evidence completeness check
* 修改 contract 本体
* 绕过 release-manager 形成正式版本

### 权限级别

* Validated Object：R
* Gate Report：W
* Release State：无
* Evidence Acceptance：无

### 备注

它是**门禁官**，不是**发布官**。

---

## 3.6 `state-manager`

**定位：** 执行态与业务态唯一写入中心

### 允许

* 写入 run state
* 写入 retry count
* 写入 cursor / checkpoint / timestamp
* 写入 task lifecycle
* 管理幂等标识
* 维护业务态快照

### 禁止

* 擅自修改 release record
* 用运行态覆盖审计态
* 无授权写入 tombstone
* 删除关键历史状态

### 权限级别

* Run State：R/W
* Business State：R/W
* Release State：只读
* Audit State：只读

### 备注

正式状态与执行状态必须分离。

---

## 3.7 `evidence-collector`

**定位：** 证据采集与证据对象组装

### 允许

* 采集输入摘要
* 采集中间产物摘要
* 采集输出摘要
* 采集日志、耗时、成本
* 生成 `EvidenceRef`
* 组装 `AuditPack` 草案

### 禁止

* 自行判定 evidence 足够发布
* 删除失败记录以“美化结果”
* 修改 release decision
* 跳过原始时间戳与 run_id

### 权限级别

* Execution Trace：R
* EvidenceRef：W
* AuditPack Draft：W
* Release Decision：无

### 备注

它负责**留证**，不负责**裁决证据是否足够**。

---

## 3.8 `release-manager`

**定位：** 正式发布与版本落库唯一中心

### 允许

* 读取 validated + gate passed + evidence complete 的对象
* 写入 `released`
* 写入 `release_record`
* 分配 version
* 写入 release registry
* 拒绝不满足条件的 candidate

### 禁止

* 接收未验证对象直接发布
* 接收无 gate 结果对象直接发布
* 接收无 evidence 的对象直接发布
* 修改历史 release 而不产生 revision 记录

### 权限级别

* Candidate：R
* Validated：R
* Gate Result：R
* Evidence Result：R
* Release State：W/A

### 备注

这是**正式发布唯一写口**。

---

## 3.9 `revision-manager`

**定位：** 修订、废弃、下线管理中心

### 允许

* 创建 revision link
* 写入 `deprecated`
* 写入 `tombstoned`
* 记录 `revision_from / revision_to`
* 支持 at-time 查询映射

### 禁止

* 静默覆盖旧 release
* 删除历史版本使其不可追溯
* 在无记录情况下替换线上版本

### 权限级别

* Release History：R
* Revision Record：W
* Deprecation State：W
* Tombstone State：W/A

### 备注

它拥有**生命周期后半段的正式写权**。

---

## 3.10 `orchestrator`

**定位：** 工作流编排器

### 允许

* 安排执行顺序
* 调用脚本/工具/API
* 管理节点依赖
* 管理异步执行
* 回传节点结果给 state / evidence / validator

### 禁止

* 直接写 release
* 直接写 validated
* 把节点完成视为 gate 通过
* 修改 contract / schema 的正式版本

### 权限级别

* Task Plan：W
* Node Runtime Data：W
* Candidate/Validated/Released：只读或无

### 备注

它是**交通调度员**，不是**法院**。

---

## 3.11 `trigger-scheduler`

**定位：** 定时/事件触发调度器

### 允许

* 注册 cron / event trigger
* 触发 run
* 写入 trigger log
* 更新心跳时间

### 禁止

* 直接执行发布
* 直接改 release 状态
* 绕过 run pipeline 触发核心动作

### 权限级别

* Trigger Spec：R
* Trigger Log：W
* Release State：无

---

## 3.12 `observability-center`

**定位：** 指标、日志、告警中心

### 允许

* 读取运行指标
* 读取日志
* 读取成本数据
* 写入 metrics
* 写入 alerts
* 触发告警与人工复核事件

### 禁止

* 直接修复业务对象
* 直接修改 release
* 删除失败指标以掩盖问题

### 权限级别

* Logs：R/W
* Metrics：R/W
* Alerts：W
* Release Objects：只读

---

## 3.13 `admin-review-console`

**定位：** 人工复核与干预界面

### 允许

* 查看 draft / candidate / validated / released
* 查看 evidence / audit / logs
* 提交 review decision
* 触发人工批准或驳回
* 发起 revision 请求
* 发起 tombstone 请求

### 禁止

* 绕过系统记录直接改数据库
* 不留痕直接改 release
* 删除 evidence 与 audit 历史

### 权限级别

* 全局对象：R
* Review Comment：W
* Manual Approval：A（受审计记录约束）
* Direct State Mutation：禁止

### 备注

人工可参与裁决，但必须**留痕**。

---

## 3.14 `artifact-registry`

**定位：** 正式产物、包、版本元数据存储中心

### 允许

* 存储 candidate artifact
* 存储 released artifact
* 存储 package metadata
* 提供版本检索
* 提供 at-time 映射查询

### 禁止

* 接收无 release record 的正式包
* 静默替换已有 released artifact
* 删除被引用的历史版本

### 权限级别

* Candidate Artifact：R/W
* Released Artifact：R/W（仅接受 release-manager / revision-manager 写入）
* Global Search：R

## 5. 关键写权归属

以下写权必须严格收口：

### 5.1 `released`

仅：

* `release-manager`
* `revision-manager`（特定修订/废弃/下线场景）

### 5.2 `validated`

仅：

* `validator`

### 5.3 `gate_result`

仅：

* `gate-engine`

### 5.4 `run_state / business_state`

仅：

* `state-manager`

### 5.5 `evidence_ref / audit_pack`

仅：

* `evidence-collector`

### 5.6 `manual_approval`

仅：

* `admin-review-console`
* 且必须写入审计记录

---

## 6. 跨模块调用规则

### 6.1 合法主链路

`nl--skill`
→ `contract-builder`
→ `skill-compiler`
→ `validator`
→ `gate-engine`
→ `evidence-collector`
→ `release-manager`
→ `artifact-registry`

### 6.2 非法跳转

以下链路一律禁止：

* `nl--skill → released`
* `orchestrator → released`
* `trigger-scheduler → release-manager`（绕过验证）
* `contract-builder → validated`
* `skill-compiler → released`
* `observability-center → state overwrite`
* `admin-review-console → direct DB patch without record`

---

## 7. 人工干预规则

允许人工介入，但必须满足：

1. 只能通过 `admin-review-console`
2. 必须写入 review record
3. 必须留下：

   * 操作者
   * 时间
   * 原因
   * 影响对象
   * 前后状态
4. 不得覆盖原 evidence
5. 不得删除失败历史

人工权限不是“绕过系统”，而是“在系统内留痕干预”。

---

## 8. 最小落地执行要求

v1 先做这五件事就够：

### 8.1 锁死正式写口

先在代码和存储层保证：

* 只有 `release-manager` 能写 `released`
* 只有 `validator` 能写 `validated`

### 8.2 切断越权路径

把：

* `nl--skill`
* `orchestrator`
* `scheduler`

对正式状态的写权限全部拿掉。

### 8.3 单独建三类记录

至少落地：

* `run_record`
* `release_record`
* `review_record`

### 8.4 evidence 独立成对象

不能再把证据混在自然语言日志里。

### 8.5 revision/tombstone 单独收口

后续修订与下线必须走专门模块，不允许谁都能改。

---

## 9. 违例定义

以下行为视为权限违例：

* 非授权模块写 `released`
* 非授权模块写 `validated`
* 生成模块直接写审计通过
* 编排器绕过 gate
* 人工直接改库不留 review record
* 删除历史 evidence 或 release record
* 静默替换历史 artifact

一旦发生，应触发：

* 告警
* 审计标记
* 人工复盘

---

## 10. 当前执行结论

结论：

**系统下一步应先做“权限收口”，再做“能力增强”。**
也就是先把每个模块“能不能写、能写到哪一步”钉死，再继续扩展 schema、evidence、observability、自动化。

置信度：0.91

关键风险点：

1. **模块名义分层了，但数据库写权限没收口**
   最终仍会出现事实性越权。

2. **review/approval 没留痕**
   人工干预会把系统重新拉回不可审计状态。

3. **evidence 仍然依附在日志里**
   后续 gate、release、复盘都会缺少稳定依据。
