# 架构纠偏文档 v1

## 0. 文档目的

本文件用于对当前系统进行**架构纠偏**，目标不是推倒重来，而是在现有架构上完成三件事：

1. 保留已经正确的方向与已有资产
2. 把职责放错位置的模块重新归位
3. 把商用级所缺失的能力补齐进现有系统

本次纠偏的核心结论：

> 系统不应再把 `nl--skill` 视为“最终执行系统”，而应将其定义为**意图编译入口（Intent Compiler）**。
> 真正拥有发布裁决权的，必须是**确定性内核（Deterministic Kernel）**。

---

## 1. 当前问题定义

当前系统的原始设想偏向：

> 用户自然语言输入
> → 系统理解需求
> → 直接生成 skill / workflow
> → 执行并返回结果

这套路径在原型阶段成立，但在商用级要求下存在明显缺口。

### 1.1 本质问题不在“能力不够”，而在“分层不清”

当前风险不是做不出来，而是以下职责容易被混在一起：

* **意图理解**
* **协议生成**
* **执行编排**
* **结果验证**
* **错误裁决**
* **上线发布**
* **审计留证**

一旦这些职责由同一层承担，系统将出现三个后果：

1. 生成逻辑与发布逻辑耦合
2. AI 的即兴发挥被误当作系统能力
3. 整体可维护性、可复核性、可追责性不足

---

## 2. 本次纠偏的总体原则

### 原则一：不重写，做加固

本次不是系统归零，而是基于现有架构做**分层重构 + 能力补齐**。

### 原则二：保留前端智能，收回最终裁决

AI 可以继续负责：

* 理解
* 补全
* 草案生成
* 候选方案生成

但 AI 不再负责：

* 最终正确性判定
* 发布资格判定
* 审计通过判定

### 原则三：所有商用行为都必须过“内核”

凡是进入：

* 发布
* 自动化执行
* 监控轮询
* 对外服务
* 成本消耗
* 长期状态写入

都必须经过**确定性校验链路**。

### 原则四：输出必须“可证据化”

系统不能只回答“做了”，必须能回答：

* 做了什么
* 用什么输入做的
* 产出了什么
* 为什么判定通过/失败
* 哪个版本负责这次结果
* 下一次如何复现

---

## 3. 新架构定位

## 3.1 系统新角色划分

### A. `nl--skill`：意图编译入口

职责：

* 接收用户自然语言
* 抽取意图
* 补全缺失参数
* 选择模板/模式
* 生成结构化合同草案
* 生成 skill scaffold 草案
* 生成后续执行计划

它的定位是：

> **把“人话”翻译成“工程输入”**

它不是：

* 生产真相的地方
* 发布裁决的地方
* 风险兜底的地方

---

### B. Skill Compiler：协议与骨架生成层

职责：

* 生成 `input schema`
* 生成 `output schema`
* 生成 `state schema`
* 生成 `trigger spec`
* 生成 `evidence spec`
* 生成 `gate requirements`
* 生成 `packaging spec`

它的定位是：

> **把模糊需求转成可执行、可验证、可审计的结构化协议**

---

### C. Deterministic Kernel：确定性执行内核

职责：

* validate
* state handling
* retries / rate limiting / timeout
* degradation
* gate execution
* evidence collection
* release decision
* revision / tombstone / at-time

它的定位是：

> **系统唯一可信裁决层**

---

### D. Orchestration Layer：编排层

可由 OpenClaw / n8n / workflow engine 组成，职责是：

* 调度
* 连接工具
* 管理执行路径
* 处理异步触发
* 组织多节点执行

它不是最终裁决层，只是：

> **执行路径的组织者，不是通过/失败的法官**

---

### E. Observability & Operations：可观测与运维层

职责：

* 运行日志
* 指标统计
* 成本统计
* 心跳检测
* 异常告警
* SLA 统计
* 失败归因

---

## 4. 新的目标架构图（逻辑）

```text
User / Admin
   ↓
NL Interface
   ↓
Intent Compiler (nl--skill)
   ↓
Contract Builder / Skill Compiler
   ↓
Deterministic Kernel
   ├─ Validator
   ├─ Gate Engine
   ├─ State Manager
   ├─ Retry / Rate Limit / Timeout
   ├─ Evidence Collector
   ├─ Release Decision
   └─ Revision / Tombstone / At-Time
   ↓
Orchestration Layer (OpenClaw / n8n / workflow)
   ↓
External Tools / APIs / Browsers / DB / Schedulers
   ↓
Observability / Alerts / Cost / AuditPack
```

## 5. 核心纠偏点

## 5.1 把“生成 skill”改成“生成候选 skill + 合同”

旧思路：

> 生成出来了，就算完成

新思路：

> 生成的是候选物，必须进入验证链路才能成为正式产物

所以系统产物要分三类：

### Draft

草案，AI 可生成，可修改，可重试，不可直接发布。

### Candidate

具备结构和协议，进入验证和 Gate 检查。

### Released

通过验证、带版本、带证据、可追踪、可上线。

---

## 5.2 把“运行成功”改成“通过验证后成功”

商用级标准下，“跑通一次”没有意义。
要改成以下判定模型：

### 运行成功 ≠ 发布成功

必须同时满足：

* 结构合法
* 依赖合法
* 输入输出契约合法
* 状态管理合法
* 失败策略合法
* 证据完整
* 成本可接受
* 观测性可接入

才算真正通过。

---

## 5.3 把“AI 判断对不对”改成“内核依据规则裁决”

AI 可以提出判断建议，但最终结果必须由：

* schema validator
* gate rules
* evidence checker
* policy checker
* package validator

共同裁定。

---

## 6. 本次加固后，系统应具备的最小商用能力

以下能力是必须补齐的，不补齐就只能算原型级，不算商用级。

## 6.1 契约层能力

必须统一存在：

* `input_schema.json`
* `output_schema.json`
* `state_schema.json`
* `error_schema.json`
* `trigger_spec.json`
* `evidence_spec.json`

目的：

* 明确输入输出边界
* 为验证与监控提供统一依据
* 防止 skill 变成临时拼接物

---

## 6.2 状态层能力

系统必须明确区分两类状态：

### 执行态

如：

* run_id
* started_at
* retries
* step_status
* error_class

### 业务态

如：

* last_check_timestamp
* latest_cursor
* last_processed_id
* version_pin

没有状态模型，定时监控与增量处理一定失控。

---

## 6.3 韧性层能力

必须内置：

* timeout
* retry with backoff
* rate limiting
* circuit breaker / graceful degradation
* idempotency guard

目的不是“更高级”，而是避免两个现实问题：

1. 外部平台波动导致假失败
2. 自动化长期运行后把自己拖崩

---

## 6.4 证据层能力

每次执行都要沉淀可审计对象：

* 输入摘要
* 关键中间产物
* 输出摘要
* 错误记录
* 判定依据
* 耗时
* 成本
* 版本号
* Gate 结果

最终形成：

* `EvidenceRef`
* `AuditPack`

---

## 6.5 运维层能力

必须可观测：

* 调用成功率
* 平均耗时
* 重试率
* 降级率
* API 成本
* Token 成本
* 连续失败次数
* 最近心跳时间

并可触发：

* 邮件告警
* 飞书告警
* 人工复核队列

---

## 7. 新的标准执行流程

商用 Skill 的标准流程应定义为：

### 第一步：自然语言接入

用户输入需求。

### 第二步：意图编译

`nl--skill` 完成：

* 意图识别
* 参数抽取
* 缺失项补全建议
* 模板匹配
* 风险初判

产物：

* intent draft
* contract draft
* scaffold draft

### 第三步：协议固化

Skill Compiler 生成：

* schema
* state 定义
* triggers
* evidence 规范
* gate 规则

### 第四步：候选产物生成

形成 candidate skill / candidate workflow。

### 第五步：确定性验证

由内核执行：

* schema validate
* dependency validate
* packaging validate
* runtime precheck
* gate check

### 第六步：执行与留证

试运行 / 正式运行时记录：

* 输入
* 状态
* 日志
* 证据
* 结果
* 成本

### 第七步：裁决与发布

只有内核可以写出：

* released
* rejected
* tombstoned
* revised

### 第八步：运维监控

持续统计、告警、回滚、复盘。

## 8. 模块边界定义

## 8.1 `nl--skill` 的权限边界

允许：

* 生成建议
* 生成草案
* 生成 scaffold
* 生成 contract 初稿
* 给出参数推断

不允许：

* 直接宣告发布成功
* 跳过 gate
* 直接写 final release state
* 覆盖审计记录
* 用自然语言替代结构化结果

---

## 8.2 编排层的权限边界

允许：

* 组织执行顺序
* 调度外部工具
* 管理计划任务
* 处理中间跳转

不允许：

* 直接作为“通过/失败”的最终标准
* 私自绕过验证链
* 以节点完成替代业务完成

---

## 8.3 内核的权限边界

必须独占：

* validation
* release decision
* evidence acceptance
* revision control
* tombstone
* final state write

---

## 10. 发布状态机建议

建议统一成如下状态：

```text
draft
→ candidate
→ validated
→ released
→ deprecated
→ tombstoned
```

补充过渡状态：

```text
draft
→ rejected

candidate
→ validation_failed

released
→ revised
```

这套状态机的意义是：

* 明确什么是草案
* 明确什么是可验证对象
* 明确什么是正式产物
* 明确什么是已下线对象

## 11. 近期最小落地路线

本次纠偏不建议大爆炸改造，建议按三步完成。

## Phase 1：角色钉死

先不大改代码，只做架构定性。

本阶段完成：

* 明确 `nl--skill = Intent Compiler`
* 明确 `Kernel = Final Judge`
* 明确 `Orchestrator != Judge`
* 明确 Draft / Candidate / Released 三类产物

这是最关键的一步。

---

## Phase 2：补协议与状态

把缺失的标准对象先补出来。

优先级最高的四项：

1. input/output schema
2. state schema
3. run record
4. release record

这一步完成后，系统从“能跑”进入“能追踪”。

---

## Phase 3：补韧性与运维

最后加：

* retry / timeout / rate limit
* evidence pack
* heartbeats
* alerting
* cost analytics

这一步完成后，系统从“能追踪”进入“能商用”。

---

## 12. 本阶段明确不做的事

为避免系统继续膨胀，v1 明确不做以下事情：

### 不做“全自动无审查发布”

商用早期必须保留人工复核点。

### 不做“所有 skill 都支持所有场景”

先做可控的类型化 skill，再扩展。

### 不做“AI 自由生成即视为合格”

所有候选产物都必须过协议和 gate。

### 不做“复杂 UI 优先”

当前优先级是内核可信，不是界面华丽。

---

## 13. 判定本次纠偏是否成功的标准

满足以下条件，才算本次纠偏落地成功：

### 结构层

* `nl--skill` 不再直接代表最终发布
* 存在统一 contract / schema 体系

### 执行层

* 候选 skill 必须进入验证链
* 编排层无法绕过内核直接发布

### 审计层

* 每次运行有 run record
* 每次发布有 release record
* 每次失败有 error/evidence 可追踪

### 运维层

* 能看到成功率、失败率、耗时、成本
* 连续失败能够触发告警

---

## 14. 最终定性

这次架构纠偏，不是把原方向否掉，而是把它从“理想化自动生成系统”修正成“商用可信 Skill 生产系统”。

核心变化只有一句话：

> **从“自然语言直接出成品”改为“自然语言驱动一条受约束、可验证、可审计的生产流水线”。**

这句话定住之后：

* `nl--skill` 依然保留
* 现有模块依然可复用
* 原有资产不会白费
* 商用级能力才有真正落地的可能

---

## 15. 当前决策结论

结论：

**现有架构不应重写，应按“入口层—编译层—内核层—运维层”进行分层加固。**
本轮工作的重点不是扩功能，而是先收回裁决权、补齐契约层、补齐状态层、补齐证据层。

置信度：0.91

关键风险点：

1. **只改表述，不改权限边界**
   如果 `nl--skill` 名义上降级，实际上仍能越过内核写最终状态，问题不会真正解决。

2. **先做新功能，后补契约与状态**
   会继续制造更多“能跑但不可控”的技术债。

3. **把编排层当治理层使用**
   n8n / workflow 适合组织流程，不适合承担发布裁决与审计法官职责。

下一步最该做的是把这份 v1 继续压缩成一页式《系统宪法》，只保留角色、权限、状态机、发布链路四部分。
