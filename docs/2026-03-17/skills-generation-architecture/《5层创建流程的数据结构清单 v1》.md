下面直接给你 **《5层创建流程的数据结构清单 v1》**。
这版是按你现在已经钉死的主链路来写的：

* `需求 → Intent Draft → Contract Draft → Candidate Skill → Build Validation → Delivery Manifest`
* 迁移对象是 **Intent 目录 + Intent 合同 + Intent → Gate Pipeline 映射**，不是旧代码本身。
* 复杂度应被压扁成 **Intent Catalog（合同）+ Orchestration（配置）+ Skills（实现）**，而不是重新长成巨型编排系统。
* Skill 可以负责实现层，但 **Gate 裁决、证据闭环、复现口径、AuditPack 达标检查** 必须由内核掌握。

# 《5层创建流程的数据结构清单 v1》

## 0. 统一约定

## 0.1 通用字段约定

| 字段             | 类型                 | 说明                                    |
| -------------- | ------------------ | ------------------------------------- |
| `id`           | `string`           | 对象唯一标识，建议 `snake_case` 或 `kebab-case` |
| `revision`     | `string`           | 当前对象所属修订号/版本快照标识                      |
| `created_at`   | `string(datetime)` | 创建时间，ISO 8601                         |
| `updated_at`   | `string(datetime)` | 最近更新时间，ISO 8601                       |
| `status`       | `string`           | 当前状态                                  |
| `source_ref`   | `object[]`         | 来源引用，指向需求、文档、repo、文件等                 |
| `evidence_ref` | `object[]`         | 证据引用，指向 EvidenceRef                   |
| `errors`       | `object[]`         | 错误列表                                  |
| `warnings`     | `object[]`         | 警告列表                                  |
| `meta`         | `object`           | 扩展元数据                                 |

# 2. ContractBundle

## 2.1 作用

把 IntentDraft 压成 **contracts-first 合同包**。
这是系统从“理解需求”进入“可验证协议”的关键对象。前面你们已经明确，迁移对象必须包含 **输入/输出/约束/证据/门禁等级**，而不是只有意图名。

---

## 2.2 结构

ContractBundle 不是单一 schema，而是一个聚合对象，内部至少包含 6 份合同：

* `input_schema`
* `output_schema`
* `state_schema`
* `error_schema`
* `trigger_spec`
* `evidence_spec`

# 4. BuildValidationReport

## 4.1 作用

这是 **创建侧最小自检报告**，不是最终治理审计。
它解决的是：候选物是不是“最小可用、最小可验证、最小可交接”的。
这里要和治理侧审计严格分开：创建侧只做 build validation，最终裁决仍归内核。这个边界你们前面已经反复钉死了。

## 4.3 最关键的 6 个检查项

建议固定写死这 6 个，不要一开始就扩：

1. `schema_link_check`
2. `required_file_check`
3. `reference_check`
4. `static_check`
5. `smoke_test`
6. `trace_check`

---

## 4.4 `handoff_ready` 判定口径

只有以下同时满足才允许 `true`：

* 必需文件齐全
* schema 引用不裂
* `SKILL.md` 中引用资源存在
* 至少 1 次 smoke/static 检查通过或合理 skip
* trace 可生成
* 无 BLOCKER 级错误

# 6. 五个对象之间的关系图

```text
IntentDraft
  └── 产出 ContractBundle

ContractBundle
  └── 编译为 CandidateSkill

CandidateSkill
  └── 自检后生成 BuildValidationReport

BuildValidationReport
  └── 通过后整理为 DeliveryManifest

DeliveryManifest
  └── 交给 skill-creator / 打包层
```

---

# 7. 五个对象的状态机映射

## 7.1 创建态主状态

```text
intent_draft.status:
draft → reviewed → locked

contract_bundle.status:
draft → validated → frozen

candidate_skill.status:
draft_candidate → compiled → handed_to_validation

build_validation_report.status:
verified_candidate | partially_verified | validation_failed

delivery_manifest.status:
prepared → handed_off → delivered
```

---

# 8. 五个对象的最小落库表建议

如果你后面要落数据库，建议一开始就 1 对象 1 表：

* `intent_drafts`
* `contract_bundles`
* `candidate_skills`
* `build_validation_reports`
* `delivery_manifests`

不要一开始就混成一个大表。

---

# 9. 最小实现顺序

建议按这个顺序落，不要并行铺太开：

### P0

1. `IntentDraft`
2. `ContractBundle`
3. `CandidateSkill`

先打通：

```text
需求 → 意图 → 合同 → 候选物
```

### P1

4. `BuildValidationReport`

把候选物变成：

```text
可自检的候选物
```

### P2

5. `DeliveryManifest`

最后才接：

```text
skill-creator / 打包交付
```

---

# 10. 一句话钉死这份清单的用途

**这 5 个对象不是文档模型，而是 5 层创建流程的“骨架承重件”。**
后面无论你是让 Codex 开工、自己写代码、做日终核对，还是做模块权限检查，都应该围绕这 5 个对象来推进，而不是围绕“又加了什么功能”。

结论：

**这版 v1 已经足够作为你们 5 层创建流程的第一版数据骨架。**
下一步最值钱的，不是继续抽象，而是把这 5 个对象分别落成：

* `pydantic/dataclass 模型`
* `JSON Schema`
* `最小样例文件`

置信度：0.92

关键风险点：

1. **对象定义出来，但没有进入主链路**
   会变成“有 schema、无系统”。

2. **BuildValidationReport 和治理审计混在一起**
   会再次打乱“创建”和“裁决”的分层。

3. **DeliveryManifest 承担了过多工程判断**
   会让打包层重新侵入创建层，边界再次混乱。
