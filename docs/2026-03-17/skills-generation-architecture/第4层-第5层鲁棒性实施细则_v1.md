# 第4层-第5层鲁棒性实施细则 v1

## 0. 文档定位

本文件不是新架构，不新增层，不替代《5层流程.md》。  
本文件是对以下两层的实施细化：

- **第4层：韧性与鲁棒性（Resilience Layer）**
- **第5层：可观测性与运维（Observability & Operations Layer）**

本文件的作用是把 5 层创建架构中原本较粗的第4层、第5层，进一步压成：

- 可执行的设计要求
- 可检查的实施清单
- 可审计的达标口径
- 可延后落地的后续补全项

---

## 1. 适用范围

本文件适用于：

- 需要长期运行的 Skill
- 需要轮询/监控的 Skill
- 需要外部依赖、浏览器、API、Webhook、账号状态的 Skill
- 需要重复执行、可能中断、可能失败恢复的 Skill
- 需要人工接管与告警的 Skill

本文件尤其适用于：

- 监控类 Skill
- 采集类 Skill
- 定时任务类 Skill
- 浏览器自动化类 Skill
- 多步骤外部交互类 Skill

本文件暂不直接覆盖：

- 治理内核实现
- 发布裁决实现
- 审计对象实现
- Review / Release / AuditPack 正式对象实现

这些仍属于治理线，不在本文件内展开。

## 2. 总原则

### 2.1 本文件只细化第4层与第5层
本文件中的所有要求，都必须被解释为：

- 第4层的实施细则
- 第5层的实施细则

不得把本文件解释成：

- 第6层
- 平行新体系
- 替代 5 层主骨架的文本

---

### 2.2 先定义失败，再设计恢复
真正的鲁棒性不是“尽量不出错”，而是：

> **先枚举失败模式，再明确处理策略。**

每个 Skill 在进入较稳运行前，至少要回答：

- 会怎么失败
- 失败后如何检测
- 失败后如何恢复
- 失败后何时报警
- 失败后何时人工接管
- 失败后哪些状态要持久化

---

### 2.3 自动化不是万能的，鲁棒性必须允许人工介入
真正商用级的 Skill 必须承认：

- 外部平台会改版
- 会弹验证码
- 会封号
- 会限流
- Cookie 会过期
- 依赖会漂移
- 服务器会中断

所以必须保留：

- 主动报警
- 人工接管
- 恢复后续流程

## 3. 第4层：韧性与鲁棒性实施细则

---

## 3.1 状态持久化（Stateful Persistence）

### 3.1.1 必要性
商用 Skill 不能是“无记忆”的。  
任何可能中断的 Skill，都必须明确：

- 当前执行到哪一步
- 最后成功的位点是什么
- 已尝试重试多少次
- 下次从哪里恢复

### 3.1.2 最小要求
至少应记录：

- `run_id`
- `stage`
- `last_successful_step`
- `retry_count`
- `last_checkpoint`
- `last_error_code`
- `updated_at`

### 3.1.3 允许的持久化形式
按复杂度递增：

1. 本地文件状态（仅原型）
2. SQLite
3. Redis
4. 其他状态存储

### 3.1.4 设计要求
- 状态写入必须在关键步骤后进行
- 恢复时必须能读到上次 checkpoint
- 不允许“失败后从零开始”成为唯一策略
- 不允许因为进程重启而丢失所有上下文

---

## 3.2 断点续传与恢复（Checkpoint / Resume）

### 3.2.1 必须支持的能力
对长流程、轮询流程、采集流程，必须回答：

- checkpoint 如何定义
- resume 从哪里开始
- 哪些步骤可重放
- 哪些步骤不可重放

### 3.2.2 建议 checkpoint 粒度
建议按“步骤级”或“对象级”记录，例如：

- 最后处理的 tweet_id
- 最后抓取的游标 cursor
- 最后成功翻页的页码
- 最后成功提交的对象标识

### 3.2.3 恢复原则
恢复时优先选择：

- 从最后成功位点继续
- 从幂等安全点重跑

不建议：

- 无脑重头执行
- 无脑跳过未完成状态

---

## 3.3 幂等性（Idempotency）

### 3.3.1 目标
同一输入被重复触发时，不得产生重复结果或副作用。

### 3.3.2 典型要求
例如：

- 同一条推文不能重复推送到 n8n 两次
- 同一条告警不能重复创建多条工单
- 同一条记录不能因为重跑被重复写入数据库

### 3.3.3 最小做法
至少有一种：

- 去重 key
- processed set
- last_processed_id
- 幂等 token
- 结果哈希对比

### 3.3.4 审计口径
没有幂等设计的 Skill，不得宣称为“商用鲁棒”。

---

## 3.4 重试策略（Retry Policy）

### 3.4.1 必须明确
每一种失败都要回答：

- 是否重试
- 重试几次
- 间隔多久
- 是否指数退避
- 重试失败后转什么状态

### 3.4.2 推荐做法
建议采用：

- 有上限重试
- 指数退避（exponential backoff）
- 带 jitter 的随机扰动
- 明确 retryable / non-retryable 区分

### 3.4.3 不允许
- 无限重试
- 所有错误都一刀切重试
- 没有最大重试次数
- 失败后静默吞掉

---

## 3.5 限流与节流（Rate Limiting / Throttling）

### 3.5.1 必要性
外部平台不稳定时，过快请求会导致：

- 封禁
- 验证码
- IP 限制
- 账号风控
- API 配额耗尽

### 3.5.2 最小要求
必须至少具备以下之一：

- 固定最小请求间隔
- 动态速率限制
- 账号级速率限制
- 租户级速率限制

### 3.5.3 浏览器型 Skill 的附加要求
对于浏览器自动化类 Skill，可增加：

- 随机延迟
- 轨迹模拟
- 非线性等待
- page-ready 检测

注意：这些是增强项，不是当前必须立即实现项。

---

## 3.6 降级策略（Graceful Degradation）

### 3.6.1 目标
当完整流程无法执行时，Skill 不应直接崩溃，而应进入：

- 可解释失败
- 可恢复等待
- 人工接管
- 部分结果输出

### 3.6.2 常见降级类型
- 浏览器失败 → 仅输出错误上下文
- 无法登录 → 发告警并进入等待
- 页面结构异常 → 输出快照与定位线索
- 数据不完整 → 输出 partial result + 标记

### 3.6.3 不允许
- 失败直接消失
- 失败后无状态记录
- 失败后无后续动作建议

---

## 3.7 环境一致性（Environment Parity）

### 3.7.1 原则
开发环境与运行环境必须尽量一致。  
“我本地能跑”不代表“线上能跑”。

### 3.7.2 最小要求
若 Skill 依赖浏览器、驱动、字体、系统库、特定 Python 版本，则必须记录：

- Python 版本
- 关键依赖版本
- 浏览器/驱动版本
- 操作系统依赖
- 容器镜像信息（若有）

### 3.7.3 推荐做法
推荐使用：

- Docker 镜像
- 锁定依赖版本
- 固定运行时配置
- 明确环境变量要求

### 3.7.4 不允许
- 环境依赖完全隐式
- 本地脚本和生产脚本依赖不同却无说明
- 浏览器 Skill 无明确运行环境口径

---

## 3.8 影子测试（Shadow Testing）

### 3.8.1 目标
正式上线前，让 Skill 在后台静默运行一段时间，比较：

- 机器判断结果
- 人工观测结果
- 历史可信结果

### 3.8.2 适用场景
尤其适用于：

- 监控 Skill
- 识别 Skill
- 页面采集 Skill
- 风险判断 Skill

### 3.8.3 最小要求
至少记录：

- 测试样本量
- 命中率/准确率
- 偏差案例
- 失败原因分类

### 3.8.4 当前口径
影子测试属于：

- 第4层实施细则
- 第5层可观测要求

但不要求当前阶段立即全部实现。  
应先吸收为后续落地项。

---

## 3.9 人机协同降级（Human-in-the-Loop）

### 3.9.1 原则
当自动化无法处理时，系统必须能：

- 主动报警
- 人工接管
- 接管后恢复后续流程

### 3.9.2 触发场景
例如：

- 验证码
- 账号冻结
- Cookie 失效
- 页面结构大改
- 高风险动作需人工确认

### 3.9.3 最小要求
至少明确：

- 什么情况下触发人工接管
- 通过什么方式报警（Webhook/邮件/消息）
- 人工完成后流程如何继续

### 3.9.4 当前实现口径
当前只要求：

- 文档吸收
- 接口预留
- 设计原则明确

不要求当前就实现管理界面。

---

## 3.10 失败模式清单（Failure Taxonomy）

### 3.10.1 必须有失败分类
每个较复杂 Skill 至少应分类：

- 网络失败
- 认证失败
- 环境失败
- 页面改版
- 风控失败
- 数据结构异常
- 外部依赖异常
- 人工介入类失败

### 3.10.2 每类失败至少要定义
- error code
- 是否重试
- 是否报警
- 是否可恢复
- 是否需要人工介入

### 3.10.3 原则
没有失败模式表的 Skill，鲁棒性设计通常不成立。

---

## 4. 第5层：可观测性与运维实施细则

---

## 4.1 日志（Logging）

### 4.1.1 最小记录项
至少记录：

- `run_id`
- skill_id / intent_id
- stage
- input summary
- output summary
- error code
- latency
- retry_count
- timestamp

### 4.1.2 原则
日志必须：

- 可搜索
- 可聚合
- 可关联到执行对象
- 可用于事后复盘

### 4.1.3 不允许
- 只打印自然语言日志
- 关键步骤无日志
- 错误发生时无 error code

---

## 4.2 指标（Metrics）

### 4.2.1 最小指标
建议至少统计：

- 成功率
- 失败率
- 平均耗时
- P95 / P99 耗时（后续）
- 重试率
- 降级率
- 人工接管率
- 最新心跳时间

### 4.2.2 成本指标
对于调用外部 API 或模型的 Skill，还应统计：

- token 消耗
- API 成本
- 浏览器运行成本
- 单次任务平均成本

---

## 4.3 告警（Alerting）

### 4.3.1 必须有告警触发条件
至少明确以下条件中的若干个：

- 连续失败 N 次
- 心跳超时
- 核心依赖失效
- 账号状态异常
- 影子测试偏差超阈值
- 重试耗尽

### 4.3.2 告警输出方式
允许：

- Webhook
- Email
- 飞书 / Slack / Teams
- 管理台待处理事件

### 4.3.3 不允许
- Skill 持续失败但无人可见
- 失败只留在日志里，没有外部提醒

---

## 4.4 心跳与存活检查（Heartbeat）

### 4.4.1 适用对象
定时监控类、长期运行类、轮询类 Skill 必须具备心跳概念。

### 4.4.2 最小要求
至少记录：

- 最近一次启动时间
- 最近一次成功时间
- 最近一次失败时间
- 当前状态

### 4.4.3 死人开关
如果连续多次未成功执行，应触发：

- 告警
- 暂停
- 人工介入

---

## 4.5 运行成本与预算控制

### 4.5.1 目标
Skill 必须不仅“能跑”，还要“跑得起”。

### 4.5.2 最小要求
至少要能回答：

- 每次执行成本大概多少
- 成本主要来自哪里
- 哪些路径会放大成本
- 是否需要成本阈值告警

### 4.5.3 适用场景
尤其适用于：

- 高频轮询
- 浏览器自动化
- 高 token 消耗 Skill
- 多租户并发任务

---

## 4.6 多租户 / 多账号观测口径

### 4.6.1 原则
如果 Skill 支持多个账号或租户，观测不能只看全局，要能分层看：

- tenant
- account
- task
- run

### 4.6.2 最小要求
至少支持其中一种维度分组统计。

---

## 4.7 运维动作接口（Operations Hooks）

### 4.7.1 目标
运维层不仅看见问题，还应具备最小操作能力。

### 4.7.2 可预留动作
- 重试单次任务
- 暂停任务
- 恢复任务
- 更新凭证/Cookie
- 标记人工接管完成

### 4.7.3 当前阶段口径
当前阶段只要求：

- 在设计中留钩子
- 在文档中明确
- 不要求立即实现控制台

---

## 5. “玩具级”与“商用级”差异说明

本段只作为：

- 审计辅助
- 检查清单
- 外部表达参考

不替代主架构。

| 特性 | 玩具级 | 商用级 |
|---|---|---|
| 状态 | 临时变量 / 无状态 | 持久化状态 + 断点续传 |
| 错误处理 | try-except 打印日志 | 分类错误 + 重试 + 降级 + 告警 |
| 环境 | 本地可跑 | 运行环境锁定 / Docker / parity |
| 数据采集 | 一次性抓取 | 可恢复、可节流、可追踪 |
| 自动化失败 | 直接报错退出 | 降级 + 人工接管 |
| 运行监控 | 几乎没有 | 日志 + 指标 + 告警 + 心跳 |
| 成本 | 不关注 | 成本统计与预算感知 |
| 准确性 | 肉眼感觉差不多 | 影子测试 / 偏差对照 |
