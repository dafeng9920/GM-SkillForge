# 《系统宪法》v1

## 序言

本系统的目标，不是“把自然语言直接变成可上线成品”，而是：

**把自然语言需求，转化为一条可约束、可验证、可审计、可运维的 Skill 生产与执行流水线。**

本宪法高于模块实现、高于工作流便利性、高于临时效率。
凡与本宪法冲突的设计、流程、模块、脚本、编排，均应以下列条款为准。

---

## 第一条：系统的根本定位

本系统是一个：

**受合同约束的 Skill 编译与治理系统**

不是：

* 纯聊天生成器
* 任意发挥的 AI 代理
* 只要跑通就算成功的自动化拼装器

系统的第一目标是：

**稳定性、可维护性、可复核性、可追责性。**

---

## 第二条：系统的四层结构

系统统一划分为四层，不得混权：

### 1. 入口层

职责：

* 接收自然语言需求
* 做意图识别
* 做参数补全
* 生成草案

代表模块：

* `nl--skill`

### 2. 编译层

职责：

* 生成 contract
* 生成 schema
* 生成 trigger / state / evidence 规范
* 生成 candidate skill / workflow

### 3. 内核层

职责：

* validate
* gate
* evidence acceptance
* release decision
* revision / tombstone
* final state write

### 4. 运维层

职责：

* logging
* metrics
* heartbeat
* alerting
* cost control
* failure tracking

---

## 第三条：唯一裁决权原则

系统中只有一个最终裁决中心：

**确定性内核（Deterministic Kernel）**

仅内核有权决定：

* 是否通过验证
* 是否允许发布
* 是否记录为正式结果
* 是否修订旧版本
* 是否 tombstone 下线

任何其他层，包括：

* `nl--skill`
* workflow
* OpenClaw
* n8n
* agent
* 人工提示词结果

都**无权直接宣告正式发布成功**。

---

## 第四条：AI 的合法职责边界

AI 允许做：

* 理解需求
* 生成草案
* 生成合同初稿
* 生成 scaffold
* 提供修复建议
* 生成候选方案

AI 不允许做：

* 直接跳过验证
* 直接写 release 状态
* 用自然语言替代结构化结果
* 用“看起来合理”替代证据
* 用一次跑通替代长期可维护

**AI 是前端智能，不是真相来源。**

---

## 第五条：合同优先原则

任何 Skill、Workflow、自动化任务，在进入正式执行与发布前，必须先有合同。

最小合同必须包含：

* `input_schema`
* `output_schema`
* `state_schema`
* `error_schema`
* `trigger_spec`
* `evidence_spec`

没有合同，不得进入正式链路。
合同不完整，只能算草案，不算候选产物。

---

## 第六条：产物分级原则

系统产物统一分为三级：

### Draft

草案。
可生成、可修改、可讨论、不可发布。

### Candidate

候选产物。
具备合同、具备结构、可进入验证。

### Released

正式产物。
已通过验证、留证、记录版本、可追溯、可运维。

禁止把 Draft 当 Released。
禁止把一次临时成功当 Released。

---

## 第七条：验证强制原则

凡进入发布、自动化执行、长期运行、状态写入、成本消耗的对象，必须先经过验证链。

最小验证链必须包括：

* schema validate
* dependency validate
* runtime precheck
* gate check
* packaging validate
* evidence completeness check

**运行成功不等于验证成功。**
**验证成功之前，不得授予正式资格。**

---

## 第八条：证据优先原则

系统判断一切结果时，优先依据证据，而非叙述。

每次执行至少沉淀：

* 输入摘要
* 输出摘要
* 关键中间产物
* 错误记录
* 耗时
* 成本
* 使用版本
* gate 结果
* evidence ref

系统必须能回答：

* 做了什么
* 何时做的
* 依据什么判定通过/失败
* 使用了哪个版本
* 如何复现

不能回答上述问题的结果，不算商用级结果。

---

## 第九条：状态机统一原则

系统正式状态统一为：

```text
draft → candidate → validated → released → deprecated → tombstoned
```

允许失败分支：

```text
draft → rejected
candidate → validation_failed
released → revised
```

禁止模块私自发明平行状态。
禁止绕过 `candidate / validated` 直接写入 `released`。

---

## 第十条：编排不等于治理

工作流引擎、调度器、节点编排器的职责是：

* 安排顺序
* 调度工具
* 管理触发
* 执行步骤

它们不是：

* 审计法官
* 发布法官
* 真相判定器

**节点跑完，不代表业务达成。**
**Workflow 完成，不代表系统通过。**

---

## 第十一条：韧性为商用底线

所有商用级 Skill 必须具备最小韧性层：

* timeout
* retry with backoff
* rate limiting
* graceful degradation
* idempotency guard

没有韧性层的 Skill，只能视为演示级，不视为商用级。

---

## 第十二条：运维不可缺席

所有正式运行对象必须接入可观测体系。

至少监控：

* 成功率
* 失败率
* 平均耗时
* 重试率
* 降级率
* 最近心跳
* 连续失败次数
* API / token 成本

连续失败达到阈值时，必须进入：

* 告警
* 人工复核
* 自动暂停
  之一。

---

## 第十三条：变更必须可追溯

一切正式变更必须可追踪：

* 谁改的
* 改了什么
* 依据什么改
* 从哪个版本变到哪个版本
* 是否影响旧结果解释

系统必须支持：

* revision
* at-time
* tombstone

禁止不可追溯覆盖。
禁止“静默替换”正式产物。

---

## 第十四条：系统禁令

以下行为一律视为违宪：

1. 让 `nl--skill` 直接拥有发布权
2. 没有 schema 就进入正式执行
3. 没有 evidence 就宣告结果成立
4. 让 workflow 节点完成替代业务完成
5. 用 AI 的自然语言解释替代结构化审计记录
6. 没有状态模型却运行长期监控型 Skill
7. 没有告警与心跳却声称商用可用

---

## 第十五条：一切实现的总口径

系统的统一执行口径只有一句话：

> **自然语言只能触发生产；合同才能进入验证；内核才能决定发布；证据才能支撑真相。**

---

## 第十六条：本阶段最高优先级

当前阶段优先级顺序固定为：

1. **收回裁决权到内核**
2. **补齐合同层**
3. **补齐状态层**
4. **补齐证据层**
5. **补齐韧性层**
6. **补齐运维层**
7. 最后才是体验优化与复杂自动化扩展

任何与该顺序冲突的开发冲动，原则上暂停。

---

## 结语

本系统未来可以很强，但强的前提不是“更会生成”，而是：

**更受约束、更能验证、更能留证、更能长期运行。**

因此，本系统的最高原则不是自由生成，而是：

**受控生产。**

结论：

**《系统宪法》应作为你们后续架构、模块边界、发布规则、审计口径的最高约束文本。**

置信度：0.92

关键风险点：

1. **宪法写出来，但没有变成权限约束**
   会沦为说明文，而不是系统规则。

2. **状态机和合同层没有先统一**
   后续模块仍会各自为政，继续积累隐性耦合。

3. **编排层继续事实性越权**
   即使名义上承认内核裁决，实际系统仍可能被 workflow 侧绕开。
