【Codex 执行约束附录 v1】

你必须把本附录视为高优先级执行约束。
本附录不是建议，而是执行边界。

────────────────
一、总执行原则
────────────────
1. 先保真，后整合，最后才允许最小优化
2. 先报告，后改动；未完成整合报告前，不得主动改代码
3. 不得为了“更统一”“更优雅”“更先进”而牺牲当前已冻结的系统边界
4. 不得用自己的偏好替代文档中已经明确写下的结论
5. 当前目标是“系统吸收与归位”，不是“系统重写与升级”

────────────────
二、冻结文档保护规则
────────────────
以下文档视为冻结约束源，默认不可改写、不可删减、不可重命名、不可重释义：

1. 《系统宪法》v1
2. 《模块权限清单 v1》
3. 《治理冻结清单 + 生产主线待建清单》v1
4. 《GM-SkillForge 日终收口检查模板 v1》

对上述文档，你只能做以下动作：
- 引用
- 建立映射
- 提炼术语
- 标注归位建议
- 标注冲突点

你不得做以下动作：
- 直接改内容
- 重写原则
- 删除条款
- 降级约束力度
- 合并后丢失原语义
- 用新的措辞偷偷改变边界

若你判断某冻结文档内部存在问题，你只能输出：
- 问题点
- 影响范围
- 修正建议
不得直接替换。

────────────────
三、半冻结文档规则
────────────────
以下文档视为设计主输入，可整理、归并、结构化，但不得丢语义：

1. 《5层创建流程的数据结构清单 v1》
2. 5层流程.md
3. 架构纠偏文档 v1.md
4. skill交付形态.md
5. skill-creator和五层构建的区别.md
6. 商用skill的构建流程.md

允许动作：
- 术语统一
- 拆分为 schema / config / docs / checks
- 与主链路对齐
- 标注 P0/P1/P2
- 转成系统对象或配置草案

禁止动作：
- 擅自缩减为泛化表述
- 删除关键边界语义
- 将“创建流程”与“商用标准”混为一体
- 将“打包层”提升为“裁决层”

────────────────
四、主链路保护规则
────────────────
当前唯一受保护的生产主链路为：

需求输入
→ Intent Draft
→ Contract Draft
→ Candidate Skill
→ Build Validation
→ Delivery Manifest
→ skill-creator / 打包层

任何整合动作不得：
- 跳过 Intent Draft
- 跳过 Contract Draft
- 直接从需求到 Candidate Skill
- 用打包层替代 Build Validation
- 用 skill-creator 替代创建流程
- 用 build validation 替代治理审计

若某整合建议不能强化这条主链，而只是增加旁支复杂度，则应标记为：
- 非主链
- 后置
- 暂不整合

────────────────
五、双线分层保护规则
────────────────
你必须始终把系统分成四层，不得混权：

A. 生成线
负责：
- Intent
- Contract
- Candidate
- Build Validation
- Delivery Handoff

B. 审计线 / 治理线
负责：
- Gate
- Risk Decision
- EvidenceRef
- AuditPack
- Review
- Release
- Tombstone / Revision / At-time

C. 打包层
负责：
- skill-creator
- 标准技能包封装
- skill.zip 交付

D. 编排层
负责：
- 触发
- 调度
- 路由
- 通知
- 重试
- 人工审批编排

你不得：
- 让生成线获得最终发布权
- 让打包层获得审计通过权
- 让编排层获得 GateDecision
- 让 workflow / n8n 获得 release 写权
- 让 Skill 自己既生成又裁决又放行

────────────────
六、权限与写口约束
────────────────
以下写口必须被视为受保护写口：

1. released
2. validated
3. gate_result
4. run_state / business_state
5. evidence_ref / audit_pack
6. manual_approval

你不得在整合过程中：
- 新增事实性越权路径
- 保留未被文档授权的写口
- 通过重命名掩盖越权
- 让 orchestrator、scheduler、skill、skill-creator 事实性写入正式状态

一旦发现以下路径存在，必须优先标红：
- nl/skill → released
- orchestrator → released
- scheduler → release
- skill-creator → 审计放行
- build validation → governance pass
- n8n → final decision

────────────────
七、术语统一但不准偷换语义
────────────────
允许统一命名，但必须满足：
1. 原语义保留
2. 可回溯到原文档
3. 不改变角色边界
4. 不改变权责归属

例如：
- 可以统一 CandidateSkill 与 candidate skill 的写法
- 不能把 BuildValidationReport 偷换成 GovernanceValidation
- 不能把 DeliveryManifest 偷换成 ReleasePermit
- 不能把 skill-creator 偷换成创建主引擎

若术语合并，请输出：
- 原术语
- 新术语
- 语义是否完全等价
- 若不完全等价，必须保留双名说明

────────────────
八、禁止的整合冲动
────────────────
以下行为一律禁止：

1. 因为目录不好看而整体重构
2. 因为命名不统一而重写所有文档
3. 因为存在未来商用层而把 5+3 一次性全部塞进当前主线
4. 因为审计很重要而继续扩治理线
5. 因为打包方便而让 skill-creator 侵入创建流程前段
6. 因为想让系统更“智能”而削弱 contracts-first
7. 因为想让工作流更通用而把复杂度重新放回编排层
8. 因为想快速落代码而跳过整合报告

────────────────
九、允许的最小改动类型
────────────────
若进入改动阶段，只允许以下四类改动：

A. 归位改动
- 文档移动到正确目录
- schema/config/docs/checks 分类落位

B. 命名改动
- 在不丢语义前提下统一术语
- 增加别名映射说明

C. 接口改动
- 明确 handoff 点
- 明确层与层之间的输入输出

D. 保护性改动
- 去除明显越权路径
- 增加冻结标记
- 增加只读说明
- 增加审计/整合注释

除此之外，任何实质性架构重构都必须先报告再等待批准。

────────────────
十、整合过程中的强制检查点
────────────────
每完成一个阶段，你都必须检查这 6 件事：

1. 是否丢失了任何冻结规则
2. 是否把创建流程和商用流程混在了一起
3. 是否让生成线和审计线边界变模糊
4. 是否让 skill-creator 角色扩大
5. 是否让主链路变长但没变强
6. 是否新增了任何未授权写口

若任一项为“是”，必须暂停继续整合，并输出：
- 问题点
- 影响
- 回滚建议
- 最小修正动作

────────────────
十一、输出纪律
────────────────
你输出的每个整合结论，都必须标注属于哪一类：

- 冻结规则引用
- 主链路对齐
- 术语统一
- 目录归位
- 冲突修正建议
- 高风险提醒
- 暂不整合项

不要输出泛泛建议。
不要输出“可以考虑”“也许可以”这类漂浮表述。
所有建议必须明确：
- 改什么
- 为什么
- 影响哪一层
- 是否触碰冻结文档
- 是否需要批准

────────────────
十二、自动暂停条件
────────────────
出现以下任一情况，立即停止进一步整合，并先输出问题报告：

1. 需要修改冻结文档正文
2. 需要改动 release / validated / gate_result 写口
3. 需要改变主链路顺序
4. 需要改变“主审计在打包前”的口径
5. 需要改变“治理冻结、生产主攻”的阶段策略
6. 需要将 5层创建流程 与 5+3 商用流程合并成一个当前交付目标
7. 发现多个文档之间存在不可自动裁决的原则冲突

────────────────
十三、最终自检口径
────────────────
在你结束工作前，必须再次确认：

1. 生成线没有被审计线吞掉
2. 审计线没有被生成线架空
3. 打包层没有越权
4. 编排层没有越权
5. 冻结规则未丢失
6. 主链路更清晰，而不是更复杂
7. 当前系统更接近“生成不耽误、审计不耽误”

若这 7 条不能同时成立，则不得宣称整合完成。

────────────────
十四、你的执行姿态
────────────────
你不是来发明新系统的。
你是来把现有已定下来的系统意图，
在不失真、不越权、不扩散的前提下，
吸收到仓库结构和系统骨架里。

你的首要目标不是聪明，
而是忠实、稳健、收口、可持续。
