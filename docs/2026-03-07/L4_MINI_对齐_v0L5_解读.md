# L4-mini 对齐 v0-L5 解读（2026-03-07）

## 目的

把以下两份 v0/L5 文档，转换成符合当前系统现实状态的执行解读：

- `docs/2026-03-04/v0-L5/14天架构.md`
- `docs/2026-03-04/v0-L5/“三哈希口径”落成validate.py 的硬校验（v0）.md`

这份文档不重写原方案，而是回答三个问题：

1. 哪些方向今天仍然正确
2. 哪些内容可以作为 `L4-mini` 上线前硬边界
3. 哪些内容不该成为当前上线阻塞项，而应留到上线后迭代

---

## 一、总判断

结论：

两份 v0/L5 文档与当前方向 **总体相符**，但其定位更接近：

- `14天架构.md` = 完整路线图
- `三哈希硬校验.md` = 审计内核硬标准

而当前更现实的目标不是“一步到位完成完整 L4/L5”，而是：

**先收缩到 `L4-mini`，先上线一个受限域、可治理、可审计、可执行的最小闭环。**

因此，当前推荐策略不是推翻 v0/L5 文档，而是：

**保留其方向与边界，按 L4-mini 口径降级执行范围。**

---

## 二、与 `14天架构.md` 的对齐结论

### 仍然完全正确的部分

以下条目与当前现实状态高度一致，应该继续保留：

1. **必须收边界**
   - 只支持有限需求模式
   - 禁止一开始就做完全自由 NL
   - 超域请求必须 fail-closed

2. **必须先锁死 `NL -> Contract` 口径**
   - 需求不能自由漂移
   - 必须通过 `DemandSpec / Contract` 把口径冻结
   - 后续生成必须服从 frozen contract

3. **必须保留 L3 交付物**
   - EvidenceRef
   - AuditPack
   - Permit
   - GateDecision
   - required_changes / next_action

4. **必须允许 fail-closed**
   - 输入不清楚，不硬做
   - 缺证据，不放行
   - 缺 permit，不发布

### 需要按 `L4-mini` 降级执行的部分

以下内容方向没错，但对当前阶段过重，不应一次性全压上线前完成：

1. `14 天` 内完成 **NL -> 全套 Skill 工厂**
   - 当前可降级为：
     - 先完成受限域 `NL -> DemandSpec`
     - 再完成最小 `SkillSpec / 执行方案`
     - 不要求一开始就稳定产出所有全套工厂资产

2. 一开始就覆盖文档中的完整资产包
   - 当前可降级为：
     - 上线前必须有：最小 execution evidence + review/compliance + permit/gate
     - 上线后再补：更完整 blueprint / runbook / nodes manifest / eval pack

3. 一开始就跑 20 条 NL 样本成功率评估
   - 当前可降级为：
     - 先对 1 到 3 个黄金场景做闭环验证
     - 等上线后再扩展小样本统计

### 当前可直接执行的 `14天架构` 核心抽取

如果只取与 `L4-mini` 最相关的条目，当前最值得继续执行的是：

1. `Day 1` 的范围锁死
2. `Day 2` 的 DemandSpec 解析
3. `Day 3` 的 Contract Builder
4. `Day 4` 的一致性 Gate
5. `Day 9-11` 的 Evidence / AuditPack / Permit

这些是当前最有价值的最小骨架。

## 三、与 `三哈希硬校验.md` 的对齐结论

### 仍然完全正确的部分

这份文档大方向与当前系统治理完全一致，而且应该作为 `L4-mini` 的底层红线：

1. **三哈希必须进入审计底座**
   - `demand_hash`
   - `contract_hash`
   - `decision_hash`

2. **canonicalize 必须单一实现**
   - 不能每个模块各算各的
   - 否则 hash 会漂

3. **黑名单字段不能进 keyset**
   - timestamp
   - run_id
   - summary
   - reason / next_action 的自由文本
   - 其他非稳定字段

4. **Permit 必须绑定审计产物**
   - 没有 L3 审计闭环，不得签 Permit

5. **迭代必须带 diff / rationale / tombstone**
   - 这是未来升级路径可审计的关键

### 当前不应阻塞 `L4-mini` 上线的部分

以下内容依然重要，但可作为“上线后加固项”，不必阻塞最小版本：

1. 完整的 reproduce / iterate 双模式都一次做满
2. 所有 diff/tombstone 目录结构一次性封死
3. 所有 keyset 机器可读配置一步到位

当前更现实的做法是：

- 上线前先落：
  - canonicalize 单一入口
  - 三哈希可计算
  - Permit 签发前校验
- 上线后再补：
  - `hash_keysets.yml`
  - full iteration diff packs
  - reproduce/iterate CLI 统一体验

### 当前对 `L4-mini` 最关键的硬要求

如果只保留最关键的三条，当前最不该放松的是：

1. **single canonicalize implementation**
2. **三哈希写入 manifest / pack**
3. **Permit 前置校验 fail-closed**

这三条如果没有，`L4-mini` 很容易上线后失控。

## 四、当前 `L4-mini` 上线前必须有 / 可以后补

### 上线前必须有

1. **受限域输入边界**
   - 不做自由 NL
   - 只收有限场景

2. **最小 `NL -> DemandSpec -> Contract` 入口**
   - 至少能把输入结构化
   - 至少能明确缺什么信息

3. **一条受控执行 lane**
   - 不求全
   - 但必须稳定、可审计、可拦截

4. **Evidence / Gate / Permit 最小闭环**
   - 这部分不能砍

5. **最小结果展示层**
   - 至少能让用户看懂：
     - 系统理解了什么
     - 准备怎么做
     - 当前能不能执行

6. **小龙虾云端执行开发臂**
   - 云端执行目标必须明确、可冻结、可回传
   - 小龙虾可以在云端完成编码开发、脚本执行、产物打包、日志回传
   - 但最终 `review / compliance / permit / final gate` 仍保留在本地治理内核
   - 目标不是“云端自治发布”，而是“云端高吞吐执行，本地集中裁决”

7. **持续巡检探针**
   - 当前已经有：`scripts/run_3day_compliance_review.py`
   - 应继续作为上线期间的总探针

### 上线后补

1. 完整自由 NL 能力
2. 全量 `Skill -> n8n` 自动部署
3. `five-stage-orchestrator-skill`
4. `three-cards-orchestrator-skill`
5. 多 lane 并行执行
6. 完整 Wave 2 selective migration
7. 更完整的三哈希 iterate / reproduce 工具链

## 五、关于“小龙虾在云端干活完成编码开发”的定位

这是当前 `L4-mini` 的关键执行臂，不应再被视为“后面再说”的附属能力。

### 正确定位

**小龙虾 = 云端执行开发臂**

负责：

- 按冻结合同执行编码开发
- 跑受控脚本与构建流程
- 生成执行日志、产物、execution report
- 回传候选改动与证据

不负责：

- 最终审查
- 最终合规裁决
- Permit 签发
- final gate
- 擅自扩 scope

### 当前目标

当前目标不是“让小龙虾什么都干”，而是：

**抓住一切可并行时间，让小龙虾在云端按照已冻结要求完成执行开发，而本地内核保留裁决权。**

### 对 `L4-mini` 的直接意义

这意味着 `L4-mini` 不是单纯的：

- 需求理解
- 方案生成
- 结果展示

而是至少要做到：

- 受限域需求进入
- 合同冻结
- 小龙虾云端执行开发
- 产物回传
- 本地 review/compliance/final gate 收口

## 七、最终结论

**v0/L5 两份文档与当前方向相符，不需要推翻；但当前应该按 `L4-mini` 口径执行，即保留其边界和硬标准，收缩其一次性交付范围。**

- `三哈希硬校验.md` 应继续作为 **审计底座硬标准**

**“先上线一个受限域、可治理、可审计、可执行的 L4-mini；再沿着 v0/L5 路线补全剩余能力块。”**
