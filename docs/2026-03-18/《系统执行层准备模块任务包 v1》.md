六、三权分立推进模式
==================================================

本模块必须由三类角色并行推进，但职责必须严格分离。

------------------------------------------
A. 执行权（Execution Wing）
------------------------------------------

职责：
- 在本模块边界内完成任务拆分后的实际落地
- 产出目录骨架、文档草案、接口草案、模块职责草案
- 只能在允许范围内推进
- 不得自作主张扩大模块范围

执行权必须完成：
1. workflow / orchestrator / service / handler / api 五子面的最小职责草案
2. 五子面的最小承接关系草案
3. 最小目录/文件骨架建议或落位
4. 正式文档初稿
5. 汇总报告初稿

执行权不得：
- 做最终边界裁决
- 做合规放行
- 做最终验收
- 进入 runtime / 外部集成 / 真实执行实现

------------------------------------------
B. 审查权（Review Wing）
------------------------------------------

职责：
- 检查执行权产物的结构一致性、命名一致性、接口一致性、层级一致性
- 检查五子面是否分工清晰
- 检查是否存在重复职责、断裂职责、穿透职责
- 只做技术审查，不做边界扩权

审查权必须输出：
1. 结构一致性审查意见
2. 职责边界审查意见
3. 命名和目录审查意见
4. 文档口径一致性审查意见
5. 对每个问题给出“阻断性 / 非阻断性”判断建议

审查权不得：
- 直接改执行产物
- 直接放行
- 直接决定跨模块扩展
- 借审查之名新增需求

------------------------------------------
C. 合规权（Compliance Wing）
------------------------------------------

职责：
- 盯住禁止项
- 检查是否倒灌 frozen 对象边界
- 检查是否提前混入 runtime / 外部执行 / 集成语义
- 检查是否把 workflow / orchestrator 当成裁决者
- 检查是否把 service / handler / api 提前做成真实执行层

合规权必须输出：
1. 越界检查结果
2. 禁止项触发检查结果
3. 对 frozen 边界影响检查结果
4. 对“执行层是否篡改治理裁决权”的检查结果
5. 是否允许提交 Codex 验收的合规建议

合规权不得：
- 直接修改执行产物
- 直接替代审查权
- 直接做最终验收结论
