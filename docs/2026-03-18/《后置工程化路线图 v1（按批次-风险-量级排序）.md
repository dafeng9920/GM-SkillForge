《后置工程化路线图 v1（按批次 / 风险 / 量级排序）》

结论：
后置工程化应拆成 **3 个主批次 + 2 个条件触发分支**，而不是继续沿用前面对象冻结那种最细颗粒。当前最合理打法是：先把“对象冻结主线”停在已完成状态，再进入“系统执行层 → 外部执行与集成 → runtime 稳态化”三段式推进。你当前材料也支持这个分界：现阶段一个人更适合完成“母体、标准、主链路、样板间”，而完整工厂、运维、告警、发布、版本演进属于后续扩建工程。

置信度：0.89

关键风险点：

1. 执行层需求倒灌到已 frozen 的 gate/review/release/audit 对象，导致前面冻结价值被稀释。材料已明确：n8n 负责现实世界编排，SkillForge 必须独占裁决、证据链、AuditPack、permit，不应把治理与裁决重新外包出去。
2. 把后置工程化误当成“小补丁”，低估 workflow / retry / queue / notification / publish / rollback / observability 的体量。材料已经把这些能力归到后置编排与运维侧，而不是当前冻结主线。
3. 单人继续按“整座工厂一次做完”的预期推进，会导致节奏失真。当前更合理角色是先定义母体与主产线，再把扩建、运维、模板化、产品化留给后续批次。

## 总路线

### 批次 1：系统执行层准备与最小落地

目标：
把已经 frozen 的主线对象，接成**最小可执行内环**，但不把治理裁决外包出去。

核心原则：

* **n8n / orchestrator 只做编排**：触发、路由、重试、人工审批、通知、外部系统集成。
* **SkillForge 内核只做治理裁决**：8 Gate、EvidenceRef、AuditPack、permit、RAG at-time/tombstone。
* 执行层要围绕 `run_intent / fetch_pack / query_rag` 这类 contracts-first 接口展开，而不是把判断逻辑写进 workflow。

建议覆盖面：

* workflow 最小编排骨架
* orchestrator 最小路由
* service / handler / api 最小承接面
* permit 检查链
* GateDecision / EvidenceRef / AuditPack 到执行链的只读承接

不应在这一批做的：

* 大量外部系统扩展
* 复杂失败恢复体系
* 全量 observability
* 生产级性能调优
* 把 n8n 变成裁决者

退出条件：

* 最小编排链能跑通
* 所有关键动作都受 permit 约束
* workflow 不自行判 PASS/FAIL
* 执行层不回写已 frozen 对象语义

量级判断：
**中等偏大，约等于当前冻结主线的 0.8–1.2 倍工程量。**
这是推断值，但与材料中的职责切分是吻合的：现实世界复杂度放到 n8n，可信裁决保留在内核。

---

### 批次 2：外部执行与集成组

目标：
把“最小可执行内环”接到真实世界里，但仍保持治理和执行分层。

建议优先接入的对象：

* GitHub / repo 抓取
* tag / release / PR / registry
* Slack / Email 通知
* webhook / HTTP / queue / DB
* 定期重审与 tombstone 下架链

材料里已经给了适合 v0/v1 的接入口：n8n 负责抓 repo、触发审核、开 PR、发布、通知、重试与补偿；SkillForge 只做裁决、证据链、AuditPack、permit。

必须一起加上的硬规则：

* 发布必须持有 permit。
* Evidence/AuditPack 不可被编排层改写。
* 输入必须确定性：`repo_url + commit_sha (+ at_time)`，拒绝 latest 自动选择。

不应在这一批做的：

* 全量多云 / 多区域 / 多租户
* 复杂审批矩阵
* 大规模 connector 泛化
* 全面业务监控平台

退出条件：

* 至少 2–3 条外部编排链跑通，例如“审计并发布”“升级重审”“tombstone 下架”。材料里也建议先用 2–3 条 workflow 切口，而不是一次铺满。
* secrets 分层成立
* 幂等、重试、补偿至少有最小闭环
* 外部状态与内核 revision / audit pack 可对齐

量级判断：
**中等到偏大，约 0.6–1.0 倍当前冻结主线工程量。**
不确定性高于批次 1，因为它取决于外部系统差异、权限、失败恢复和状态同步。这一点和材料里“真实世界落地受权限、环境、责任限制”的判断一致。

---

### 批次 3：runtime 执行组

目标：
让系统从“能跑”升级到“能长期稳定地跑”。

建议覆盖：

* queue / retry / timeout policy
* failure branch / compensating actions
* audit log / evidence log / metrics / alert
* tombstone 后行为
* version rollout / rollback
* 稳态运行与资源治理

材料已经明确，n8n 的现实优势就在重试、失败分支、并行、队列化和运维能力；而你当前不该把这些塞回对象冻结主线。

不应在这一批做的：

* 再次改写 frozen 主对象
* 把 runtime 语义硬塞进 audit/release 对象
* 先做“漂亮大盘”再做真正的 failure control

退出条件：

* 关键链路具备最小可观测性
* 关键失败场景可重试 / 可补偿 / 可回滚
* 运行时策略不绕过 permit / audit pack / evidence
* 稳态运行不依赖人工盯盘

量级判断：
**偏大，约 0.8–1.5 倍当前冻结主线工程量。**
这是后置工程里最容易失控的一组，因为它从“结构正确”切到“长期稳定”。材料也明确把运维、观测、告警、发布、版本演进视为后续团队扩建项，而不是当前单兵应一次吞下的内容。

---

## 两个条件触发分支

### 分支 A：Audit 修复

只在以下情况触发：

* 后续执行层倒灌，污染了 Audit frozen 边界
* 新发现对象层与真实执行链不一致
* Frozen 结论与现有文件事实不一致

它不是主线步骤，是**返修分支**。

### 分支 B：Audit 重校验

只在 Audit 修复后触发。
它也不是主线步骤，而是**返修后的复核分支**。

## 风险排序

### 最高风险：外部执行与集成

原因：

* 外部系统差异最大
* 鉴权和 secrets 最敏感
* 幂等 / 回退 / 状态同步最容易出事故
* 容易让 n8n 越权

材料里已经明确：n8n 可以全包“干活”，但不能做最终裁决；permit、Evidence、AuditPack 不能被编排层改写。

### 第二风险：runtime 稳态化

原因：

* 一旦做不好，就会不断倒灌前面的 frozen 边界
* 很容易从“最小策略”膨胀成“全平台化”
* 单人最容易在这里失真

### 第三风险：系统执行层最小落地

原因：

* 这是最先开的执行层入口
* 一旦 contracts-first 失守，后面就会乱
* 但它比后两者更可控，因为你们已有明确的 governor/orchestrator 分层原则。

## 量级排序

### 最适合当前继续推进的顺序

1. **系统执行层准备与最小落地**
2. **外部执行与集成**
3. **runtime 执行与稳态化**

### 粗略工程量

如果把你当前已完成的“冻结主线”记作 **1.0**：

* 系统执行层：**0.8–1.2**
* 外部执行与集成：**0.6–1.0**
* runtime 稳态：**0.8–1.5**

合计：
**约 2.2–3.7 倍于当前冻结批次工程量。**

这也是为什么材料里反复强调：你现在做的是“母体、标准、主链路、样板间”，而完整工厂、运维、模板化、产品化，不适合单兵长期全扛。
