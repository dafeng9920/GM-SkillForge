## 3. 对第4层的映射

### 状态持久化（Stateful Persistence）

- 内容摘要：
  - 明确记录 `run_id / stage / checkpoint / retry_count / last_error_code`
- 为什么属于第4层：
  - 它解决的是中断后如何恢复，而不是怎么观察系统
- 补强了第4层的什么能力：
  - 状态管理
  - 故障恢复
- 当前状态：
  - 当前吸收

### 断点续传（Checkpoint / Resume）

- 内容摘要：
  - 定义 checkpoint、resume 起点、可重放与不可重放步骤
- 为什么属于第4层：
  - 它直接决定长流程在失败后的恢复能力
- 补强了第4层的什么能力：
  - 故障恢复
  - 失败模式设计
- 当前状态：
  - 当前吸收

### 幂等性（Idempotency）

- 内容摘要：
  - 用去重 key、processed set、幂等 token 等控制重复副作用
- 为什么属于第4层：
  - 它属于重复执行后的鲁棒性设计，不是运维可见性设计
- 补强了第4层的什么能力：
  - 幂等
  - 重试安全
- 当前状态：
  - 当前吸收

### 重试策略（Retry / Backoff）

- 内容摘要：
  - 明确 retryable / non-retryable、上限次数、backoff、jitter
- 为什么属于第4层：
  - 它属于失败后的恢复策略
- 补强了第4层的什么能力：
  - 重试
  - 故障恢复
- 当前状态：
  - 当前吸收

### 限流与节流（Rate Limiting / Throttling）

- 内容摘要：
  - 控制请求速率，避免封禁、风控、配额耗尽
- 为什么属于第4层：
  - 它本质上是外部依赖下的稳态保护机制
- 补强了第4层的什么能力：
  - 失败模式设计
  - 降级前保护
- 当前状态：
  - 后续落地

### 降级策略（Graceful Degradation）

- 内容摘要：
  - 完整流程失败时仍输出可解释失败、部分结果或人工接管信号
- 为什么属于第4层：
  - 它定义失败后的服务退化方式
- 补强了第4层的什么能力：
  - 降级
  - 人机协同
- 当前状态：
  - 当前吸收

### 环境一致性（Environment Parity）

- 内容摘要：
  - 锁定运行环境、依赖、浏览器/驱动/镜像信息
- 为什么属于第4层：
  - 它减少环境漂移导致的失效
- 补强了第4层的什么能力：
  - 失败模式设计
  - 故障恢复前预防
- 当前状态：
  - 当前吸收

### 影子测试（Shadow Testing）

- 内容摘要：
  - 后台静默跑一段时间，对比人工结果和历史结果
- 为什么属于第4层：
  - 它本质上是在上线前验证鲁棒性与偏差
- 补强了第4层的什么能力：
  - 影子测试
  - 失败模式设计
- 当前状态：
  - 后续落地

### 人机协同降级（Human-in-the-Loop）

- 内容摘要：
  - 自动化失败时触发人工接管，并保留恢复后续流程的能力
- 为什么属于第4层：
  - 它定义自动化失效后的接管机制
- 补强了第4层的什么能力：
  - 人机协同
  - 降级
- 当前状态：
  - 当前吸收

### 失败模式分类（Failure Taxonomy）

- 内容摘要：
  - 定义网络、认证、环境、改版、风控、数据结构异常等失败类目
- 为什么属于第4层：
  - 它是所有恢复策略与降级策略的前提
- 补强了第4层的什么能力：
  - 失败模式设计
  - 重试 / 报警 / 人工接管的分类基础
- 当前状态：
  - 审计清单化

## 5. 交叉项与边界说明

### 横跨第4层和第5层的条目

- 影子测试
  - 第4层角度：验证偏差与鲁棒性
  - 第5层角度：需要样本量、偏差记录、可比较指标

- 人机协同降级
  - 第4层角度：失败后的恢复/接管策略
  - 第5层角度：需要告警、事件记录、运维动作入口

- 失败模式分类
  - 第4层角度：设计恢复路径
  - 第5层角度：决定日志、指标、告警分类口径

### 为什么它们不是新层

- 它们没有脱离“韧性设计”与“可观测运维”这两个核心语义
- 它们是层内细化，不是新的主链责任中心
- 它们依附于现有第4层、第5层，不独立承担主骨架职责

### 为什么不能脱离 5层主骨架单独成立

- 它们不定义需求进入、对象生成、治理判定或发布判定
- 它们只补强第4/第5层的实施深度
- 一旦脱离 5 层主骨架，就会被误读为新体系或新层
