# SkillForge 架构决策与三维 RAG 设计整理文档

时间锚点：**2026-02-16 11:00 (UTC+8)**

---

# 一、总体战略决策

## 1. 架构路线

SkillForge 采用：

> **纯新建 + Contracts-First + 门禁优先 + 审计可复现**

不拷贝 GM OS 旧代码，仅继承理念与合同口径。

---

## 2. 不拷贝 GM OS 的原因

### 拷贝风险

* 带入历史耦合与复杂运行时结构
* 破坏 contracts-first 原则
* AI 军团推进复杂度上升
* 失去“可复核产线”纯度

### 真正需要的能力只有：

* Gate（门禁）
* Audit（审计）
* Registry（注册）
* Execution Pack（执行包）

这些可以轻量重写。

---

## 3. 项目不可变约束（宪法级）

1. 不拷贝 GM OS 旧实现
2. Contracts-first
3. CI = 门禁
4. 实现层永远可替换
5. 未来 GM OS 由 SkillForge 反向长出（adapter 形式）

---

# 二、L5 审计包出厂硬标准

L5 审计包必须具备完整证据链。

---

## 必须包含（MUST）

### 1️⃣ manifest.json

* 列出包内所有文件
* sha256 + 字节数
* skill_id
* pack_version
* build_time
* provenance

---

### 2️⃣ policy_matrix.json

* rule_id
* 描述
* 检测逻辑
* 结果（pass / fail / warn）
* evidence_refs

> 不再使用“GM OS 宪法”命名
> 采用中立命名：POL-SEC-xxx

---

### 3️⃣ static_analysis.log

原始扫描日志（可 gzip）。

---

### 4️⃣ harness_benchmark.json

包含：

* runs
* success_rate
* p95_latency_ms
* error_distribution
* resource_peaks

L5 ≥ 1000 次测试。

---

### 5️⃣ source_lineage.diff

展示从原始代码到增强代码的变更。

---

### 6️⃣ original_repo_snapshot.json

* upstream_url
* commit_sha
* license
* fetched_at

---

### 7️⃣ repro_env.yml 或 Dockerfile

必须记录：

* 依赖版本
* timeout
* max_bytes
* rate_limit

---

## 强烈建议（SHOULD）

* signature.asc（签名）

---

## 可选（MAY）

* stress_test_vectors.csv（需标明 mock/real）

---

## CI 强制校验规则

1. manifest 覆盖所有文件
2. sha256 校验一致
3. evidence_ref 必须存在
4. repro_env 必须包含运行限制字段

---

# 三、三维 RAG 设计（Entity × Artifact × Time）

目标：

1. 可储存
2. 可复现
3. 可增量更新
4. 可逻辑删除
5. 可按时间回放

---

## 三维定义

### 1️⃣ Entity 维

`skill_id`

---

### 2️⃣ Artifact 维

* schema
* policy
* example
* test
* audit_pack
* notes
* 其他结构化资产

---

### 3️⃣ Time 维

* effective_at（生效时间）
* revision_id
* status（ACTIVE / DEPRECATED / TOMBSTONED）

---

# 四、物理结构

```
storage/
  skills/<skill_id>/
    revisions/<revision_id>/
      meta.json
      manifest.json
      artifacts/
        schemas/
        orchestration/
        examples/
        contract_tests/
        audit_pack/

db/
  skillforge.sqlite
```

---

# 五、数据库核心模型

## skills

| field      | description |
| ---------- | ----------- |
| skill_id   | 主键          |
| title      | 名称          |
| created_at | 创建时间        |
| updated_at | 更新时间        |

---

## revisions

| field           | description                      |
| --------------- | -------------------------------- |
| revision_id     | 主键                               |
| skill_id        | 归属                               |
| effective_at    | 生效时间                             |
| status          | ACTIVE / DEPRECATED / TOMBSTONED |
| manifest_sha256 | 版本指纹                             |
| path            | 存储路径                             |

---

## artifacts

| field         | description                   |
| ------------- | ----------------------------- |
| artifact_id   | 主键                            |
| revision_id   | 所属版本                          |
| artifact_type | schema / policy / example ... |
| sha256        | 文件指纹                          |

---

## tombstones

逻辑删除记录。

永远不物理删除历史版本。

---

## rag_chunks

检索层数据。

RAG 只负责定位，不负责改变事实。

---

# 六、操作 API 规范

## 新增版本

```
append_revision(skill_id, revision_id, effective_at)
```

---

## 作废版本

```
deprecate_revision(skill_id, revision_id, effective_at, reason)
```

---

## 删除某个 artifact

```
tombstone_artifact(skill_id, artifact_id, effective_at, reason)
```

---

## 时间回放

```
get_snapshot(skill_id, at_time)
```

---

## 语义检索

```
semantic_search(query, filters)
```

---

# 七、核心原则

1. 所有“删除”必须是 tombstone
2. 物理文件不可覆盖
3. RAG 不得修改事实层
4. manifest 是唯一事实来源
5. revision 是事件流，不是替换

---

# 八、实施优先级

## v0 必须完成

* contracts
* gates
* audit pack
* revision 体系

---

## v1 再增强

* 向量索引
* embedding
* 性能优化
* 外部集成

---

# 九、最终结构关系

SkillForge =

> 产线标准 + 门禁 + 审计 + 版本控制 + 证据链

未来 GM OS =

> 多 Skill 编排 + 运行时 + 宪法治理

关系：

> GM OS 加载 SkillForge 产物
> SkillForge 不依赖 GM OS

---

# 十、结论

当前架构是：

* 可复现
* 可回放
* 可审计
* 可扩展
* 不被历史复杂度拖死

三维 RAG 设计合理，
但必须建立在 **revision + manifest 的事实层** 之上。
