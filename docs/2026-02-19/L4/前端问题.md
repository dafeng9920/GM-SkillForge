看清了。现在这版“丑”的主要原因不是配色，而是 **信息层级没站起来 + 留白方式不对 + 关键状态没被“产品化”**。我按“最少改动、最快变好看”给你一套可直接交给 Codex 的改造清单。

## 1) 先拆问题点（你这张图里最伤观感的 6 件事）

1. **大面积空黑**：聊天区/工作区都很空，但没有“空状态引导”，看起来像坏了。（0.9）
2. **消息气泡像悬浮孤岛**：不在同一条基线里，没有时间/角色/动作的统一排版。（0.85）
3. **输入区层级太低**：底部输入框像“附属品”，CTA 不聚焦。（0.85）
4. **顶部 Stepper 不像 Stepper**：1 分歧/2 认知/3 采纳/4 执行没有“当前状态/进度”。（0.8）
5. **工作窗核心状态太弱**：右上角“许可：待定 / 门：空闲”是核心资产，但现在像装饰。（0.88）
6. **按钮语义不清**：右下角“发送/执行”并列，但用户不知道先后、也不知道执行会发生什么。（0.84）

## 2) 最小改动就能明显变好看的方案（MVP 版）

### A. 给两窗一个明确角色：Draft vs Work（0.86）

* 左侧标题改成：**Draft（自由推理，不直接发布）**
* 右侧标题改成：**Work（受 Gate/Permit 约束，可产生交付物）**
* 顶部 Stepper 变成“当前阶段高亮 + 可点击回看”

  * 当前阶段用实心/强调色，其他阶段弱化

### B. 右侧 Work 窗必须有“三件套固定栏”（0.9）

在右侧顶部下面加一个固定的 **Workbench Header**，三块卡片横排（或竖排也行）：

1. **Permit 状态卡**：`待定 / PASS / FAIL / TOMBSTONE`（带原因入口）
2. **Gate 进度卡**：8 Gate 进度条 + 当前 Gate 名称
3. **Artifacts 卡**：AuditPack / required_changes / evidence（有就点开）

> 这一下就把“系统强项”视觉化了：你们不是聊天，是交付流水线。

### C. 空状态要“像产品”，别空黑（0.88）

两边都做空状态组件：

* Draft 空：给 3 个示例提示（“描述需求 / 输入约束 / 想要输出形态”）
* Work 空：给 2 个主按钮

  * **从 Draft 生成合同**（主 CTA）
  * **粘贴/导入 AuditRequest**（次 CTA）

### D. 底部输入区重做为“一个主动作 + 一个次动作”（0.84）

右下角不要“发送/执行”并列。

* Draft：只有 **发送**（Enter）+ **换行**（Shift+Enter）
* Work：只有一个主按钮 **执行（Run）**，旁边一个次按钮 **预检（Validate）**
* 执行按钮旁边显示“将运行：8 Gate / 生成 L3 AuditPack”（让用户敢点）

### E. 消息列表统一排版（0.82）

* 同一列对齐：头像/角色/时间在一条竖线
* 气泡宽度限制（比如 60–72ch），别铺太宽
* 系统提示（欢迎语）不要占一大条，改成顶部小提示条或空状态里展示

## 3) 半天内能再上一档的“高级但不复杂”增强

1. **Work 窗右侧抽屉**：点 required_changes → 展开 checklist（可勾选）（0.8）
2. **EvidenceRef 可点击**：点击跳转到“证据面板”（log/片段/文件路径）（0.78）
3. **Gate Timeline**：8 Gate 以时间线显示（通过/失败/跳过），失败节点可展开 reason/next_action（0.82）

  | "DOC_UPDATE"

## 4) 强制交互规则（写死，避免 UI 语义漂移）

1. **无 permit（permit.status !== VALID）**：`Run` 必须禁用，并显示原因（来自 permit.reason_short）
2. **Gate RUNNING**：`Run` 禁用，允许 `Validate`（可选）
3. `decision=FAIL`：自动弹出 `RequiredChangesDrawer`（或顶部出现“查看整改清单”主按钮）
4. Timeline 上点 FAIL 节点：默认打开对应的 `required_changes`（按 required_changes_refs）
5. 点击任意 EvidenceRef：统一打开 `EvidenceDrawer`，并能显示 locator + sha256 + excerpt

## 5) 后端对接最小返回（Codex 对接用）

工作窗只需要一个聚合接口（或 websocket 推送）返回 `WorkbenchSession`：

* `GET /workbench/session/:id`
* `POST /workbench/run`（返回 session_id + 开始推送 gate_timeline 更新）
* `POST /workbench/validate`

这样前端不需要理解 8 Gate 细节，只渲染 session。

## 4) `ui/contracts/workbench_mappings.ts`（TS 常量加载器：UI 只读 YAML，不手写魔法字符串）

```ts
// ui/contracts/workbench_mappings.ts
// 约定：构建时用 yaml-loader 或先把 YAML 转 JSON（推荐）。
import workbenchNodes from "./workbench_nodes.yml";
import errorUI from "./error_ui.yml";
import rcPolicy from "./required_changes_policy.yml";

export type GateNode =
  | "intake_repo"
  | "license_gate"
  | "repo_scan_fit_score"
  | "draft_skill_spec"
  | "constitution_risk_gate"
  | "scaffold_skill_impl"
  | "sandbox_test_and_trace"
  | "pack_audit_and_publish";

export type Severity = "INFO" | "WARN" | "ERROR" | "BLOCKER";

export const NODE_ORDER: GateNode[] = workbenchNodes.node_order;

export const NODE_META: Record<GateNode, any> = Object.fromEntries(
  workbenchNodes.gate_nodes.map((g: any) => [g.node, g])
);

export const SEVERITY_RANK: Record<Severity, number> = rcPolicy.severity_rank;

export const GATE_NODE_RANK: Record<GateNode, number> = rcPolicy.gate_node_rank;

export function sortRequiredChanges<T extends { severity: Severity; gate_node: GateNode; issue_key: string }>(
  items: T[]
): T[] {
  return [...items].sort((a, b) => {
    const sa = SEVERITY_RANK[a.severity] ?? 0;
    const sb = SEVERITY_RANK[b.severity] ?? 0;
    if (sb !== sa) return sb - sa;

    const ga = GATE_NODE_RANK[a.gate_node] ?? 999;
    const gb = GATE_NODE_RANK[b.gate_node] ?? 999;
    if (ga !== gb) return ga - gb;

    return (a.issue_key || "").localeCompare(b.issue_key || "");
  });
}

// error 文案：强制从 orchestration/error_codes.yml 查（前端可通过后端聚合接口拿到）
export const ERROR_UI_RULES = errorUI;
```
