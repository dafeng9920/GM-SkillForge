# 世界级项目操盘提示词

> 版本: v1.0
> 创建日期: 2026-02-21
> 用途: 顶级产品/架构/开发视角的项目审视与优化

---

## 使用说明

将以下提示词发送给顶级大模型，请求从"乔布斯级产品经理 + 世界级架构大师 + 顶尖工程师"三重视角审视项目，输出可落地的优化建议。

---

## 提示词内容

```
你是世界顶级大模型。现在请你同时扮演以下三个角色：

【角色一：乔布斯级产品经理】
- 用户至上，追求极致体验
- "少即是多"，砍掉一切不必要的复杂度
- 产品要让人"一见钟情"
- 关注情感连接，而不仅仅是功能堆砌

【角色二：世界级架构大师】
- 系统设计要优雅、可扩展、可维护
- 关注边界、契约、治理
- 在复杂度和灵活性之间找到最佳平衡
- 架构要经得起时间考验

【角色三：顶尖工程师】
- 代码即文档，简洁即美德
- 关注性能、可靠性、可观测性
- 拒绝过度工程，追求实用主义
- 技术选型要务实，不追新技术

---

【项目背景】

项目名称：SkillForge L4.5
项目定位：AI 驱动的技能治理与编排平台

核心特性：
1. n8n 编排 + SkillForge 裁决的双层架构
2. 严格的 Fail-Closed 安全边界
3. 完整的审计追踪（run_id, evidence_ref, replay_pointer）
4. 五段式工作流（trigger → collect → process → deliver → report）
5. 三张卡片产品视图（understanding → plan → execution_contract）

后端能力：
- /api/v1/n8n/run_intent（执行意图）
- /api/v1/n8n/fetch_pack（审计包查询）
- /api/v1/n8n/query_rag（知识库检索）
- /api/v1/n8n/import_external_skill（外部技能导入）
- /api/v1/n8n/health（健康检查）

安全约束：
- n8n 只能触发与路由，不可注入 gate_decision/release_allowed/permit_token
- 所有裁决由 SkillForge 内部完成
- 禁止字段：gate_decision, release_allowed, permit_token, evidence_ref, permit_id, run_id

前端规划：
- v1.0：5 页（执行中心 / 审计查询 / 系统监控）
- v1.1：治理增强（Gate/Permit/Work-items 三 Tab）

设计风格：
- 审计中台（白底高对比，证据导向）
- 状态色固定（ALLOW绿/BLOCK红/PENDING灰/WARN橙）
- 信息层级（结论 → 标识 → 细节 → 证据）

---

【你的任务】

请从产品、架构、开发三个视角，回答以下问题：

## 一、产品视角（乔布斯级）

1. **核心价值**
   - 这个产品要解决的根本问题是什么？
   - 用户最痛的那个点，我们解决得够不够极致？
   - 如果只能保留一个功能，应该是哪个？

2. **用户体验**
   - 新用户第一次打开，会不会"哇"？
   - 哪些操作可以简化？哪些步骤可以砍掉？
   - 错误提示够不够"人性化"？用户被 BLOCK 时，知道怎么办吗？

3. **差异化亮点**
   - 相比竞品，我们最独特的点是什么？
   - 这个点用户能感知到吗？
   - 如何让这个亮点更突出？

## 二、架构视角（世界级）

1. **系统边界**
   - n8n 和 SkillForge 的职责划分是否清晰？
   - 边界契约是否足够稳定？
   - 有哪些"灰色地带"需要明确？

2. **治理模型**
   - Gate 流水线设计是否合理？
   - Permit 生命周期是否完整？
   - 审计追踪是否能"还原现场"？

3. **扩展性**
   - 新增一个 Gate 需要改多少代码？
   - 新增一种错误码需要改动哪些模块？
   - 如果用户量增长 100 倍，架构扛得住吗？

## 三、开发视角（顶尖工程师）

1. **技术债务**
   - 当前代码最大的"坏味道"是什么？
   - 哪些地方过度工程了？
   - 哪些地方又欠考虑了？

2. **可观测性**
   - 出问题时，能在 5 分钟内定位吗？
   - 日志/指标/追踪是否完整？
   - 有哪些"盲区"需要补？

3. **交付效率**
   - 从需求到上线，最快的路径是什么？
   - 哪些环节可以自动化？
   - 如何确保每次发布都不"翻车"？

---

【输出要求】

1. 每个视角输出 3-5 条**最关键**的建议（不是面面俱到，而是直击痛点）
2. 建议必须**具体可落地**，避免空泛（如"优化性能"要说出"优化哪个接口的哪个查询"）
3. 给出**优先级排序**（P0/P1/P2）
4. 最后用 100 字总结：如果要让你用一个词形容这个项目，你会用什么词？为什么？

- [P1] ... → ...

### 扩展性
- [P2] ... → ...

### 技术债务
- [P0] ... → ...

### 可观测性

> "这个词是：______。因为______。"
```

## 预期输出示例

```markdown
## 产品视角（乔布斯）

### 核心价值
- [P0] 当前产品解决了"AI 技能的安全执行"问题，但用户感知不够。→ 在首页增加"安全分数"可视化，让用户一眼看到 SkillForge 拦截了多少风险操作。
- [P0] 错误提示太技术化。→ 将 `N8N_FORBIDDEN_FIELD_INJECTION` 改为"安全规则阻止了此次操作，请联系管理员"。

### 用户体验
- [P1] 执行结果页信息密度太高。→ 采用"渐进式披露"：先只显示 ALLOW/BLOCK，点击展开看详情。
- [P1] run_id 太长，用户记不住。→ 增加"一键复制"和"二维码分享"功能。

### 差异化亮点
- [P1] replay_pointer 是独特能力，但用户不知道。→ 在每个结果页增加"时间机器"按钮，点击可回放当时的执行状态。

## 架构视角（世界级）

### 系统边界
- [P0] n8n 和 SkillForge 的职责清晰，但缺少"边界测试"。→ 增加集成测试，模拟 n8n 注入禁止字段的场景，确保 100% 被拦截。

### 治理模型
- [P1] Gate 流水线缺少"回滚"机制。→ 增加 rollback API，当 S4 决策失败时可以回滚到 S3 重新审计。

### 扩展性
- [P2] 新增 Gate 需要改 3 处代码（路由/服务/测试）。→ 引入"Gate 注册表"，新增 Gate 只需配置，无需改代码。

## 开发视角（顶尖工程师）

### 技术债务
- [P0] 前端使用了 mock 数据，未直连真实 API。→ v1.0 必须全部切换为真实 API 调用。
- [P1] 错误处理不统一，有的返回 HTTP 500，有的返回 200+error_code。→ 统一为 200 + error_envelope 格式。

### 可观测性
- [P1] 缺少端到端的 trace_id。→ 在请求头增加 X-Trace-ID，贯穿 n8n → SkillForge → 前端。

### 交付效率
- [P2] 前端组件未形成 Design System。→ 提取 5 个基础组件（StatusBadge、CopyableText、JsonViewer、Timeline、Card），形成可复用库。

## 一句话总结

> "这个词是：**守护者**。因为 SkillForge 不是简单的编排工具，而是 AI 世界的安全守护者——它在用户看不见的地方，用严格的规则保护每一次执行。"
```
