# SkillForge L4.5 世界级项目审视报告

> 评审日期: 2026-02-21
> 评审角色: 乔布斯级产品经理 + 世界级架构大师 + 顶尖工程师
> 项目版本: v1.7

---

## 一、产品视角（乔布斯）

### 1.1 核心价值

| 优先级 | 问题 | 建议 |
|--------|------|------|
| **P0** | 产品解决了"AI 技能安全执行"问题，但用户**感知不到**安全的价值 | 在首页增加**"守护分数"**可视化：显示 SkillForge 拦截了多少风险操作、保护了多少次执行 |
| **P0** | Fail-Closed 很强，但**用户被 BLOCK 时会感到挫败** | 错误提示改为"人性化"语言 + **一键修复按钮**。例如 `N8N_FORBIDDEN_FIELD_INJECTION` → "安全规则阻止了此次操作 → [查看原因] [联系管理员]" |
| **P1** | 功能堆砌，核心价值不突出 | 砍掉 v1.0 的 5 页，**只保留 1 页**：执行意图页。其他功能作为该页面的 Tab 或抽屉 |

**一句话**：用户不关心"Gate 流水线"，他们只关心"我的任务执行成功了吗？为什么失败？怎么办？"

### 1.2 用户体验

| 优先级 | 问题 | 建议 |
|--------|------|------|
| **P0** | 首屏信息过载，用户看到一堆技术字段 | **渐进式披露**：首屏只显示 ALLOW/BLOCK 大图标，点击展开看 run_id/evidence_ref，再点击看 JSON |
| **P1** | run_id 太长，用户记不住、输不对 | 增加**一键复制** + **二维码分享** + **短链接**（如 `sf.io/x8k2`） |
| **P1** | 表单字段太多，用户填写负担重 | **智能预填**：根据用户历史自动填充 repo_url/requester_id；commit_sha 支持从 Git 自动获取 |
| **P2** | 深色模式缺失，夜间使用刺眼 | v1.1 增加**深色模式**，与审计中台风格配套 |

**一句话**：好的 UI 是"一眼看懂，一步操作"，不是"先研究十分钟文档再填表"。

### 1.3 差异化亮点

| 优先级 | 亮点 | 如何放大 |
|--------|------|---------|
| **P0** | **replay_pointer（时间机器）** | 在每个结果页增加"⏪ 回放"按钮，点击可回到**执行时的系统状态**。这是竞品没有的能力 |
| **P1** | **三张卡片（understanding → plan → execution_contract）** | 将其做成**可打印的 PDF 报告**，用户可以下载存档或分享给审计方 |
| **P2** | **六步流水线可视化** | 做成**动画演示**，让用户"看到"技能被安全导入的过程 |

**一句话**：用户不会因为你有 8 个 Gate 而选择你，但会因为"我能回放 30 天前的任何一次执行"而记住你。

## 二、架构视角（世界级）

### 2.1 系统边界

| 优先级 | 问题 | 建议 |
|--------|------|------|
| **P0** | n8n 和 SkillForge 的边界清晰，但**缺少边界测试** | 增加集成测试套件：模拟 n8n 注入禁止字段的 100+ 场景，确保 100% 被拦截 |
| **P1** | 边界契约分散在多个文件中 | 创建**单一契约文件** `boundary_contract.yaml`，所有禁止字段/白名单字段集中管理 |
| **P2** | 前端和后端的契约不够正式 | 使用 OpenAPI 3.1 规范定义所有 API，并自动生成前端类型 |

**一句话**：边界不是"文档里写的"，而是"测试里验证的"。

### 2.2 治理模型

| 优先级 | 问题 | 建议 |
|--------|------|------|
| **P1** | Gate 流水线缺少**回滚机制** | 增加 `rollback_to_gate` API，当 S4 决策失败时可回滚到 S3 重新审计 |
| **P1** | Permit 生命周期不完整（只有签发，没有撤销/续期流程） | 增加 `revoke_permit` / `renew_permit` API，并记录撤销原因 |
| **P2** | 审计追踪能"还原现场"，但**还原成本高** | 提供 `replay_pack` API，一键还原当时的 RAG 索引、配置、代码版本 |

**一句话**：好的治理是"能进能出"，不是"只进不出"。

### 2.3 扩展性

| 优先级 | 场景 | 当前代价 | 优化建议 |
|--------|------|---------|---------|
| **P1** | 新增一个 Gate | 改 3 处代码（路由/服务/测试） | 引入 **Gate 注册表**，新增 Gate 只需配置 `gates/registry.yaml` |
| **P1** | 新增一种错误码 | 改 2 处代码（后端/前端） | 错误码集中定义在 `error_codes.yaml`，前端自动读取生成映射 |
| **P2** | 用户量增长 100 倍 | 当前架构无法水平扩展 | 引入 **消息队列**（n8n → Kafka → SkillForge），解耦触发和裁决 |

**一句话**：扩展性的本质是"配置化 + 解耦"，不是"硬编码 + 紧耦合"。

## 三、开发视角（顶尖工程师）

### 3.1 技术债务

| 优先级 | 债务 | 偿还方案 |
|--------|------|---------|
| **P0** | 前端使用 mock 数据，未直连真实 API | v1.0 **必须全部切换为真实 API**，删除所有 mock 代码 |
| **P0** | 错误处理不统一（有的 500，有的 200+error_code） | 统一为 **200 + error_envelope**，并在 API 文档中强制规定 |
| **P1** | 前端组件未形成 Design System | 提取 5 个基础组件：`StatusBadge`、`CopyableText`、`JsonViewer`、`Timeline`、`Card` |
| **P2** | 测试覆盖率不足（<60%） | 设置 CI 门禁：PR 合并前测试覆盖率 **≥80%** |

**一句话**：技术债务像信用卡，早晚要还，越晚还利息越高。

### 3.2 可观测性

| 优先级 | 盲区 | 补充方案 |
|--------|------|---------|
| **P0** | 缺少端到端 trace_id | 在请求头增加 **X-Trace-ID**，贯穿 n8n → SkillForge → 前端，并在日志中统一输出 |
| **P1** | 性能指标缺失 | 增加 Prometheus 指标：`skillforge_gate_duration_seconds`、`skillforge_execution_total` |
| **P1** | 告警规则缺失 | 配置告警：Gate 耗时 >5s、错误率 >1%、Permit 签发失败率 >0.1% |
| **P2** | 前端错误监控缺失 | 接入 Sentry，捕获前端 JS 错误并关联 run_id |

**一句话**：没有可观测性的系统是"黑盒"，出了问题只能猜。

### 3.3 交付效率

| 优先级 | 瓶颈 | 优化方案 |
|--------|------|---------|
| **P1** | 从需求到上线周期长 | 建立 **Feature Flag 系统**，代码合并后可通过开关控制上线 |
| **P1** | 手动测试占比高 | 增加 **E2E 自动化测试**（Playwright），覆盖核心流程：执行意图 → 查看结果 → 跳转审计 |
| **P2** | 发布时缺少自动校验 | 建立 **发布检查清单**：API 契约兼容性、数据库迁移、前端构建、测试覆盖率 |
| **P2** | 文档和代码不同步 | 强制要求 PR 中包含文档更新，否则无法合并 |

**一句话**：交付效率 = 自动化程度 - 沟通成本。

## 四、综合建议（优先级排序）

### P0（必须做，v1.0 阻塞项）

1. **前端直连真实 API**，删除所有 mock 代码
2. **错误提示人性化** + 一键修复按钮
3. **渐进式披露 UI**：首屏只显示 ALLOW/BLOCK
4. **增加边界测试套件**，确保禁止字段 100% 被拦截
5. **增加 X-Trace-ID**，实现端到端追踪

### P1（应该做，v1.1 核心项）

6. **首页增加"守护分数"可视化**
7. **"时间机器"按钮**，突出 replay_pointer 亮点
8. **Gate 注册表**，新增 Gate 只需配置
9. **错误码集中定义**，前端自动读取
10. **E2E 自动化测试**，覆盖核心流程

### P2（可以做，后续迭代）

11. **深色模式**
12. **三卡 PDF 报告**
13. **消息队列解耦**
14. **Feature Flag 系统**

## 五、一句话总结

> **这个词是：守护者。**
>
> 因为 SkillForge 不是简单的"AI 编排工具"，而是 AI 世界的**安全守护者**——它在用户看不见的地方，用严格的规则保护每一次执行。就像机场安检，用户可能觉得麻烦，但当他们知道"有守护者在保护我"时，会感到安心。
>
> **核心建议**：把"守护"变成可见、可感知、可信任的产品体验。
